# Spec 16 — Realms and Accounts

## Two realms, one DB

We run **two realms** on the same MaNGOS world DB, separated only by
`mangosd.conf` settings. Same map data, same quests, same spawns —
different gameplay rules:

| Realm | Name (Warcraft-themed) | Purpose | Lockouts | Refresh timers | Default settings |
|---|---|---|---|---|---|
| **Dev/Test** | `Westworld-Test` | All automated tests; OnDemand experimentation; iteration. | Disabled or short (`InstanceUnloadDelay`, raid/dungeon reset frequencies). | Accelerated. | Permissive (XP rate multipliers OK, GM commands enabled). |
| **Prod** | `Westworld` | Living-server target. | Default Vanilla 1.12.1 (5-day daily, 7-day weekly raid resets). | Default. | Vanilla 1.12.1 defaults. |

Both realms appear on the same `realmd`. Tests connect to `Westworld-Test`.
Production progression and OnDemand activities serve users on `Westworld`.

Realm names are configured in `realmd.realmlist` and `mangosd.conf`. The
exact `mangosd.conf` deltas for the test realm live at
`docker/mangosd/mangosd-test.conf` (with the prod variant at
`docker/mangosd/mangosd-prod.conf`).

## Account + character explicit lists

Every account and every character is **explicitly declared** in config.
There is no random or implicit roster — even production. A test that
runs against "any GATHFG1-like bot" is wrong; tests run against named
accounts.

### Account declaration

`Config/accounts/` is the authority. Each file lists accounts:

```json
{
  "Realm": "Westworld-Test",
  "Accounts": [
    {
      "AccountName": "TESTBOT1",
      "Password": "PASSWORD",
      "GmLevel": 0,
      "Purpose": "Foreground test bot — primary"
    },
    {
      "AccountName": "TESTBOT2",
      "Password": "PASSWORD",
      "GmLevel": 0,
      "Purpose": "Background test bot — primary"
    },
    {
      "AccountName": "SHODAN",
      "Password": "PASSWORD",
      "GmLevel": 6,
      "Purpose": "Production GM liaison + test director (Spec/03 invariant)"
    }
  ]
}
```

Files:

- `Config/accounts/test-realm.json` — every test-realm account, with
  documented purpose per account.
- `Config/accounts/prod-realm.json` — every prod-realm account, with
  documented purpose (faction/class/spec target).

`StateManagerSettings.json` references which file(s) to load.

### Character declaration

`Config/characters/` mirrors `Config/accounts/`. Each character ties to
exactly one account on exactly one realm:

```json
{
  "Realm": "Westworld-Test",
  "Characters": [
    {
      "AccountName": "TESTBOT1",
      "CharacterName": "Equinox",
      "Race": "Tauren",
      "Gender": "Male",
      "Class": "Warrior",
      "SpecHint": "Fury",
      "Faction": "Horde",
      "Purpose": "Combat / movement parity FG primary"
    }
  ]
}
```

Tests that need a specific account/character cite it by name, not by
"any FG bot." The fixture loads `Config/characters/test-realm.json`,
resolves the character row, and proceeds.

### Production roster shape

Prod roster size = 3,000 (target). The roster spec satisfies the
coverage rules in
[`Spec/05_PROGRESSION.md#rosterplanner`](05_PROGRESSION.md#rosterplanner):
faction split, class coverage, profession coverage, spec diversity,
PvP rank bands.

The roster is generated by a one-time bootstrap tool
(`tools/RosterGenerator/`) that takes a target count and produces
`Config/accounts/prod-realm.json` + `Config/characters/prod-realm.json`
with full coverage. Hand-edit afterward if needed; the generator runs
again only when the prod count changes.

### Test-data fixture acknowledgment

As of 2026-05-11, the existing LiveValidation fixture uses dynamic
account/character naming patterns (TESTBOT1/2, GATHFG1/BG1, etc.) and
some tests treat character selection as flexible. This is technical
debt against this spec. The migration plan: every test rewrites to use
named characters from `Config/characters/test-realm.json`. Tracked as
a slot under
[`Plan/04_PHASE3_UI_DEFAULT.md`](../Plan/04_PHASE3_UI_DEFAULT.md)
because the UI-as-default-app rewrite touches the same fixture surface.

## OnDemand reserved-pool

The OnDemand engine (see
[`Plan/03_PHASE2_ONDEMAND_ENGINE.md`](../Plan/03_PHASE2_ONDEMAND_ENGINE.md))
operates over a **reserved pool**. The pool is ~80 accounts (Alterac
Valley size; the largest single-activity demand) on either realm.

Reserved-pool accounts are declared in
`Config/accounts/<realm>-ondemand-pool.json` with no fixed character
assignments — when an OnDemand activity launches, StateManager creates
or recycles characters on those accounts to match the activity's
requirements (faction, class, spec). At activity end, characters are
deleted or held for the next request.

The reserved pool is **siloed** from the autonomous progression
roster: they never participate in autonomous progression, are not
counted in `wwow.statemanager.bots.online` for progression panels, and
their gear/loot is ephemeral.

## Hot-reload account/character add

When the operator launches a new OnDemand activity via UI, the chosen
config may require accounts/characters not yet declared. StateManager:

1. Receives the new activity launch request with the activity's
   `Config/activities/<id>.json` file.
2. Resolves required accounts/characters from the reserved pool.
3. If the pool is short, creates new accounts via realmd SOAP and adds
   them to `Config/accounts/<realm>-ondemand-pool.json` (atomic write).
4. Hot-reload broadcasts the config change so all subscribers see the
   new accounts.
5. Spawns/recycles characters per the activity's role template.
6. Outfits (loadout), parties, teleports, hands off to human.

This is the hot-reload account-add path. The dev/test realm sees this
flow most; production usually has a steady-state pool.
