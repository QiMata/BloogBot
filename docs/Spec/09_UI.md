# Spec 09 — UI

## Surface

One UI: `UI/WoWStateManagerUI/` — WPF (MVVM, .NET 8). The Aspire AppHost
(`UI/Systems/Systems.AppHost/`) is dev-only orchestration, not a UI.

**The UI is the solution's default startup project.** Pressing F5 in
Visual Studio (or running the published exe) launches the UI. The UI
spawns and silently connects to a running `WoWStateManager.exe` — the
operator does not start StateManager manually.

There is **no HTTP layer**. The UI uses the existing protobuf/TCP IPC
to StateManager (port 8088). The UI implements an
`IStateManagerClient` that wraps the protobuf calls in MVVM-friendly
async methods. Only one UI process connects at a time — single source
of truth for config edits (per decision 2026-05-12 Q11).

## Startup contract

1. UI launches.
2. UI checks for a running StateManager on port 8088.
3. If none, UI launches `WoWStateManager.exe` as a child process with
   stdout/stderr piped to a UI log panel.
4. UI handshakes (`ProtocolHandshake` per
   [`Spec/08_PROTOCOLS.md`](08_PROTOCOLS.md)) and starts the polling
   loops.
5. On UI close, StateManager keeps running (so tests and headless ops
   can survive UI close) — or the UI offers a "Shut down StateManager
   too" prompt on exit. Default: leave running.

## Dashboard tab — the single operator surface

The existing Dashboard tab is the operator console. It contains:

### Service health row

Live status (green/yellow/red) for: realmd, mangosd, SOAP, StateManager,
PathfindingService, SceneDataService, DecisionEngineService,
PromptHandlingService.

Each service shows: state, uptime, git SHA, schema version, last error.

### Bot population panel

Counts by:

- Total online / offline / disconnected / crashed.
- By faction (Horde / Alliance).
- By mode (FG / BG).
- By level band (1-9 / 10-19 / 20-29 / 30-39 / 40-49 / 50-59 / 60).
- By status (idle / leased / progressing / failing).

Click-through shows the per-bot snapshot summary.

### Activities panel

- **OnDemand activity instances** (one row per active instance).
  Each row: catalog row, current stage (Spawning/Outfitting/Partying/
  Travelling/Engaged/TearDown), elapsed time, assigned bots with
  roles, human participant.
- **Autonomous activity events** — rolling list of recent group forms
  (dungeon parties, BG queue pops, raid forms) detected by
  coordinators.
- **Completed / failed counts** for OnDemand (rolling 24h + 7-day).
- **OnDemand stage-duration histogram** (Spawn→Engaged P50/P95/P99).

Click-through on an OnDemand instance shows the live launcher state
machine and exposes the **Terminate** button.

### On-Demand Activity catalog editor

- Catalog rows in a grid (filter by family, level range, faction).
- Per-row enable/disable.
- Per-row config editing (form-based, not raw JSON).
- Save validates against StateManager (authoritative); UI client-side
  validation is for immediacy only.
- Activity-request form: pick activity, set params (role, loot
  policy, start-when-ready), submit. The UI shows the resolution
  response (selected bots, ETA, legality warnings) before the request
  is committed.

### Pathfinding panel

- Queue depth, latency P50/P95/P99.
- Timeouts (rolling 24h).
- Route-pack hits/bypasses by pack id.

### Error triage panel

Top-N normalized errors by count, with labels for service, activity,
reason. From the taxonomy in
[`Spec/12_ERROR_TAXONOMY.md`](12_ERROR_TAXONOMY.md). Click-through shows
the most recent 50 occurrences with correlation ids for log lookup.

### Connection churn panel

Connect/disconnect counters by service and normalized reason. Surfaces
the noisy categories suppressed by the logging profile.

### Logging health panel

Suppressed log counts by category. Surfaces noisy log streams that the
operator might want to permanently suppress or promote to alerts.

### Long-term performance history panel

This is the new operator surface (per 2026-05-12 Q19): "show me how the
automated bots have been doing over the last few hours."

Time-series charts (Grafana-embedded or built-in charts) for:

- **Progression:** XP/hour gained per faction × level band; gold earned
  per bot; gear-acquisition rate (BoE pickups, vendor purchases).
- **Combat:** kills per hour per spec; death rate per zone;
  combat-resolved encounters vs wipes.
- **Economics:** AH listing count over time; AH bid/buy rate; median
  price per tracked item; gold velocity.
- **Travel:** average travel-leg duration; transport-missed count;
  path P99; stuck incidents.
- **Server health:** mangosd CPU/memory; realmd connection count; DB
  IOPS.

Resolution: 1-minute samples for the last 6 hours; 1-hour samples for
last 7 days. Operator-configurable window.

Data source: StateManager aggregates from in-process counters + reads
Prometheus (when running) via summary API. The UI does not query
Prometheus directly (per decision R3).

### Server actions panel

- Restart StateManager (with confirmation prompt).
- Restart any single service.
- Reload config from disk (without UI edit).
- Trigger snapshot bulk fetch.
- **Launch a new OnDemand Activity** — opens the activity request
  form. Per-activity config files live at `Config/activities/<id>.json`
  and are listed here; the operator selects one, the request form
  loads with that config pre-filled (level, role, prereqs).
- **View FG screenshot on failure** — when a bot has logged a failure
  in the last 5 minutes, its most recent failure-event screenshot is
  available. Read-only; overwritten per run.

## Hot reload flow (UI side)

1. User edits a config in `ActivityConfigView`.
2. Client-side validation passes; user clicks Save.
3. `ActivityConfigViewModel.SaveCommand` calls
   `StateManagerClient.SaveActivityConfigAsync(payload)`.
4. StateManager schema-validates; on success, atomic-writes the file
   and broadcasts `ConfigChangedEvent`.
5. UI receives `SaveAck` with the resulting state.
6. UI refreshes the affected views.
7. If any subscriber failed ACK, UI receives `SaveAck { Rolledback: true,
   FailedSubscribers[] }` and surfaces the rollback to the user.

## Activity request flow (UI side)

1. User opens `ActivityRequestView`.
2. Picks activity, location, level range; sets optional params.
3. UI calls `RequestActivityAsync(payload)` — StateManager performs
   legality validation and bot selection in-flight, returns the
   resolution.
4. UI shows: selected bots, missing role wait state, travel ETA,
   legality warnings.
5. User confirms; UI commits the request.
6. UI subscribes to the resulting `ActivityInstance` and renders its
   lifecycle in `ActivitiesSummary`.

## Polling cadence

| Panel | Default poll |
|---|---|
| Service health | 5 s |
| Bot population | 2 s |
| Activities (active + queue) | 1 s |
| Pathfinding stats | 5 s |
| Error triage | 10 s |
| Connection churn | 10 s |
| Logging health | 30 s |

Cadence is configurable per panel; the user can pause polling.
