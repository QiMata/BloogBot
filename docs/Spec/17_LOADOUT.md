# Spec 17 — LoadoutSpec

## Overview

`LoadoutSpec` is the single-message description of everything a character
needs to become raid-ready (or activity-ready) before the activity starts.
It is the contract handed from **StateManager** (writer) to **BotRunner**
(reader) exactly once per bot via `ObjectiveType.APPLY_LOADOUT`. The bot owns
execution pacing and reports progress through
`WoWActivitySnapshot.LoadoutStatus` (`LOADOUT_NOT_STARTED →
LOADOUT_IN_PROGRESS → LOADOUT_READY | LOADOUT_FAILED`). Empty/zero-valued
fields mean "don't change" so the spec is additive.

There are three load-bearing readers/writers:

- **`CharacterSettings.Loadout`** (`LoadoutSpecSettings` POCO) is the
  JSON-friendly source of truth that lives in
  `Config/characters/<realm>.json` and is hand-edited or generated by
  fixtures. Per
  [`Spec/02_STATEMANAGER.md`](02_STATEMANAGER.md), StateManager hot-reloads
  this file.
- **`LoadoutTask`** in `Exports/BotRunner/Tasks/LoadoutTask.cs` is the
  online, in-character executor. On `APPLY_LOADOUT` it builds a plan and
  dispatches GM commands (`.learn`, `.setskill`, `.additem`, `.additemset`,
  `.levelup`) through the bot's own chat sink one at a time, advancing
  only when each step is satisfied via snapshot polling or a matching
  `IWoWEventHandler` ack.
- **Phase 2's OnDemand `Outfitting` stage** (see
  [`Plan/03_PHASE2_ONDEMAND_ENGINE.md`](../Plan/03_PHASE2_ONDEMAND_ENGINE.md))
  composes a `LoadoutSpec` per activity (`Config/activities/<id>.json`)
  for every reserved-pool bot it allocates to the human's group, runs the
  spec through the `LegalityFixupPlan` for any extra GM-command fixes
  (lockout reset, attunement grants, faction patches), then hands the
  composed spec off via `APPLY_LOADOUT` before the party teleports to
  staging.

Out-of-band fields (`HonorRank`, `FactionReps`, `CompletedQuestIds`,
`TalentTemplate`) are intentionally **not** executed by `LoadoutTask` —
they are applied offline by the coordinator before the bot logs in (see
§5 "Two execution paths"). The POCO carries them so a single config row
fully describes the prep, not because the online executor consumes them.

## GM-command translation

Every command below is dispatched verbatim with the literal placeholders
substituted (no shell quoting). The dispatch channel column tells you
whether the command leaves StateManager (SOAP), leaves the coordinator
(MySQL), or is sent by the bot itself (bot chat — required when the
command depends on `GetSelectedPlayer()`).

Substitutions used throughout:

- `<bot>` — the bot's WoW character name (not account name); resolved by
  the coordinator before dispatch.
- `<id>`, `<value>`, `<level>`, `<delta>`, `<count>`, etc. — the
  corresponding POCO field value.

### Staging teleport (not a `LoadoutSpec` field — completes the prep)

After the loadout is `LOADOUT_READY`, the launcher teleports the party
to staging. The two supported forms are:

- Named tele location (preferred when defined in MaNGOS `command`):
  `.tele name <bot> <namedLocation>` — confirmed via
  `LiveBotFixture.BotChat.cs:790` (`TeleportToNamedAsync`).
- Coordinate-based (used when the staging coord is not a registered
  tele): bot chat `.go xyz <x> <y> <z> <mapId>` — confirmed via
  `LiveBotFixture.GmCommands.cs:161` (`TeleportAsync`).

`.teleport name <bot> <mapId> <x> <y> <z>` is **not** registered in this
MaNGOS build and must not be used (see comment at
`LiveBotFixture.GmCommands.cs:151`).

## Two execution paths

The same `LoadoutSpec` JSON document drives both execution paths; the
mode the bot is in selects how each field is fulfilled.

### Test mode / OnDemand `Outfitting` stage (every field GM-driven)

Used by:

- `OnDemandActivityLauncher`'s `Outfitting` stage (Phase 2; see
  [`Plan/03_PHASE2_ONDEMAND_ENGINE.md`](../Plan/03_PHASE2_ONDEMAND_ENGINE.md))
  for any bot allocated to a human's group.
- `Tests` mode in StateManager — every LiveValidation fixture treats
  prep as GM-driven (`LiveBotFixture` is the canonical execution
  surface via SOAP on port 7878).

In this mode every field is fulfilled via GM commands:

1. **Pre-login (coordinator)**: `HonorRank` (MySQL),
   `FactionReps` (`.modify rep`), `CompletedQuestIds` (`.quest add` +
   `.quest complete`), `TalentTemplate` (`.reset talents` + talent
   grants).
2. **Login + world entry**: StateManager dispatches a single
   `APPLY_LOADOUT` action carrying the proto (`AutomatedModeHandler.cs:64`
   uses `LoadoutSpecConverter.BuildApplyLoadoutAction`).
3. **Online (`LoadoutTask`)**: spells, skills (incl. riding), item set,
   supplemental items, equip items, elixirs, target level — in that
   order (`LoadoutTask.BuildPlan`, `LoadoutTask.cs:240`).
4. **Staging teleport**: `.tele name <bot> <staging>` (or `.go xyz`).
5. Bot reports `LOADOUT_READY` via `WoWActivitySnapshot`; activity starts.

### Automated mode (organic where possible, GM where forced)

Used by `AutomatedModeHandler` for non-OnDemand, long-running bots
(`Services/WoWStateManager/Modes/AutomatedModeHandler.cs:39`). The bot is
expected to gain progression naturally; `LoadoutSpec` is the *target*,
not a fast-forward script. The handler still dispatches `APPLY_LOADOUT`
exactly once per world-entry when `Loadout` is set
(`AutomatedModeHandler.cs:64`), but the underlying executor differs by
field:

- `SpellIdsToLearn`, `MountSpellId` — `TrainerAgent` walks the bot to a
  class/profession trainer (`Exports/BotRunner/Tasks/TrainerVisitTask.cs`,
  `Exports/BotRunner/Tasks/TrainSpellTask.cs`) instead of `.learn`.
- `TargetLevel` — real XP via questing and grinding
  (`Exports/BotRunner/Tasks/Progression/LevelUpTrainerTask.cs`).
- `RidingSkill`, `Skills[]` — trained at the appropriate NPC; `.setskill`
  is only a force-path for tests.
- `EquipItems[]`, `SupplementalItemIds[]` — sourced from drops, quest
  rewards, vendor, AH; not `.additem`.
- `ArmorSetId` — generally unused outside test mode.
- `ElixirItemIds[]` — used pre-engagement via the consumable buff
  pipeline, but the items must come from inventory the bot earned.
- `FactionReps[]`, `CompletedQuestIds[]`, `TalentTemplate`, `HonorRank` —
  still bootstrapped offline by the coordinator (these cannot be earned
  fast enough to be useful as gating data).

Per the `UseGmCommands` flag on `CharacterSettings` (`CharacterSettings.cs:170`),
a long-running bot may be granted GM-shortcut rights at runtime when an
operator explicitly forces an activity; that flag is the override toggle
between these two modes for an individual bot.

## Validation rules and failure modes

All failures normalize to the `FailureReason` enum in
[`Spec/12_ERROR_TAXONOMY.md`](12_ERROR_TAXONOMY.md). The mapping is:

| Trigger | Surface | `FailureReason` | Detail string carries |
|---|---|---|---|
| `Loadout` JSON fails `Newtonsoft.Json` parse (unknown property when strict, malformed array, wrong type) | StateManager `ConfigLoader` at startup / hot-reload | `catalog_invalid` | offending file path + JSON line/column |
| `SpellIdsToLearn` contains an id that returns `"Spell ID <id> does not exist"` from `.learn` | `LearnSpellStep` after `MaxRetriesPerStep = 20` retries (`LoadoutTask.cs:36`) | `task_unrecoverable` | `"step 'learn spell <id>' exceeded 20 retries without being satisfied"` (`LoadoutTask.cs:117`) |
| `EquipItems[i].ItemId` not present in `mangos.item_template` (`.additem` SOAP fault) | `AddItemStep` / `EquipItemStep` | `item_unavailable` | item id + faultstring from SOAP |
| `EquipItems[i].ItemId` exists but cannot be located in bags after retries | `EquipItemStep.TryExecute` returns false until retry budget exhausts | `task_timeout` | `"step 'equip item <id>' exceeded 20 retries"` |
| `Skills[i].SkillId` not in `mangos.skill_line_ability` (`.setskill` SOAP fault) | `SetSkillStep` | `server_rejected` | skill id + faultstring |
| `FactionReps[i].FactionId` not in `mangos.faction_template` (`.modify rep` SOAP fault) | coordinator pre-login | `missing_reputation` | faction id + faultstring |
| `CompletedQuestIds[i]` not in `mangos.quest_template` (`.quest add` SOAP fault) | coordinator pre-login | `server_rejected` | quest id + faultstring |
| `staging` named-location key (from the activity's `Config/activities/<id>.json`) does not resolve via `.tele name` | OnDemand launcher post-loadout staging step | `enter_world_failed` | named location + raw SOAP response |
| Bot logged off mid-loadout (disconnect, crash, queue kick) | `LoadoutTask` plan still resident on bot; coordinator observes `socket_disconnect_unexpected` first | `bot_disconnected` | account name + last completed step index |
| `APPLY_LOADOUT` enqueue rejected by listener (dead/ghost guard, capacity) | `AutomatedModeHandler.cs:67` resets the dispatch gate and retries on next snapshot | n/a (retry path) | warning only |

33. `.additem 13454` — Greater Arcane Elixir, then `UseItemTask`
34. `.additem 20007` — Mageblood Potion, then `UseItemTask`
35. `.levelup <60 - currentLevel>` — runs last so item-required-level checks already passed.

**Staging teleport** (post-`LOADOUT_READY`)

36. SOAP: `.tele name Furyguts ragefirechasm` (or `.go xyz` to the configured staging coord).

`LoadoutTask` emits a `CommandAckEvent` (`Pending` + `Success/TimedOut`)
for every step that goes through the chat sink, and reports
`LOADOUT_READY` once every step is satisfied.
