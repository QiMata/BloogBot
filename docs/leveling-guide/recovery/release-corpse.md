# Release Corpse — `CORPSE` → `CMSG_REPOP_REQUEST` → ghost form + graveyard teleport (single-opcode flow)

Foundational recovery cycle. Sibling to [corpse-run.md](./corpse-run.md): where corpse-run.md describes the *full* death → release → reclaim/spirit-heal/`.unstuck` cycle, this doc isolates the **release-only** transition — the single-opcode `CMSG_REPOP_REQUEST` dispatch that flips `DeathState = CORPSE` → `DEAD` (ghost form, `PLAYER_FLAGS_GHOST = 0x10`) and triggers the server-side graveyard teleport — so the shipped `Exports/BotRunner/Tasks/ReleaseCorpseTask.cs` (status `done` per [Plan/Activities/recovery.md](../../Plan/Activities/recovery.md)) has a single authoritative DE/snapshot contract to point at. This is the foundational *first* leg of any recovery cycle: every downstream Activity ([corpse-run.md](./corpse-run.md) reclaim, [spirit-healer.md](./spirit-healer.md) durability-tax branch, [stuck-recovery.md](./stuck-recovery.md) R14 "I am hopeless" flip) assumes the bot has already released.

The server-side state machine `WorldSession::HandleRepopRequestOpcode` lives at [`D:/MaNGOS/source/src/game/Handlers/MiscHandler.cpp:48-68`](D:/MaNGOS/source/src/game/Handlers/MiscHandler.cpp); the wire-side write executes `BuildPlayerRepop()` ([`Player.cpp:5040-5094`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) then `ScheduleRepopAtGraveyard()` ([`Player.cpp:5442-5448`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) → `RepopAtGraveyard()` ([`Player.cpp:5450-5505`](D:/MaNGOS/source/src/game/Objects/Player.cpp)). The 1.12.1 opcode is `CMSG_REPOP_REQUEST = 346 (0x15A)` at [`D:/MaNGOS/source/src/game/Protocol/Opcodes_1_12_1.h:349`](D:/MaNGOS/source/src/game/Protocol/Opcodes_1_12_1.h). The task is the simplest in the Recovery family — 19 lines, one tick, one packet.

## Overview

A character with `DeathState == CORPSE` (set by `Player::KillPlayer` at [`Player.cpp:5161-5179`](D:/MaNGOS/source/src/game/Objects/Player.cpp), which arms `m_deathTimer = CORPSE_REPOP_TIME = 360 000 ms` per [`Player.h:66`](D:/MaNGOS/source/src/game/Objects/Player.h) and sets the `PLAYER_FIELD_BYTE_RELEASE_TIMER` bit at [`Player.cpp:5167`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) sends `CMSG_REPOP_REQUEST` with an empty payload. Server-side `HandleRepopRequestOpcode` gates on `!_player->IsAlive() && !_player->HasFlag(PLAYER_FLAGS, PLAYER_FLAGS_GHOST)` ([`MiscHandler.cpp:52-53`](D:/MaNGOS/source/src/game/Handlers/MiscHandler.cpp)) — i.e. the player must be dead AND not already a ghost. If `DeathState == JUST_DIED` (the race window where the bot was killed between two server-side world updates), the handler force-runs `KillPlayer` first to advance to `CORPSE` ([`MiscHandler.cpp:60-64`](D:/MaNGOS/source/src/game/Handlers/MiscHandler.cpp)) before invoking `BuildPlayerRepop`.

`BuildPlayerRepop` performs eight observable mutations in order: (1) BG-only `CastSpell(this, 2584, true)` "Waiting to Resurrect" at [`Player.cpp:5043-5044`](D:/MaNGOS/source/src/game/Objects/Player.cpp); (2) `RemovePet(PET_SAVE_REAGENTS)` — pet is dismissed and reagents refunded; (3) NE-only `CastSpell(this, 20584, true)` — wisp form transform + speed buff at [`Player.cpp:5049-5050`](D:/MaNGOS/source/src/game/Objects/Player.cpp); (4) `CastSpell(this, 8326, true)` — `SPELL_AURA_GHOST` + speed buffs at [`Player.cpp:5051`](D:/MaNGOS/source/src/game/Objects/Player.cpp) — this is the aura that *visually and mechanically* makes the player a ghost (`PLAYER_FLAGS_GHOST = 0x00000010` is set as a side effect); (5) `CreateCorpse()` ([`Player.cpp:5065 / 5181-5199`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) — a `Corpse*` object is materialised at the player's pre-release XYZ, with `CORPSE_RESURRECTABLE_PVE` or `CORPSE_RESURRECTABLE_PVP` type per the `PLAYER_EXTRA_PVP_DEATH` flag; (6) `SetHealth(1)` — HP transitions `0 → 1` (the ghost form's nominal HP); (7) `SetWaterWalking(true)` + `SetRooted(false)` — ghost-form movement properties; (8) `SendCorpseReclaimDelay()` + `corpse->ResetGhostTime()` — arms the corpse-reclaim cooldown table (see [corpse-run.md](./corpse-run.md) R4 for the 30/60/120 s delay curve); finally `m_deathTimer = 0` + `SetDeathState(DEAD)` at [`Player.cpp:5092-5093`](D:/MaNGOS/source/src/game/Objects/Player.cpp).

`ScheduleRepopAtGraveyard` sets `m_repopAtGraveyardPending = true` if `IsInWorld() && GetSession()->IsConnected()` ([`Player.cpp:5442-5448`](D:/MaNGOS/source/src/game/Objects/Player.cpp)); otherwise it runs `RepopAtGraveyard()` synchronously. The pending flag is consumed inside `Player::Update` at [`Player.cpp:1734-1738`](D:/MaNGOS/source/src/game/Objects/Player.cpp) — gated on `!HasPendingMovementChange()` so the teleport does not race an in-flight movement opcode. `RepopAtGraveyard()` ([`Player.cpp:5450-5505`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) selects the closest `WorldSafeLocsEntry` via either `bg->GetClosestGraveYard(this)` for BG maps ([`Player.cpp:5460-5466`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) — same BG-vs-PvE branch cited by [spirit-healer.md](./spirit-healer.md) R12 — or `sObjectMgr.GetClosestGraveYard(GetPositionX(), GetPositionY(), GetPositionZ(), GetMapId(), GetTeam())` for PvE ([`Player.cpp:5468`](D:/MaNGOS/source/src/game/Objects/Player.cpp)), then `TeleportTo(ClosestGrave->map_id, x, y, z, orientation, TELE_TO_NOT_UNSUMMON_PET, ...)` ([`Player.cpp:5484`](D:/MaNGOS/source/src/game/Objects/Player.cpp)). For clients > 1.7.1 the orientation comes from `sObjectMgr.GetWorldSafeLocFacing(ClosestGrave->ID)` so the ghost spawns facing the Spirit Healer at the destination graveyard ([`Player.cpp:5475-5479`](D:/MaNGOS/source/src/game/Objects/Player.cpp)).

Critically, if the bot does NOT send `CMSG_REPOP_REQUEST`, the server **auto-fires** the same `BuildPlayerRepop + ScheduleRepopAtGraveyard` path after `m_deathTimer` decays to zero — the auto-release branch at [`Player.cpp:1697-1704`](D:/MaNGOS/source/src/game/Objects/Player.cpp) inside `Player::Update`. Default budget = `CORPSE_REPOP_TIME = 6 * MINUTE * IN_MILLISECONDS = 360 000 ms` ([`Player.h:66`](D:/MaNGOS/source/src/game/Objects/Player.h)) — gated additionally by `!GetMap()->Instanceable()` per the conditional compile at [`Player.cpp:1693-1696`](D:/MaNGOS/source/src/game/Objects/Player.cpp), so dungeon/raid maps DO auto-release; only non-instanceable maps gate on the `!Instanceable()` branch under the older client-build define. The engine's idempotency contract (R6 below) is built around this auto-fire window.

The client-side packet path is `IDeadActorNetworkClientComponent.ReleaseSpiritAsync(CancellationToken)` at [`DeadActorClientComponent.cs:69-95`](../../../Exports/WoWSharpClient/Networking/ClientComponents/DeadActorClientComponent.cs) — empty 0-byte payload, `await _worldClient.SendOpcodeAsync(Opcode.CMSG_REPOP_REQUEST, Array.Empty<byte>(), ct)`. `WoWSharpObjectManager.ReleaseSpirit()` at [`Exports/WoWSharpClient/WoWSharpObjectManager.Movement.cs:1185-1190`](../../../Exports/WoWSharpClient/WoWSharpObjectManager.Movement.cs) is the BG entry point (fires the opcode); the FG entry point invokes Lua `RepopMe()` per [Plan/Activities/recovery.md §ReleaseCorpseTask:85-90](../../Plan/Activities/recovery.md). The task is one tick: `ObjectManager.ReleaseSpirit(); PopTask("ReleaseSent");` ([`Exports/BotRunner/Tasks/ReleaseCorpseTask.cs:13-18`](../../../Exports/BotRunner/Tasks/ReleaseCorpseTask.cs)).

The live test anchor is `BotRunner.Tests.LiveValidation.DeathCorpseRunTests.Death_ReleaseAndRetrieve_ResurrectsBackgroundPlayer` at [`Tests/BotRunner.Tests/LiveValidation/DeathCorpseRunTests.cs:52`](../../../Tests/BotRunner.Tests/LiveValidation/DeathCorpseRunTests.cs); also exercised by `SpiritHealerTests.SpiritHealer_Resurrect_PlayerAliveWithSickness` at [`Tests/BotRunner.Tests/LiveValidation/SpiritHealerTests.cs:36`](../../../Tests/BotRunner.Tests/LiveValidation/SpiritHealerTests.cs) — line 58 dispatches `ObjectiveType.ReleaseCorpse` and line 60-66 polls `IsGhostSnapshot` within 15 s.

## Failure modes

| Failure | Detection | Recovery |
|---|---|---|
| **Send while alive** (R1 fired on an alive player — caller bypassed `CanReleaseSpirit` gate) | Server silently rejects at `MiscHandler.cpp:52` (`IsAlive()` returns true). No `SMSG_*` response. Post-send poll: `PlayerFlags & PLAYER_FLAGS_GHOST == 0` after 5 s | Re-check `DeathStateDetection.IsStrictAlive(player)` — if true, the bot did not actually die. Pop with reason `NotDead`. Likely cause: GM-level 6 bot at HP 0/1 under damage (per `DeathStateDetection.cs:59-61` warning) — the server never sets `UNIT_STAND_STATE_DEAD` for GMs, so the bot is not technically dead |
| **Send while already ghost** (R1 fired twice for the same death) | Server silently rejects at `MiscHandler.cpp:52` (`HasFlag(PLAYER_FLAGS_GHOST)` returns true). No SMSG response | Treat as a no-op; the first R1 succeeded. Verify via snapshot poll and proceed to R11 branch selection. Idempotency contract (R3) avoids this if the engine waits 5 s before retransmit |
| **JUST_DIED race** (R1 fired in the brief window between server tick that ran `KillPlayer` and the next tick that updated the player) | Server-side handler at `MiscHandler.cpp:60-64` force-runs `KillPlayer()` to advance `JUST_DIED → CORPSE` before processing the repop | No engine action needed — the server handles the race. Log noise: `LOG_LVL_DEBUG "HandleRepopRequestOpcode: got request after player ... was killed and before he was updated"` |
| **No graveyard on map** (`GetClosestGraveYard` returns null) | Post-R5 poll: `player.Position` unchanged after R4 (ghost flag set but no teleport). Map has no `game_graveyard` row | "Stuck ghost" branch per [corpse-run.md](./corpse-run.md) Failure-modes table. Issue `.unstuck` (corpse-run.md R7) — on success the bot teleports to faction-fallback graveyard (Westfall ID 4 / Barrens ID 10) per `D:/MaNGOS/source/src/game/Commands/TeleportCommands.cpp:1087`. Burns the 1 h `.unstuck` cooldown |
| **Pending-movement-change defer** (R5 delayed by `HasPendingMovementChange()` gate at `Player::Update:1734-1738`) | Post-R4 ghost-flag is set but `player.Position` does not change for > 1 tick (~50 ms) | Wait another tick. The teleport will fire when the in-flight movement opcode resolves. If position is still unchanged after 5 s, the bot is stuck in a movement-change deadlock — escalate per [stuck-recovery.md](./stuck-recovery.md) R3 |
| **Auto-fire race** (R1 sent at `t = CORPSE_REPOP_TIME - 100 ms`; server auto-fires before R1 reaches the handler) | Server auto-fires `BuildPlayerRepop` from `Player::Update:1700-1701`; the inbound R1 is silently rejected at `MiscHandler.cpp:52` (already-ghost gate). No client-visible signal | No engine action needed — the desired transition still occurred. Idempotency contract (R3) makes this a no-op. Log noise is minimal because `MiscHandler.cpp:52` returns silently (no LOG line) |
| **Combat-rez accepted mid-release** (R1 sent at `t = T`; `SMSG_RESURRECT_REQUEST` arrives at `t = T - 50 ms`; the rez accept fires at `t = T + 10 ms`) | Two server-side transitions in the same tick window. Resolution is deterministic: `BuildPlayerRepop` (release) runs first; then if the rez-response arrives BEFORE `SetDeathState(DEAD)` settles, the rez restore may layer onto an already-ghost player | Engine should treat this as an R8 violation (caller did not gate on combat-rez pending). Post-event snapshot will show one of three states: (a) ghost + corpse → run R11 branch (a)/(b); (b) alive + full HP → rez succeeded, no further action; (c) ghost + alive flag mid-flight → wait one tick, re-snapshot |
| **Transport-release alive resurrect** (bot was on a transport when ghost teleport fires; alive branch at `Player.cpp:5493-5497` runs `ResurrectPlayer(1.0f)`) | Post-R4 ghost-flag clears within one tick (bot is alive at 100% HP, not ghost) | Pop with reason `TransportResurrected`; do NOT push downstream Recovery tasks. Rare — only fires when the corpse-side branch at `Player.cpp:5482` evaluates `IsAlive()` true, which itself is unusual (alive corpse-run is the dead-state-flag-mismatch edge case) |

## Live-test acceptance

A live integration test exercising this Activity must satisfy R8 (clean cross-test state) by:

1. **Server-up assertion** — `ServerHealthcheck` returns 200 before the test starts; if not, FAIL FAST (root CLAUDE.md R4). Per Tests/CLAUDE.md "LiveValidation Test Pattern": skip if fixture not ready (`Skip.IfNot(_bot.IsReady, ...)`).
2. **Pre-test snapshot predicate** — bot resolved as BG action target (`LiveBotFixture.ResolveBotRunnerActionTargets(...).Single(t => !t.IsForeground)`), strict-alive AND `PlayerFlags & 0x10 == 0`. Shodan is director-only (cf. WWoW CLAUDE.md "Shodan: Production GM Liaison, Test Director") and never the test subject.
3. **Death trigger** — Shodan-directed `_bot.StageBotRunnerCorpseAt...Async(target.AccountName, ...)` (e.g. Razor Hill staging at [`DeathCorpseRunTests.cs:152-156`](../../../Tests/BotRunner.Tests/LiveValidation/DeathCorpseRunTests.cs) OR Valley spirit-healer staging at [`SpiritHealerTests.cs:45-47`](../../../Tests/BotRunner.Tests/LiveValidation/SpiritHealerTests.cs)). Poll snapshot until `IsStrictAlive == false AND DeathStateDetection.IsCorpse(player) == true` with timeout ≤ 5 s.
4. **Pre-release snapshot predicate** — assert `DeathStateDetection.CanReleaseSpirit(player) == true` AND `PlayerFlags & 0x10 == 0` AND `player.Bytes1[0] & 0xFF == UNIT_STAND_STATE_DEAD = 7` ([`DeathStateDetection.cs:23-36`](../../../Exports/BotRunner/Helpers/DeathStateDetection.cs)).
5. **Release dispatch** — `_bot.SendActionAsync(account, new ObjectiveMessage { ObjectiveType = ObjectiveType.ReleaseCorpse })`; assert `ResponseResult.Success` (per [`DeathCorpseRunTests.cs:166-168`](../../../Tests/BotRunner.Tests/LiveValidation/DeathCorpseRunTests.cs)).
6. **Ghost-flag predicate** — poll `(snapshot.Player?.PlayerFlags & 0x10) != 0` within **10 s** (corpse-run anchor uses 10 s at [`DeathCorpseRunTests.cs:170-174`](../../../Tests/BotRunner.Tests/LiveValidation/DeathCorpseRunTests.cs); spirit-healer anchor uses 15 s at [`SpiritHealerTests.cs:58-66`](../../../Tests/BotRunner.Tests/LiveValidation/SpiritHealerTests.cs)). FAIL with `"Never transitioned to ghost state"` (corpse-run-test message) or `"target never transitioned to ghost state after ReleaseCorpse."` (spirit-healer-test message).
7. **Health-transition predicate** — assert `player.Health == 1` (the `SetHealth(1)` mutation from `BuildPlayerRepop` at `Player.cpp:5074`). Note: current shipped tests do NOT poll this explicitly; it's a recommended hardening hook for the planned async refactor under S1.13.
8. **Aura predicate** — poll `player.Auras` contains spell `8326` (`SPELL_AURA_GHOST`); for `RACE_NIGHTELF` bots also assert `20584` (wisp transform). Same hardening-hook note as #7 — not in current tests but documented for parity with `Player.cpp:5049-5051`.
9. **Graveyard-teleport predicate** — poll `player.Position` differs from pre-release death position by ≥ 50 yd (typical graveyard-to-corpse distance for Razor Hill / Valley of Trials) within **5 s** after R6 fires. Bot should arrive at the closest `WorldSafeLocsEntry` for its `(MapId, Team)`. For BG mode (R10), the destination depends on faction-controlled-base state — query `bg->GetClosestGraveYard(this)` server-side rather than predicting client-side.
10. **Idempotency predicate** — fire R5 a second time (deliberate duplicate); assert no state change AND no error response (silent reject at `MiscHandler.cpp:52`). The task should pop with `"ReleaseSent"` both times.
11. **Disconnect guard** — mid-test client disconnect (per `WorldSocket.cpp:91`) must fail the test (R8) — do not paper over with `ReconnectTask` inside the release-corpse test (separate Activity, see [reconnect.md](./reconnect.md)).
12. **Screenshot capture** — per R11 in root CLAUDE.md, capture ForegroundBotRunner client screenshots at: (a) corpse state (HP 0, stand-state dead, no ghost flag), (b) post-release at death XYZ (ghost flag set, before graveyard teleport — sub-tick window may not capture), (c) post-graveyard-teleport (ghost flag set + position at nearest graveyard). Overwrite `Tests/artifacts/recovery/release-corpse/latest-{corpse,ghost,graveyard}.png`.
13. **Final state dump** — write `Snapshot.player` JSON (`PlayerFlags`, `Position`, `MapId`, `Health`/`MaxHealth`, `Auras`, `Bytes1`, `PlayerFieldBytes`) + the `CommandAckEvent` for the `ReleaseCorpse` dispatch + delta-snapshots for the 3 ticks pre/post-release to `Tests/artifacts/recovery/release-corpse/snapshot-final.json` so an agent can post-mortem the timing observables without re-running.
