# Stuck Recovery — Stale-Forward → L1/L2/L3 Escalation → `.unstuck` Fallback

Foundational recovery cycle. Every other Activity assumes the bot can recover from a stalled movement loop without operator intervention. Stuck recovery sits between the per-frame physics layer (`MovementController` stale-forward detector) and the per-tick route layer (`NavigationPath.ObserveMovementStuckRecovery` → forced `CalculatePath`), and exits to the in-game `.unstuck` chat command (spell `20939` Undying Soul, 1 h CD, gated `Level >= 10`, not-in-combat, not-in-BG, not-taxi-flying) when the route layer alone cannot resolve the stall. The server-side state machine `HandleUnstuckCommand` lives at `D:/MaNGOS/source/src/game/Commands/TeleportCommands.cpp:1061` — same source already cited by [corpse-run.md](./corpse-run.md) R7.

---

## Overview

`MovementController.ObserveStaleForwardAndRecover` (`Exports/WoWSharpClient/Movement/MovementController.cs:1720`) counts consecutive physics frames where the bot has horizontal-movement intent (`MOVEFLAG_FORWARD | MOVEFLAG_BACKWARD | MOVEFLAG_STRAFE_LEFT | MOVEFLAG_STRAFE_RIGHT`) but post-physics displacement < `STALE_FORWARD_DISPLACEMENT_EPSILON = 0.05f` per frame. After `STALE_FORWARD_NO_DISPLACEMENT_THRESHOLD = 15` consecutive stalled frames, `_consecutiveStuckLevels` increments and the controller fires `OnStuckRecoveryRequired(level, position)` (`MovementController.cs:177`). Levels are clamped at 3. Each fire bumps `WoWSharpObjectManager.MovementStuckRecoveryGeneration` (a monotonically-increasing counter exposed at `Exports/WoWSharpClient/WoWSharpObjectManager.cs:271` and reachable from any task via `IObjectManager.MovementStuckRecoveryGeneration` — declared at `Exports/GameData.Core/Interfaces/IObjectManager.cs:200`).

`NavigationPath.ObserveMovementStuckRecovery` (`Exports/BotRunner/Movement/NavigationPath.cs:2170`) polls that counter every tick via the `_stuckRecoveryGenerationProvider` (wired in `NavigationPathFactory.cs:132` and `Helpers/NavigationPathFactory.cs:28/53`). On a generation bump it runs `UpdateStuckRecoveryStationaryCounter` then forces a fresh `CalculatePath(... reason: NavigationTraceReason.MovementStuckRecovery)` (`NavigationPath.cs:2197`) and calls `PromoteWaypointAfterStuckRecovery` (`NavigationPath.cs:2198`). Vertical-layer escalation (D3, loop-25) at `NavigationPath.cs:1320-1358` flips the route into the safer-alternate rebuild branch after `VerticalLayerEscalationThreshold = 3` consecutive same-z-band vertical-layer-mismatch replans. The matching `StuckRecoveryTask` is `not-started` per [Plan/Activities/recovery.md](../../Plan/Activities/recovery.md) — current callers consume the event indirectly via `TravelTask.cs:935` (which reads `MovementStuckRecoveryGeneration` to gate retries) and the `NavigationPath` self-observation.

The `IsOnNavmesh` gate (S9.2 in [Plan/10_PARALLEL_BRM_BAKE.md:59](../../Plan/10_PARALLEL_BRM_BAKE.md) and tracked in [Plan/02_PHASE1_ACTION_TASK_FOUNDATION.md:62](../../Plan/02_PHASE1_ACTION_TASK_FOUNDATION.md)) is the **proposed but not-yet-shipped** `IObjectManager` probe that prevents an `L3 JumpRecover` action from depositing the bot onto an off-mesh polygon. The exit fallback is `ChatHandler::HandleUnstuckCommand` (`D:/MaNGOS/source/src/game/Commands/TeleportCommands.cpp:1061`); same gate-set as the dead-state fallback in [corpse-run.md](./corpse-run.md) R7.

| Failure | Detection | Recovery |
|---|---|---|

| **`.unstuck` 1-hour CD already burned** | `IsSpellReady(20939) == false` returned by R10/R11 guards | Fall back to `SpiritHealerTask` if reachable (dead path); else `LeaseReturnTask` (alive path). See `D:/MaNGOS/source/src/game/Commands/TeleportCommands.cpp:1068` for the spell-readiness gate and `TeleportCommands.cpp:1082-1083` for the 1 h `AddCooldown` |
| **R12 violation — jump on off-mesh poly** (bot performs L3 jump recover, lands off-mesh, subsequent `CalculatePath` returns `no_path` forever) | `NavigationPathResult.ResultTag == "no_path"` for `>= 3` consecutive `CalculatePath` calls right after an L3 fire AND distance from `_lastKnownGoodPosition > 5 yd` | Issue `.unstuck` immediately (skip further L3 jump cycles). | **Vertical-layer cascade** (D3 escalation fires per R7 but `vertical_layer_mismatch` keeps recurring after escalation — handoff §5 Unknown 2) | `_consecutiveVerticalLayerReplans` resets per R7 escalation but immediately re-arms within the same z-band on the next tick | Belt-and-suspenders already in place: `_lastVerticalLayerReplanTick = nowTick` set inside the R7 escalation block suppresses re-detection on the immediate next tick. | **Stuck while taxi-flying** | `player.MovementFlags & MOVEFLAG_ONTRANSPORT` AND on-taxi (path was set by `TakeFlightPathTask`) AND bot is stationary > 60 s | NOT a stuck-recovery case — the taxi flight finished or was interrupted server-side. ## Live-test acceptance

A live integration test exercising stuck recovery must satisfy R8 (clean cross-test state) by:

1. **Server-up assertion** — `ServerHealthcheck` returns 200 before the test starts; if not, FAIL FAST (R4 in root CLAUDE.md).
2. **Pre-test snapshot predicate** — `player.Health > 0 AND (player.MovementFlags & STALE_RECOVERY_MOVEMENT_MASK) == NONE` before triggering movement (clean idle start).
3. **Stuck trigger** — drive the bot into a known wall pocket (e.g., Razor Hill GY → due-N into the rocky cliff face) and issue `MoveToward` with a destination behind the rock. Poll until `MovementController._consecutiveStuckLevels >= 1` with timeout ≤ `(STALE_FORWARD_NO_DISPLACEMENT_THRESHOLD frames * frame_period_ms) + 500 ms` (~1 s at 50 Hz physics).
4. **L1 → L2 → L3 escalation predicate** — after each L fires, poll `MovementStuckRecoveryGeneration` increment and verify subsequent `NavigationPath` plan-version delta (proves R3 forced `CalculatePath` ran). Total L1→L3 window should fit within `STALE_FORWARD_SUPPRESS_AFTER_RECOVERY_MS + STALE_FORWARD_SUPPRESS_L2_MS + STALE_FORWARD_SUPPRESS_L3_MS + 3 * THRESHOLD_FRAMES ≈ 3 s`.
5. **`.unstuck` fallback predicate** — when L3 escalation does not produce displacement within 5 s, the test asserts the bot issues `.unstuck`. If alive: poll for `SPELL_AURA_KNOCKBACK_2` (spell `20939` Undying Soul) on `player.Auras` AND HP unchanged. If dead: poll for `HasAura(15007) == true` AND `player.Position` near `ClosestGrave` AND `player.PlayerFlags & PLAYER_FLAGS_GHOST == 0` (the unstuck-while-dead path resurrects via graveyard teleport).
6. **`IsOnNavmesh` gate test** (forward-looking) — when the proposed probe ships, assert `IObjectManager.IsOnNavmesh(player.Position) == true` after every L3 recovery action; if `false`, assert the recovery fell back to R12 (`.unstuck`) instead of `JumpRecover`.
7. **Disconnect guard** — mid-test client disconnect (per `WorldSocket.cpp:91`) must fail the test (R8) — do not paper over with `ReconnectTask` inside the stuck-recovery test (separate Activity).
8. **Screenshot capture** — per R11 in root CLAUDE.md, capture ForegroundBotRunner client screenshots at: (a) stuck-frame (L1 fire), (b) post-route-rebuild (R3 fire), (c) post-recovery (`MovementController._consecutiveStuckLevels == 0` after `NotifyExternalStuckRecoveryApplied`). Overwrite `Tests/artifacts/recovery/stuck-recovery/latest-{l1,replan,recovered}.png`.
9. **Final state dump** — write `Snapshot.player` + last 20 `NavigationExecutionSample` entries + last 5 `NavigationTraceSnapshot.LastReplanReason` values to `Tests/artifacts/recovery/stuck-recovery/snapshot-final.json`.

Existing test anchors: `Tests/WoWSharpClient.Tests/Movement/MovementControllerTests.cs` exercises the L1/L2/L3 *signal* path (recorded fixture parity, not live). `Tests/BotRunner.Tests/Movement/NavigationPathTests.cs` exercises the `ObserveMovementStuckRecovery` route-rebuild + D3 vertical-layer escalation (per loop-25 D3 IMPL — focused tests 113/0/7).
