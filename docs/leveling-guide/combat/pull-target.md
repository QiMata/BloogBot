# Pull Target — Target Validation → Pull-Method Selection → Range / LOS-Corner Setup → `CMSG_CAST_SPELL` or `CMSG_ATTACKSWING` Pull-Initiation → Aggro Confirmation → Hand-off to `PvERotationTask`

Foundational pre-engagement cycle. Every Combat Activity that targets a specific hostile mob (questing pulls, dungeon-trash chain pulls, raid trash, world-PvE grinds, BG flag-defender engages) routes through the per-profile `PullTargetTask` *before* the per-spec `PvERotationTask` takes over. The pull layer's job is: (1) **isolate** the intended target from adjacent linkable mobs (Hunter's Mark / Distract / LOS-corner-pull), (2) **establish** initial threat and aggro on a single chosen target before the bot is in melee range, (3) **hand off** to `PvERotationTask` once aggro is confirmed (target's `IsInCombat == true` AND `target.TargetGuid == player.Guid`). This doc covers the generic **pull** path; the sibling [melee-rotation.md](./melee-rotation.md) covers the melee swing cycle and [cast-spell.md](./cast-spell.md) covers the generic spell-cast cycle that pulls reuse for the pull spell itself.

The shipped runtime split is: per-profile `PullTargetTask` classes at `BotProfiles/<Spec>/Tasks/PullTargetTask.cs` (27 of 27 spec files exist on disk per [Plan/Activities/combat.md §PullTargetTask](../../Plan/Activities/combat.md); shape varies by archetype — see Overview below). Each derives from `BotRunner.Tasks.BotTask` ([`Exports/BotRunner/Tasks/BotTask.cs`](../../../Exports/BotRunner/Tasks/BotTask.cs)) and picks up the S1.0 async shim per R25 (`TickAsync` → `OnTick` → legacy `Update()`). The pull spell itself reuses the generic [cast-spell.md](./cast-spell.md) `CMSG_CAST_SPELL` send path via `ObjectManager.CastSpell(name)` at [`Exports/WoWSharpClient/SpellcastingManager.cs:204-239`](../../../Exports/WoWSharpClient/SpellcastingManager.cs); ranged auto-shot pulls (Hunters post-mark) reuse `StartRangedAttack` at [`SpellcastingManager.cs:389`](../../../Exports/WoWSharpClient/SpellcastingManager.cs) (which currently aliases to `StartMeleeAttack` — BG ranged-auto-shot opcode parity is a per-class skill follow-up, not S0.8 scope). Hunter pet pulls add `CMSG_PET_ACTION` via `PetAttackAsync` at [`Exports/WoWSharpClient/Networking/ClientComponents/CombatSpellNetworkClientComponent.cs:357-369`](../../../Exports/WoWSharpClient/Networking/ClientComponents/CombatSpellNetworkClientComponent.cs). Approach / LOS-corner / chase movement is delegated to the inherited `NavigateToward(...)` method on `BotTask`, which drives the `MovementController` MSG_MOVE family.

The server-side state machine is shared with the generic spell-cast path: `HandleCastSpellOpcode` at [`D:/MaNGOS/source/src/game/Handlers/SpellHandler.cpp:321`](D:/MaNGOS/source/src/game/Handlers/SpellHandler.cpp) → `Spell::prepare` → `Spell::CheckCast` → on launch, per-target processing in `Spell::DoAllEffectOnTarget` adds initial threat for hostile spells via the gate at [`D:/MaNGOS/source/src/game/Spells/Spell.cpp:1454-1462`](D:/MaNGOS/source/src/game/Spells/Spell.cpp) (`!m_spellInfo->IsPositiveSpell()` AND `!SPELL_ATTR_EX_NO_THREAT` AND `!SPELL_ATTR_EX2_NO_INITIAL_THREAT` → `m_caster->SetInCombatWithVictim(unit)` + `unit->SetInCombatWithAggressor(m_caster)` + `m_threatManager.addThreat(...)` accumulator). Threat-add itself runs through `ThreatManager::addThreat` at [`D:/MaNGOS/source/src/game/Threat/ThreatManager.cpp:391-424`](D:/MaNGOS/source/src/game/Threat/ThreatManager.cpp) (then `addThreatDirectly` at line 427 inserts the `HostileReference` into the `ThreatContainer`). The mob's AI runs `CreatureAI::AttackStart` once it sees the new hostile reference — `CreatureAI.cpp` drives the path-toward-aggressor sequence that the bot will observe as `nearbyUnits[<targetGuid>].targetGuid == player.Guid` AND `MovementFlags & MOVEFLAG_FORWARD` on the next snapshot.

## Overview

`PullTargetTask` is per-spec; the body shape varies by archetype. Five archetypes ship across the 27 profiles:

| Archetype | Spec exemplar | Pull spell(s) | Pull range | Body shape |
|---|---|---|---|---|
| **Hunter ranged + pet** | `HunterMarksmanship.Tasks.PullTargetTask` ([file](../../../BotProfiles/HunterMarksmanship/Tasks/PullTargetTask.cs)) | `HuntersMark` + auto-shoot (`StartRangedAttack`); `_petSent` gate for `Pet.Attack()` | 28 yd (Hunter's Mark range; `distanceToTarget < 28`) | Send pet → cast Hunter's Mark if missing → otherwise `StartRangedAttack` → pop + push `PvERotationTask` |
| **Caster ranged-pull (Frostbolt / Shadow Bolt / Curse of Agony)** | `MageFrost.Tasks.PullTargetTask` ([file](../../../BotProfiles/MageFrost/Tasks/PullTargetTask.cs)); `WarlockAffliction.Tasks.PullTargetTask` ([file](../../../BotProfiles/WarlockAffliction/Tasks/PullTargetTask.cs)) | Mage: `Frostbolt` (preferred) or `Fireball` (fallback); Warlock: `CurseOfAgony` (preferred) or `ShadowBolt`; Mage adds `+3 yd` per `Frost Channeling` talent rank (tab 3, talent 11) | Mage: `28 + GetTalentRank(3,11)*3` yd; Warlock: 27 yd | LOS-check (`Player.InLosWith(target) == true`) → 250 ms `Wait.For` stabilisation → `StopAllMovement()` → cast pulling spell → pop + push `PvERotationTask` |
| **Rogue stealth-open** | `RogueSubtlety.Tasks.PullTargetTask` ([file](../../../BotProfiles/RogueSubtlety/Tasks/PullTargetTask.cs)) | `Stealth` at 25 yd → `Ambush` (from-behind, if ready) → fallback `Garrote` (from-behind, non-Elemental) → fallback `CheapShot` | 25 yd (stealth-arm) → 3.5 yd (open) | Stealth-arm at 25 yd → close to ≤ 3.5 yd via `NavigateToward` → from-behind position check (`Player.IsBehind(target)`) → cast opener → pop + push `PvERotationTask` |
| **Warrior dungeon throw / Charge** | `WarriorFury.Tasks.PullTargetTask` ([file](../../../BotProfiles/WarriorFury/Tasks/PullTargetTask.cs)) | Dungeon (`MapId != 0 && MapId != 1`): `Throw` 5-30 yd, then walk-up wait 30 ticks; Outdoor: `Charge` 8-25 yd if `Player.InLosWith(target) == true` | Throw: 5-30 yd; Charge: 8-25 yd; melee close: <3 yd | Dungeon branch sets `_threwRanged` flag, casts Throw, waits 30 ticks for aggro; Outdoor branch casts Charge if in LOS, falls through to melee close at `< 3 yd`. Pop + push `PvERotationTask` on close. |
| **Plain melee close** | `PaladinProtection.Tasks.PullTargetTask` ([file](../../../BotProfiles/PaladinProtection/Tasks/PullTargetTask.cs)); used by Druid Feral, Warrior Protection, Paladin Retribution, and the placeholder body in most other plate/leather specs | None (body-pull / aggro-on-proximity) | `< 3 yd` melee close; aggro from mob's natural detection range | Validate target → close via `NavigateToward(target.Position)` → pop + push `PvERotationTask` on close |

All five share the same skeleton: (1) **early pop gates** — `target == null || target.Health <= 0 || target.TappedByOther` → `StopAllMovement` + `Pop()` (the canonical "abort the pull and reselect" exit); (2) **already-in-combat gate** — `Player.IsInCombat || Aggressors.Any()` → `StopAllMovement` + `Pop()` + `Push(new PvERotationTask(BotContext))` (covers the race where a different mob aggroed onto us mid-approach); (3) **range check** — measured via `Player.Position.DistanceTo(target.Position)`, used to decide cast / shoot / charge / melee-close vs. continued navigation; (4) **pull dispatch** — single tick that fires the chosen pull mechanism, immediately pops, and pushes the per-spec `PvERotationTask`; (5) **navigation fall-through** — when none of the above gates fire, drives `NavigateToward(target.Position)` and returns without dispatch. Hunter and Warlock variants add a 250 ms `Wait.For` stabilisation hold to allow `StopAllMovement` to flush before the cast (avoids the R6 movement-cancel from [cast-spell.md](./cast-spell.md) on Frostbolt / Shadow Bolt / Curse of Agony cast-time spells).

**LOS-corner pull pattern.** The "corner-of-room mob isolation" idiom — bot positions at the edge of a doorway, line-of-sight breaks the linked group, fires the pull spell on the target, then retreats around the corner so the target follows alone and adds reset because they cannot pathfind to LOS — is **not encoded as a dedicated task today**. The shipped behaviour is: (a) navigation pathfinder picks the route to "within `pullRange` yd of `target.Position` AND `Player.InLosWith(target) == true`" — this naturally lands the bot at the doorway-edge in dungeons because the mob's `Position` is past the corner and LOS-blocking geometry forces the path to terminate at the wall; (b) the pull dispatch fires (Frostbolt / Hunter's Mark / Throw); (c) the per-spec `PvERotationTask` resumes — there is no explicit "retreat around corner" step. The corner-pull *separation* relies on mob path-failure for adds (mob AI's `AttackStart` requires reachable target geometry; when adds can't path to the bot they revert to leash-and-reset per `CreatureAI.cpp`'s `AttackStart` → `MoveChase` failure path). Bot-side LOS-corner-retreat is a planned per-spec enhancement, not shipped — covered by [#Cross-References](#cross-references) → "Future: corner-pull retreat sub-rule".

**Pull-and-kite.** The Hunter ranged pull combined with `Pet.Attack()` is the canonical "send tank-pet, retreat caster" pattern. `ObjectManager.Pet.Attack()` ultimately sends `CMSG_PET_ACTION` with the pet-command payload constructed at [`CombatSpellNetworkClientComponent.cs:357-369`](../../../Exports/WoWSharpClient/Networking/ClientComponents/CombatSpellNetworkClientComponent.cs) (`PetAttackAsync(targetGuid, ct)`). The pet generates threat as the primary aggro magnet while the Hunter's auto-shot threat stays below pet threat (pet's `Growl` and base-attack threat scale to keep the Hunter under the 110%-melee / 130%-ranged pull threshold from [melee-rotation.md](./melee-rotation.md) R10). Bot-side kite intent (move backward while pet tanks) lives in `CombatRotationTask.StartKite` at [`Exports/BotRunner/Tasks/CombatRotationTask.cs:433-471`](../../../Exports/BotRunner/Tasks/CombatRotationTask.cs); the Hunter pull task does not itself kite — it pops and lets `PvERotationTask` decide when to engage `IsKiting` per per-spec gating.

## Failure modes

| Failure | Detection | Recovery |
|---|---|---|
| **Target tapped mid-approach** (another player / bot landed first hit; `target.TappedByOther == true` flag flips during the chase) | Per-tick check at R1 entry; `target.TappedByOther` reads from `nearbyUnits[<targetGuid>].dynamic_flags` (server `UNIT_DYNFLAG_TAPPED_BY_PLAYER` bit) | R1 fires: pop without dispatch. Caller reselects the next aggressor / quest objective. Bot does not contest the tap (loot rights are lost) |
| **Pull spell missed / resisted** (Frostbolt fizzles, Hunter's Mark resisted on level-difference target, Shadow Bolt full resist) | Server `SMSG_SPELL_LOG_MISS` arrives; `target.HasDebuff(<pullspell>) == false` AND `target.IsInCombat == false` AND `target.TargetGuid != player.Guid` post-dispatch | `PullTargetTask` is single-pull-and-done — control already transitioned to `PvERotationTask` (R13). The rotation layer's `EnsureTarget` will re-acquire on the next `Aggressors.Any() == false` cycle (15 s combat timeout per [melee-rotation.md](./melee-rotation.md) R14). Re-pull is not auto — caller must push a new `PullTargetTask` |
| **Adds pulled by cast / Hunter's Mark proc** (cast-spell aggro is **single-target**, but cleave AI on the target's linked-mob group can pull adds if proximity is too tight; Charge / Hunter's Mark sometimes proc nearby mobs via `creature_linking` DB rows) | Post-dispatch, `Aggressors.Count() > 1` AND multiple `nearbyUnits[*].targetGuid == player.guid`. R2 fires on the *next* `PullTargetTask` push attempt — rotation handles the multi-mob via R13 AoE branch in [melee-rotation.md](./melee-rotation.md) | Rotation-layer AoE branch (Whirlwind / Cleave / IntimidatingShout per [melee-rotation.md](./melee-rotation.md) R13); CC / kite via per-spec utility. Pull-corner geometry tuning is upstream (`ActivityResolver` / pathfinder) — bot does not re-route mid-pull |
| **LOS-corner pull failure** (intended doorway-edge LOS-break is geometrically incorrect; either bot does not stop at the doorway OR LOS clears past the doorway and adds aggro) | `Aggressors.Count() > expectedSinglePull` post-dispatch within ≤ 2 s | No bot-side mitigation in `PullTargetTask`; falls through to rotation-layer AoE / kite. LOS-corner-retreat-after-pull is a planned per-spec sub-rule, not shipped today (see Overview "LOS-corner pull pattern") |
| **Movement-cancel on cast-time pull** (Mage Frostbolt 2.5s cast; bot moved > 0.5 yd during cast — usually because `Wait.For` 250 ms hold was insufficient and a stale `MoveToward` from prior tick was still in-flight) | `SMSG_CAST_RESULT == SPELL_FAILED_MOVING`; `Player.IsCasting == false` instantly after dispatch; cast-time spell never landed | Mage / Warlock body's `StopAllMovement` + `Wait.For` 250 ms hold (R6); if still failing, length of the wait is the tuning knob. | **Charge LOS-fail (Warrior outdoor)** (`Player.InLosWith(target) == false` between bot and target despite being in 8-25 yd range — usually because target stands behind a wall or below a ledge) | R11 gate skips Charge silently; bot falls through to `NavigateToward(target.Position)` and the close-to-melee branch (`distance < 3` → push `PvERotationTask`) | Acceptable failure mode; bot just walks instead of charging. | **Stealth break before from-behind position (Rogue)** (`Stealth` armed at 25 yd but stealth broke during close — e.g., proximity to a wandering patrol or a passive detection mob) | At ≤ 3.5 yd, `Player.HasBuff(Stealth) == false` AND not `IsInCombat` (yet — broken-stealth is silent until aggro fires). ## Live-test acceptance

A live integration test exercising pull-target must satisfy R8 (clean cross-test state per root CLAUDE.md) by:

1. **Server-up assertion** — `ServerHealthcheck` returns 200 before the test starts; if not, FAIL FAST (root CLAUDE.md R4). `CombatLoopTests` does this via `Skip.IfNot(_bot.IsReady, _bot.FailureReason)` at [`CombatLoopTests.cs:45`](../../../Tests/BotRunner.Tests/LiveValidation/CombatLoopTests.cs).
2. **Pre-test snapshot predicate** — both bots strict-alive AND `Player.IsInCombat == false` AND `Aggressors.Any() == false` AND `Player.IsCasting == false` AND `Player.HasBuff(Stealth) == false`. `.gm off` required (per WWoW MEMORY: GM mode corrupts UnitReaction bits and breaks aggro / threat tests).
3. **Target acquisition predicate** — find a level-appropriate, non-tapped, non-friendly mob visible in `NearbyUnits` within 20 s. Assert `target.TappedByOther == false`, `target.Health > 0`, `target.Faction` is hostile to bot's faction (per `IsValidAttackTarget`).
4. **Pull dispatch** — for Hunter scenario `Pull_HuntersMarkLandsBeforeMelee`: declare `AssignedActivity = "QuestKill[BoarsAt(...)]"` via `StageBotRunner*Async` per [WWoW CLAUDE.md "Test Isolation Rules"](../../../CLAUDE.md). The bot's natural Activity-Objective-Task-Action cascade should produce `HunterMarksmanship.Tasks.PullTargetTask` for the engage step. Test does NOT construct a raw `ObjectiveMessage` (would bypass DecisionEngine — violates test-isolation rules).
5. **Hunter's Mark debuff landing predicate** — poll `target.HasDebuff(HuntersMark) == true` within 5 s of pull entry. Verify `nearbyUnits[<targetGuid>].auras` snapshot contains the Hunter's Mark aura.
6. **Pet attack predicate (Hunter)** — poll `Pet.TargetGuid == target.guid AND Pet.IsInCombat == true` within 3 s of dispatch. Verify the `CMSG_PET_ACTION` packet was issued (via packet-capture sidecar if available).
7. **Frostbolt-pull cast-time predicate (Mage variant)** — `Pull_FrostMageLandsBoltBeforeMelee`: poll `Player.IsCasting == true` within 200 ms of pull entry; poll `Player.IsCasting == false` AND `target.Health < target.MaxHealth` AND `target.HasDebuff(<frost slow>)` within `castTime + 500 ms`. Verify `MovementFlags == MOVEFLAG_NONE` for cast duration (R6 movement-flush).
8. **LOS-corner pull predicate (planned, not-started)** — `Pull_LosCornerIsolatesSingleMob`: stage bot at known LOS-corner geometry (e.g., the Wailing Caverns first-room doorway with a linked-mob group beyond); dispatch pull; assert `Aggressors.Count() == 1` post-dispatch (only the intended target engages, linked mobs reset). Geometry-specific anchor — depends on tile (XX,YY) bake data being correct (per WWoW pathfinding overhaul status).
9. **Charge pull predicate (Warrior outdoor)** — `Pull_WarriorChargeClosesGap_FromMidRange`: target at 15 yd with clear LOS; dispatch pull; assert `Player.MovementFlags & MOVEFLAG_CHARGING != 0` within 100 ms; assert `distance < 3` within 1 s; assert `target.HasDebuff("Charge Stun")` within 200 ms.
10. **Aggro-confirmation predicate** — within 5 s of dispatch, assert `target.IsInCombat == true AND target.TargetGuid == player.Guid AND Aggressors.Any(a => a.Guid == target.Guid) == true`. This is the implicit R17 deferred check.
11. **Task-stack transition predicate** — after dispatch, assert the bot's active task transitions from `PullTargetTask` → `PvERotationTask` within 2 ticks. Observable via `WoWActivitySnapshot.current_task_id` (Phase 2+) or `current_action` blip from `CastSpell` → `StartMeleeAttack` (current).
12. **Pull-failure detection (negative test)** — `Pull_RespawnedMobImmuneToPull_FallsThroughToRotation`: target an invulnerable / GM-immune mob; dispatch pull; assert the pull spell fires; assert `target.HasDebuff(<pullspell>) == false` after 2 s; assert rotation layer's `EnsureTarget` 15 s combat-timeout fires per [melee-rotation.md](./melee-rotation.md) R14.
13. **Disconnect guard** — mid-test client disconnect (per `WorldSocket.cpp:91`) must fail the test (R8) — do not paper over with `ReconnectTask` inside the pull-target test (separate Activity, see [../recovery/reconnect.md](../recovery/reconnect.md)).
14. **Screenshot capture** — per R11 in root CLAUDE.md, capture FG client screenshots at: (a) pre-pull (target acquired, distance > pullRange), (b) at-pull-range (`distance <= pullRange`, LOS clear), (c) post-dispatch (pull spell visible mid-air / pet running in / debuff applied), (d) on-aggro-confirm (mob running toward bot, `target.TargetGuid == player.Guid`). Overwrite `Tests/artifacts/combat/pull-target/latest-{prepull,atrange,dispatch,aggro}.png`.
15. **Final state dump** — write `Snapshot.player` (HP/Mana/Energy/Rage/MovementFlags/Auras/CombatReach) + `Snapshot.target` (HP/Auras/TargetGuid/IsInCombat) + last 20 `WoWActivitySnapshot` deltas (target HP curve + aura applications + pet state) + `ThreatTracker._threatByGuid` map snapshot to `Tests/artifacts/combat/pull-target/snapshot-final.json` for agent post-mortem.

Existing test anchors: closest is `BotRunner.Tests.LiveValidation.CombatLoopTests` at [`CombatLoopTests.cs:49`](../../../Tests/BotRunner.Tests/LiveValidation/CombatLoopTests.cs) which exercises target acquisition + chase + `StartMeleeAttack` (the BG counterpart of the pull path, but via `ObjectiveType.StartMeleeAttack` atomic dispatch — bypasses per-spec `PullTargetTask`).
