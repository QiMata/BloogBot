# Mage Teleport + Portal â€” Class Gate â†’ Spell+Cooldown Gate â†’ Reagent Gate â†’ Interrupt Gates â†’ 10s Cast â†’ spell_target_position Lookup â†’ Teleport / Portal-Creation

Third Travel sub-family after [`hearthstone.md`](hearthstone.md) (universal teleport-to-bind) and [`flight-path.md`](flight-path.md) (paid cross-continent spline). Where Hearth is **bind-locked once-per-hour** and Flight is **gold-cost ~3-4 minute server-spline**, Mage Teleport is **class-gated free instant-teleport to any of the 6 capitals** â€” Stormwind / Ironforge / Darnassus (Alliance) and Orgrimmar / Undercity / Thunder Bluff (Horde). The companion **Portal** spells (group-portal variant) let a mage open a 1-minute portal at their feet that any party member can click to teleport to the same capital â€” the only multi-player travel option short of warlock summon (iter-30+).

**Why this is `mage-teleport` and not `mage-portal`**: both spell families share the same DE-execution layer (cast 10s, server-side `TARGET_LOCATION_DATABASE` lookup, position-jump teleport detection) â€” the difference is **who-arrives-where**: self-teleport jumps the caster, portal creates a GameObject the party clicks. The shipped `MageTeleportTask` ([`Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs`](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs), 132 LOC) currently handles the **self-teleport** path; portal-creation routes through the same task with a different spell-id (per `MageTeleportData.cs`) but the caster doesn't teleport â€” the portal-object spawns and party-members invoke `MageTeleportTask`-equivalent on their end.

The shipped task is a **five-state machine** (`Check â†’ StopAndCast â†’ WaitForCast â†’ DetectTeleport â†’ Complete`) â€” mirrors [`UseHearthstoneTask`](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)'s shape (same `10000ms` cast, same `15000ms` teleport-detect timeout, same `100f` position-delta threshold) but adds three class/spell-gate predicates at the `Check` state: `Player.Class == Class.Mage`, `IsSpellReady(spellName)`, and `MageTeleportData.ForDestination` returns non-null.

## Overview

`MageTeleportTask` lifecycle splits into **three logical phases**:

**Phase 1 â€” Class + Spell + Cooldown gate (`Check` state).** Bot validates four predicates at task push: (a) `Player.Class == Class.Mage` ([`MageTeleportTask.cs:68-73`](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)); (b) `ObjectManager.IsSpellReady(_spellName)` ([:75-80](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)) which checks both spell-known AND cooldown; (c) `MageTeleportData.ForDestination(context, destinationName)` returns non-null (factory pattern at [:45-51](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)); (d) reagent-presence â€” **NOT currently checked by the task**, parity gap with `UseHearthstoneTask`'s item-presence check ([`UseHearthstoneTask.cs:54-55`](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)). Server-side `Spell::CheckCast` rejects with `SPELL_FAILED_REAGENTS` if Rune of Teleportation (17031) / Rune of Portals (17032) missing â€” task today relies on server-side rejection rather than client-side pre-check.

**Phase 2 â€” Cast dispatch + interrupt monitor (`StopAndCast + WaitForCast` states).** Bot dispatches `ObjectManager.StopMovement(...)` ([`MageTeleportTask.cs:88`](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)) + `ObjectManager.CastSpell(_spellName)` ([:89](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)) which routes through `SpellcastingManager.CastSpell` â†’ `CMSG_CAST_SPELL(spellId, target=self)`. Cast bar lasts **10 seconds** ([`MageTeleportTask.cs:30`](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs) `CastDurationMs = 10_000`). Interrupt detection identical to Hearthstone: `!IsCasting AND TickCount64 - _castStartMs > 2000` ([`MageTeleportTask.cs:99-105`](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)) â€” 2s buffer to avoid false-positive at cast-start.

**Phase 3 â€” Teleport detection (`DetectTeleport + Complete` states).** Server runs `Spell::EffectTeleportUnits` ([`SpellEffects.cpp:2251`](D:/MaNGOS/source/src/game/Spells/SpellEffects.cpp)) with `EffectImplicitTargetB[effIdx] == TARGET_LOCATION_DATABASE` (mage-tele branch) and looks up `SpellTargetPosition` via `sSpellMgr.GetSpellTargetPosition(spellId)` ([:2270](D:/MaNGOS/source/src/game/Spells/SpellEffects.cpp)). Same-map dest â†’ `NearTeleportTo`; cross-map dest â†’ `TeleportTo` (e.g. Stormwind=0 from Darnassus=1). Bot polls `Position.DistanceTo(_startPosition) > 100f OR MapId != _startMapId` ([`MageTeleportTask.cs:111-118`](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)) for 15s post-cast. Cross-continent teleports (Darnassus from Stormwind) trigger map-id change; same-continent (Stormwind from Ironforge: both mapId=0) trigger distance threshold (typical city-to-city same-map delta ~3000-4000y).

**Portal variant divergence**. Group-portal spells (10059, 11416-11420) cast the SAME 10s channel but DO NOT teleport the caster â€” they spawn a `GAMEOBJECT_TYPE_TRANSPORT` at the caster's feet which other party members click to invoke their own `TARGET_LOCATION_DATABASE` teleport. Caster-side Phase 3 detection FAILS for portals â€” DE must use a different completion predicate (e.g. observe `SMSG_SPELL_GO` ack OR spawn of GameObject by mage GUID). The shipped `MageTeleportTask` does NOT distinguish portal vs self-teleport at Phase 3 today; portal cast will hit the 15s timeout and log "Teleport detection timeout" ([`MageTeleportTask.cs:120-124`](../../../Exports/BotRunner/Tasks/Travel/MageTeleportTask.cs)). Open SCo.TRAVEL.MAGE.1 = portal-spawn-detect via `IObjectManager.GameObjects` filter by `GameObjectType.Transport` AND `Position.DistanceTo(caster) < 5f`.
