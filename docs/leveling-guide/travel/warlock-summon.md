# Warlock Ritual of Summoning — Class Gate → Soul Shard Gate → Party-Context Gate → 10s Ritual Cast → Ritual GO + 2 Helpers → CMSG_SUMMON_RESPONSE → Target Teleport

Fourth Travel sub-family after [`hearthstone.md`](hearthstone.md), [`flight-path.md`](flight-path.md), and [`mage-teleport.md`](mage-teleport.md). Where Hearth is **universal single-charge teleport-to-bind**, Flight is **paid server-arbitrated cross-continent spline**, and Mage-Tele is **class-gated free instant-teleport to capitals** — Warlock Ritual of Summoning is **class-gated multi-player teleport-to-warlock**: a 3-Warlocks-3-helpers-3-target ritual that drops a `GAMEOBJECT_TYPE_SUMMONING_RITUAL` portal at the caster's feet, requires **two same-party helpers** to click it, and **prompts the absent target** to accept via `CMSG_SUMMON_RESPONSE`. The summoned member teleports directly to the warlock's coord.

**Closest analog**: Mage-Portal (`mage-teleport.md` group-portal variant). Both are **multi-player travel** — the caster does NOT teleport themselves; instead, they spawn a clickable object that party members invoke. The Mage-Portal model is "many travel to one destination capital"; the Warlock-Summon model is "one absent target travels to the warlock's current coord". Both share the constraint that their Phase 3 completion is NOT caster-side teleport detection (mage-portal caster stays put; warlock-summon caster stays put) — the completion oracle is **target-side teleport detection** OR a portal-spawn + helper-click count predicate.

**Why this isn't "warlock-portal"**: there is no warlock-portal spell. The full vanilla 1.12.1 Warlock travel kit is just Ritual of Summoning (spell **698**, 1 reagent Soul Shard, 5min cooldown, 10s cast, requires 3 party members at the warlock's location). Meeting Stones — which work mechanically similarly via the same `SPELL_EFFECT_SUMMON_PLAYER = 85` server effect ([`SpellEffects.cpp:143,5730`](D:/MaNGOS/source/src/game/Spells/SpellEffects.cpp)) — are documented separately under the `dungeon/` family (iter-31+); they use `MeetingStoneSummonTask` ([`Exports/BotRunner/Tasks/Travel/MeetingStoneSummonTask.cs`](../../../Exports/BotRunner/Tasks/Travel/MeetingStoneSummonTask.cs)).

The shipped `WarlockSummonTask` ([`Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs), 110 LOC) is a **five-state machine** (`CheckPrereqs → CastRitual → WaitForHelpers → WaitForAccept → Complete`) — symmetric to `MageTeleportTask`'s shape (same `Class != X → pop`, same explicit `StopMovement` before cast, same `CastSpell("Ritual of Summoning")` dispatch by name) but with **two extra wait states** for the helper-click and target-accept handshakes. The TODO comments at [`WarlockSummonTask.cs:86,102`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs) note that the helper-click and accept-detection are **not yet wired** — Phase 2 advances on a fixed 10s timer and Phase 3 hits the 30s timeout for every cast today.

**Brief correction (1)**: The user prompt said "30s ritual timeout" — that is correct for the bot-side `RitualTimeoutMs = 30_000` at [`WarlockSummonTask.cs:26`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs), but the **server-side target auto-decline window is 2 minutes** (`MAX_PLAYER_SUMMON_DELAY = 2*MINUTE` at [`Player.h:742`](D:/MaNGOS/source/src/game/Objects/Player.h)). The bot's 30s budget is shorter than the server's 120s window — if the target is slow to accept, the bot will pop the task while the SMSG_SUMMON_REQUEST is still pending on the target's screen.

**Brief correction (2)**: The user prompt said "CMSG_SUMMON_RESPONSE = 0x2AC" — verified exactly: `0x2AC = 684 decimal`, see [`Opcodes_1_12_1.h:687`](D:/MaNGOS/source/src/game/Protocol/Opcodes_1_12_1.h) and the dispatch wiring at [`Opcodes.cpp:760`](D:/MaNGOS/source/src/game/Protocol/Opcodes.cpp).

**Brief correction (3)**: The user prompt said "3rd-player accept". The mechanic is that the **target** of the ritual (the absent party member being summoned) accepts. They are not necessarily the "3rd player" in the party; they are the explicit target argument passed to the cast (per `_targetPlayerGuid` at [`WarlockSummonTask.cs:22,31`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs)). The party can have 2-40 members; the 2 helpers are the same-party warlock-adjacent clickers, and the target is whoever the ritual is aimed at.

**Brief correction (4)**: There is **no `CMSG_RITUAL_*` opcode**. The ritual flow uses three reused opcodes: `CMSG_CAST_SPELL` (warlock casts 698), `CMSG_GAMEOBJ_USE` (helpers click the ritual GO — see [`GameObject.cpp:744`](D:/MaNGOS/source/src/game/Objects/GameObject.cpp) `AddUniqueUse`), and `CMSG_SUMMON_RESPONSE = 684 / 0x2AC` (target accepts). The "ritual" concept lives entirely in the server-side GameObject + spell-effect machinery, not in dedicated network opcodes.

## Overview

`WarlockSummonTask` lifecycle splits into **four logical phases** atop the five-state machine:

**Phase 1 — Class + reagent + party gate (`CheckPrereqs` state).** Bot validates two predicates at task push: (a) `Player.Class == Class.Warlock` ([`WarlockSummonTask.cs:42-47`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs)); (b) at least one `ObjectManager.Items[i].ItemId == 6265` (Soul Shard) is present in inventory ([`WarlockSummonTask.cs:50-65`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs)). **Missing today**: party-context gate (`IObjectManager.GroupMembers.Count >= 3` — 2 helpers + 1 target); target-presence gate (`GroupMembers.Any(m => m.Guid == _targetPlayerGuid)`); target-already-here gate (`GroupMembers.Single(m => m.Guid == _targetPlayerGuid).MapId == Player.MapId AND distance < 5y` — no point summoning someone already adjacent). Server-side guards live in spell-effect's `EffectSummonPlayer` ([`SpellEffects.cpp:5730-5749`](D:/MaNGOS/source/src/game/Spells/SpellEffects.cpp)) and the ritual GO's `AddUniqueUse` ([`GameObject.cpp:730-760`](D:/MaNGOS/source/src/game/Objects/GameObject.cpp)).

**Phase 2 — Ritual cast + GO spawn (`CastRitual + WaitForHelpers` states).** Bot dispatches `ObjectManager.StopMovement(...)` ([`WarlockSummonTask.cs:71`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs)) + `ObjectManager.CastSpell("Ritual of Summoning")` ([:72](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs)) which routes through `SpellcastingManager.CastSpell` → `CMSG_CAST_SPELL(spellId=698, target=_targetPlayerGuid)`. The 10s cast completes server-side and spawns a `GAMEOBJECT_TYPE_SUMMONING_RITUAL` GO at the warlock's feet (`reqParticipants=3` per `gameobject_template`). Two same-party helpers must walk to the ritual circle and `CMSG_GAMEOBJ_USE` it — the server increments `m_UniqueUsers` per click ([`GameObject.cpp:739`](D:/MaNGOS/source/src/game/Objects/GameObject.cpp)). When `m_UniqueUsers.size() >= reqParticipants` (3 = warlock + 2 helpers), the GO transitions to `GO_STATE_ACTIVE` and `FinishRitual` fires ([`GameObject.cpp:791-819`](D:/MaNGOS/source/src/game/Objects/GameObject.cpp)). **Bot today** advances out of `WaitForHelpers` on a fixed `10_000ms` timer ([`WarlockSummonTask.cs:88-91`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs)) — assumes helpers will click. Production should poll `IObjectManager.GameObjects.SingleOrDefault(g => g.Type == GameObjectType.SummoningRitual && g.OwnerGuid == player.Guid).UseCount >= 3`. Open SCo.TRAVEL.WARLOCK.1.

**Phase 3 — Target-accept handshake (`WaitForAccept` state).** Once `FinishRitual` fires, the server calls `Spell::EffectSummonPlayer` ([`SpellEffects.cpp:5730-5749`](D:/MaNGOS/source/src/game/Spells/SpellEffects.cpp)) which calls `pPlayerTarget->SendSummonRequest(m_caster->GetObjectGuid(), m_caster->GetMapId(), zoneId, x, y, z)` ([`Player.cpp:19998-20006`](D:/MaNGOS/source/src/game/Objects/Player.cpp)). The server sends `SMSG_SUMMON_REQUEST (683 / 0x02AB)` to the target with `(summonerGuid, zoneId, autoDeclineMs = MAX_PLAYER_SUMMON_DELAY * 1000 = 120_000ms)`. The target's client shows the standard accept-decline dialog. On `[ACCEPT]` click, target's client emits `CMSG_SUMMON_RESPONSE (684 / 0x2AC)` with `(summonerGuid)`. Server's `HandleSummonResponseOpcode` ([`MovementHandler.cpp:1009-1018`](D:/MaNGOS/source/src/game/Handlers/MovementHandler.cpp)) calls `_player->SummonIfPossible(true)` ([`Player.cpp:20009-20036`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) which validates `m_summon_expire > time(nullptr)` AND `!IsTaxiFlying()` then calls `TeleportTo(m_summon_mapid, m_summon_x, m_summon_y, m_summon_z, GetOrientation())`. **Bot today** hits the 30s timeout for every cast because the SMSG_MEETINGSTONE_COMPLETE-equivalent detection isn't wired ([`WarlockSummonTask.cs:94-103`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs)). Open SCo.TRAVEL.WARLOCK.2 = poll target's `MapId / Position` for jump-to-warlock-coord OR watch for `SMSG_MOVE_TELEPORT_ACK` ack from the target side.

**Phase 4 — Cooldown apply + completion (`Complete` state).** Spell 698 has a **5-minute cooldown** (300_000ms server-side `RecoveryTime`); applied via `AddCooldown` in `FinishRitual` at [`GameObject.cpp:798-800`](D:/MaNGOS/source/src/game/Objects/GameObject.cpp). Reagent (1 Soul Shard) consumed at cast-complete (NOT cast-start — same model as mage-tele) per server's `Spell::TakeReagents`. Bot pops the task at [`WarlockSummonTask.cs:105-107`](../../../Exports/BotRunner/Tasks/Travel/WarlockSummonTask.cs).

1. **Server-up assertion** — `ServerHealthcheck` returns 200; fail-fast if not (root R4).

9. **SMSG_SUMMON_REQUEST observed** — within 1s of `UseCount == 3`, assert target's snapshot shows pending summon request (via `RecentNetworkEvents` or equivalent multi-bot view).

13. **Cooldown-application assertion** — post-ritual, assert `Player.GetSpellCooldown(698) >= 299_000ms`; subsequent `WarlockSummonTask` push within 300s pops at `CheckPrereqs` with `not ready` reason.
14. **Combat-interrupt test** — push task; spawn aggressor mid-ritual-cast; assert task pops within 2s of aggressor-pull; assert `Inventory.Items.Count(i => i.ItemId == 6265)` unchanged (NO reagent consumed on interrupt).
15. **Target-on-taxi test** — set target on taxi flight via `.taxi` GM command; push ritual; assert target IS summoned (server-side `SummonIfPossible` cancels taxi); assert target's `Coinage` decreased by flight-gold (caught taxi cost).
