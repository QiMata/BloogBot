# Scheduled Transports (Boats + Zeppelins) — KeyFrame-Driven Period → TransportPhase State Machine → MOVEFLAG_ONTRANSPORT During Ride → Cross-Map SMSG_NEW_WORLD on Continent Transitions

Fifth and **final** Travel sub-family — closing the family 5/5 after [`hearthstone.md`](hearthstone.md), [`flight-path.md`](flight-path.md), [`mage-teleport.md`](mage-teleport.md), and [`warlock-summon.md`](warlock-summon.md). Where Hearth is **universal teleport-to-bind**, Flight is **paid cross-continent spline**, Mage-Tele is **class-gated free city-teleport**, and Warlock-Summon is **multi-player teleport-to-warlock** — Scheduled Transports are **free time-gated zero-class-requirement long-haul** travel. The user **walks to a dock or tower**, **waits up to ~7 minutes** for the boat/zeppelin to arrive, **walks onto the transport while it's docked**, **rides the GO spline** to the destination (typically 1-3 minutes), and **walks off at the destination dock/tower**.

**Why this is `transport-boat-zep` not "scheduled-transport"**: while the underlying mechanism (`GenericTransport` server-side, `TransportPhase` state machine bot-side) is **identical for boats and zeppelins**, the in-world identity matters for the DE router. Boats are Alliance-faction-friendly (Menethil↔Auberdine is the Alliance cross-continent route, Auberdine↔Teldrassil for Night Elf starting area) plus the Booty Bay↔Ratchet neutral route used by both factions. Zeppelins are Horde-only (Orgrimmar↔Grom'gol↔Undercity tri-route). Players approaching a wrong-faction dock can still ride (no GM gate), but the surrounding zone is hostile — DE-layer R7 below filters by faction-suitability.

**Distinct from flight-path**. Unlike `flight-path.md` ([`Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)), scheduled transports have **no NPC interaction**, **no gold cost**, **no taximask flip**, **no `CMSG_ACTIVATETAXI` opcode**, and **no `FLIGHT_MOTION_TYPE` motion generator**. The bot does not need to "discover" the dock/tower the way it needs to discover flight masters via `SendLearnNewTaxiNode`. Instead, the bot **walks to a known dock coord** (preconfigured in `TransportData.cs` static catalogue) and **waits for the transport's `GenericTransport` server-side spline to deposit it at the dock's wait position**.

**Embedded in `TravelTask.ExecuteTransportLeg`**. Unlike Hearth/Flight/Mage-Tele/Warlock-Summon which are standalone tasks, the scheduled-transport flow lives inside `TravelTask.ExecuteTransportLeg` ([`TravelTask.cs:502-605`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)) which delegates phase logic to `TransportWaitingLogic` ([`Exports/BotRunner/Movement/TransportWaitingLogic.cs`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs), 400+ LOC). `TravelTask` decomposes a cross-map travel-Objective into a `List<RouteLeg>` via `CrossMapRouter.PlanRoute(...)` ([`TravelTask.cs:228-256`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)) — each leg is `TransitionType.{Walk, FlightPath, Boat, Zeppelin, Elevator, DungeonPortal}`. Boat/Zep legs route through `ExecuteTransportLeg`; elevator legs share the same code path (only differing in the at-stop detection constants).

## Overview

`TravelTask.ExecuteTransportLeg` ([`TravelTask.cs:502-605`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)) drives a **six-phase state machine** via `TransportWaitingLogic`:

**Phase 1 — Approaching the wait position (`TransportPhase.Approaching`).** Bot walks to the boarding stop's `NavigationPosition` (typically `WaitPosition`, or `ApproachPosition` if configured per [`TransportData.cs:27-30`](../../../Exports/BotRunner/Movement/TransportData.cs)). Per [`TransportWaitingLogic.cs:213-230`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs): when `DistanceXY(currentPosition, navigationPosition) <= BoardingRadius` (typically 12-15y for boats, larger for zeppelins), transition to `WaitingForArrival`.

**Phase 2 — Waiting for the transport (`TransportPhase.WaitingForArrival`).** Bot stays at wait position. Per tick, scan `IReadOnlyList<DynamicObjectProto> nearbyObjects` (within `TransportNearbyObjectRadius = 80f` at [`TravelTask.cs:39`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)) for an object matching the configured transport's `GameObjectEntry` + `DisplayId`. Per [`TransportObjectIdentity.MatchesTransport(...)`](../../../Exports/BotRunner/Movement/TransportObjectIdentity.cs) — strict entry+displayId+name match. When found AND `TryFindTransportAtStop` returns true (boat within 20y / zeppelin within 45y of wait position per [`TransportWaitingLogic.cs:139-198`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs)), AND the transport is `IsTransportReadyToBoard` (dock-stability check: 5s with `<0.05y` position drift per [`TransportWaitingLogic.cs:45-46`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs)), transition to `Boarding`. **7-minute max wait** (`SCHEDULED_TRANSPORT_MAX_WAIT_TIME_SEC = 420f`) — if exceeded, transition to `Complete` (effectively abort).

**Phase 3 — Boarding (`TransportPhase.Boarding`).** Bot walks toward `CreateBoardingWaypoint(transportObject, _boardingStop)` which computes a position **on the transport's deck** (transport GO position + small offset matching where players naturally step). Per [`TransportWaitingLogic.cs:293-337`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs): bot detects boarding success when `isOnTransport = (player.TransportGuid != 0 || (player.MovementFlags & MOVEFLAG_ONTRANSPORT) != 0)` becomes true (per [`TravelTask.cs:1101-1103`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs) `IsOnTransport` helper). Optional second-step: if `_boardingStop.TransportBoardingOffset` is configured (local-space coord on the transport model), walk to that exact offset BEFORE transitioning to `Riding`. This is the offset that anchors the bot to a specific spot on the deck (e.g. Frezza's NPC at the OG zeppelin tower per Phase 5.3.6 logic at [`TravelTask.cs:27-36`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)). **60s boarding timeout** (`SCHEDULED_TRANSPORT_BOARDING_TIMEOUT_SEC`) — if the bot misses the boarding window (transport departs while bot is still walking), `MissedBoardingAttempt` flag flips, `TravelTask` resets to `WaitingForArrival` ([`TravelTask.cs:531-539`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)).

**Phase 4 — Riding the spline (`TransportPhase.Riding`).** Bot stays still. Per [`TransportWaitingLogic.cs:339-369`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs): returns `null` (no movement waypoint) while `isOnTransport == true`. Server-side `Transport::Update` ([`Transport.cpp:280-342`](D:/MaNGOS/source/src/game/Transports/Transport.cpp)) ticks the transport's `m_pathProgress` against the KeyFrame timeline, calling `UpdatePosition` each tick to move the GO along the spline. Player follows by being attached to the transport's passenger list. **Cross-map transitions** fire `Transport::TeleportTransport` ([`Transport.cpp:310-315`](D:/MaNGOS/source/src/game/Transports/Transport.cpp)) when `m_currentFrame->Node->mapid != GetMapId()` — server emits `SMSG_TRANSFER_PENDING (63) + SMSG_NEW_WORLD (62)` to all passengers; the bot's `MapId` flips mid-ride. The shipped `TravelTask.ExecuteTransportLeg` has explicit handling at [`TravelTask.cs:510-514`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs): `if (player.MapId != leg.MapId && !IsCrossMapTransportLeg(leg)) → CompleteCurrentLeg("transport_map_changed")` — for cross-map boats (Menethil↔Auberdine = map 0 → 1), the `IsCrossMapTransportLeg` predicate is true so the bot keeps riding through the map change.

**Phase 5 — Disembarking (`TransportPhase.Disembarking`).** When `IsTransportAtStop(nearbyObjects, _destinationStop)` returns true at [`TransportWaitingLogic.cs:358`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs), bot walks toward the destination stop's `WaitPosition` (computed via `CreateDisembarkWaypoint(transportObject, _destinationStop)`). When `isOnTransport` becomes false (bot steps off the transport collision mesh), transition to `Complete` ([`TransportWaitingLogic.cs:371-390`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs)). **10s disembark timeout** (`DISEMBARK_TIMEOUT_SEC`) — if bot doesn't step off (server bug or geometry trap), force complete.

**Phase 6 — Complete (`TransportPhase.Complete`).** `TravelTask.ExecuteTransportLeg` checks `HasReachedTransportExit(player, leg, ...)` at [`TravelTask.cs:547-559`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs): if `player.MapId == leg.ExitStop.MapId AND distance < max(ExitStop.BoardingRadius, WalkLegArrivalRadius=15y) AND verticalDelta < TransportExitVerticalArrivalTolerance=25y`, calls `CompleteCurrentLeg("transport_arrived ...")`. Else `FailOrReplan("transport_completed_without_arrival")` — triggers replan up to `MaxReplans = 3`.

## Failure modes

| Failure | Detection | Recovery |
|---|---|---|
| **No transport route between source and dest maps** | `CrossMapRouter.PlanRoute(...)` returns empty list ([`TravelTask.cs:239-245`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)); `_route == null OR Count == 0` | DE-layer R1 pre-push gate. Cascade to flight / hearth / mage-tele / land |
| **Walk-leg arrival at wrong deck (lower-spiral z=50 vs upper-deck z=53.6)** | `TransportWaitingLogic.HandleApproaching` returns true (within `BoardingRadius`) but at lower Z; Phase 2 wait position is wrong; transport arrives but bot is below its detection radius | Phase 5.3.6 tightened walk-leg vertical tolerance 6y → 1.5y in native-off-mesh mode at [`TravelTask.cs:27-36`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs). Pre-Phase-5.3.6 behavior occasionally caused this failure (cross-cite `OgZeppelinCliffFallParityTests` regression family) |
| **Transport never arrives at wait position** (7-min timeout) | `_phaseElapsed > GetMaxWaitTimeSec()` ([`TransportWaitingLogic.cs:262-267`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs)); transition to `Complete` (effectively abort) | `TravelTask` sees `TransportPhase.Complete` without `HasReachedTransportExit` → `FailOrReplan("transport_completed_without_arrival")`. Replan up to 3×; final cascade to flight/land |
| **Missed boarding window** (transport departs while bot is still walking onto deck) | `MissedBoardingAttempt` flag set at [`TransportWaitingLogic.cs:302-304`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs); `TravelTask.ExecuteTransportLeg` emits diagnostic + transitions back to `WaitingForArrival` at [`TravelTask.cs:531-539`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs) | Wait for next cycle (4-8min depending on transport); replan if cumulative `transport_timeout = 480s` ([`TravelTask.cs:45`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)) exceeded |
| **Transport-GO display-ID mismatch (anti-bot detection edge case)** | `TransportObjectIdentity.MatchesTransport(obj, _transport)` returns false for all nearby objects — server may have spawned a different display variant; `TryFindTransportAtStop` never finds the match | Update `TransportData.cs` `DisplayId` field per recorded fixture; verify match with `OgZepValidate` recorded fixture |
| **Combat mid-Approaching / mid-WaitingForArrival** | Mob aggros at dock; bot's combat task pre-empts movement but TravelTask is still on stack | DE should pause TravelTask + push combat task. Today not wired — open SCo.TRAVEL.TRANSPORT.1 |
| **Combat mid-Riding** (rare — pre-spawned mobs on the transport itself) | Bot is attached to transport spline; combat fires but bot can't move off; `MovementFlags & MOVEFLAG_ONTRANSPORT` still set | Server-side combat does NOT detach from transport (verified `Spell::EffectTeleportUnits` exempts `IsTaxiFlying` but transports use different mechanism). Bot fights from the deck; if killed, releases corpse → respawns at graveyard, leg fails → replan |
| **Cross-map transition incomplete** (`SMSG_NEW_WORLD` arrives but client/server out of sync) | `player.MapId` change observed but `IsOnTransport == false` briefly; per [`TransportWaitingLogic.cs:345-355`](../../../Exports/BotRunner/Movement/TransportWaitingLogic.cs) `HandleRiding` keeps ride alive across brief disconnects | Cross-cite Phase 5.3.4 "Detour-walkable but physically-blocked" geometry — `WaitForMapTransition` retries up to 5s before triggering `Complete` |
| **Stuck at destination dock** (`HasReachedTransportExit` returns false after disembark) | `player.MapId == leg.ExitStop.MapId BUT distance > BoardingRadius` ([`TravelTask.cs:1216-1236`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs)); `FailOrReplan("transport_completed_without_arrival")` | Replan up to 3×; final cascade to land travel from current MapId to true destination |
| **Bot can't pathfind to dock** (walk-leg fails before transport-leg starts) | Walk-leg stall detected at [`TravelTask.cs:759-773`](../../../Exports/BotRunner/Tasks/Travel/TravelTask.cs) `TRAVEL_WALK_STALL`; `_walkStallRecoveryCount` increments; replan via `NavPath.RecalculateAfterMovementStall` | Cross-cite pathfinding skill — known regression family `LongPathingRouteTests::OgZeppelinCliffFallParityTests` etc. (PFS loop 18-24 memory entries) |
| **Disconnect mid-ride** | Client drops during transport-spline; reconnect handler re-attaches player to transport position via `m_passengers` list at server side | Bot's `TransportPhase.Riding` returns null waypoint; on reconnect, `IsOnTransport` re-evaluates and ride continues until destination |

## Live-test acceptance

A live integration test exercising this Activity must satisfy R8 (clean cross-test state) by:

1. **Server-up assertion** — `ServerHealthcheck` returns 200; fail-fast if not (root R4).
2. **Pre-test snapshot predicate** — `Player.Position ~= boardStop.WaitPosition` (within 100y); `Player.MapId == boardStop.MapId`; `Player.TransportGuid == 0`; `Player.IsInCombat == false`. Setup: `.tele` bot near dock; `.unaura`/`.cast cancel` to clear interrupt-state.
3. **No-NPC-interaction assertion** — assert `RecentNetworkEvents` contains no `CMSG_GOSSIP_HELLO` / `CMSG_ACTIVATETAXI` for the duration of the test (validates "no FM interaction" per R7 of this doc).
4. **No-cost assertion** — assert `Player.Coinage` unchanged across the full leg (validates "no cost" — distinct from flight-path R6 gold-deduction).
5. **No-taximask-flip assertion** — assert `IFlightMasterNetworkClientComponent.AvailableTaxiNodes` unchanged.
6. **Push `TravelTask(targetMapId, targetPosition)`** via fixture with a cross-continent destination (e.g. OG → UC via zeppelin, or Menethil → Auberdine via boat). `CrossMapRouter` should plan a `[Walk → Transport → Walk]` route.
7. **Walk-leg arrival assertion (Phase 1)** — within `30s + pathfinding-budget`, `player.Position.DistanceTo(boardStop.NavigationPosition) <= BoardingRadius`.
8. **WaitingForArrival timeout assertion (Phase 2)** — `_phaseElapsed <= SCHEDULED_TRANSPORT_MAX_WAIT_TIME_SEC = 420s` for boats/zeps. Test should allow up to one full period (~8 min).
9. **Boarding success assertion (Phase 3)** — within `60s` of transport arrival at dock, `IsOnTransport(player) == true` (`TransportGuid != 0` OR `MovementFlags & MOVEFLAG_ONTRANSPORT != 0`).
10. **Cross-map transition observation (Phase 4 — for cross-continent transports)** — assert `Player.MapId` flips mid-ride; assert `SMSG_TRANSFER_PENDING + SMSG_NEW_WORLD` observed in `RecentNetworkEvents`.
11. **Disembark assertion (Phase 5)** — within `10s` of destination-stop detection, `IsOnTransport(player) == false`.
12. **Complete assertion (Phase 6)** — `HasReachedTransportExit(player, leg, ...)` returns true; `player.Position.DistanceTo(exitStop.WaitPosition) <= max(BoardingRadius, 15y)`; `verticalDelta <= 25y`.
13. **No-replan assertion** — `_replanCount == 0` for a clean ride (replans indicate stall or boarding-miss).
14. **Disconnect guard** — mid-ride disconnect must fail the test (R8).
15. **Screenshot capture** — (a) pre-walk (bot at far-zone marker); (b) at-dock (bot waiting at WaitPosition); (c) transport-arrived (boat/zep visible adjacent to bot); (d) on-deck (bot mid-ride with MOVEFLAG_ONTRANSPORT); (e) post-disembark (bot at destination dock). Overwrite `Tests/artifacts/travel/transport/latest-{walk,dock,arrived,deck,exit}.png`.
16. **Final state dump** — write `Snapshot.Player.Position` + `MapId` + `TransportGuid` + `MovementFlags` + last 20 `TRAVEL_TRANSPORT` diag messages + ride-duration to `Tests/artifacts/travel/transport/snapshot-final.json`.
