# Flight-Path Task — Per-Task DE-Rule Layer over Per-NPC Interaction + Network Topology

Second Travel sub-family after [`hearthstone.md`](hearthstone.md). Where hearthstone is **universal-class single-charge cooldown-amortized teleport-to-bind**, the flight-path task is **paid server-arbitrated cross-continent spline travel** — the only travel option for cross-continent trips that doesn't require a class-specific spell (mage-tele) or a multi-player ritual (warlock-summon).

**Documentation layering**. This monorepo already has two flight-path documents at different abstraction layers:

1. **Network topology** — [`../systems/flight-paths.md`](../systems/flight-paths.md) (212 LOC) — Alliance/Horde FP hubs, neutral cross-faction FPs, cost scaling, boats+zeppelins, class-specific alternatives, cross-continent route planning. Answers: "Which flight master? Which destination? What's it cost? What's the alternative?"
2. **Per-NPC interaction protocol** — [`../npc/flight-master.md`](../npc/flight-master.md) (194 LOC, 15 R-rules, iter-24) — `UNIT_NPC_FLAG_FLIGHTMASTER = 0x08`, 4-CMSG opcode flow (`CMSG_TAXINODE_STATUS_QUERY` / `CMSG_TAXIQUERYAVAILABLENODES` / `CMSG_ACTIVATETAXI` / `CMSG_ACTIVATETAXIEXPRESS`), 11-error-gate cascade in `Player::ActivateTaxiPathTo`, taximask one-bit-per-node, server-arbitrated multi-stop via `taxi_path_transitions`. Answers: "How does the bot talk to the flight master?"

**This doc (per-task DE-rule layer)** sits between the two — it answers: "When does the DecisionEngine push `TakeFlightPathTask`, with what node IDs, gated by what predicates, and how does it recover from failure?" The shipped task at [`Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs) (165 LOC) is a **seven-state machine** (`FindFlightMaster → NavigateToFM → InteractWithFM → WaitForTaxiWindow → ActivateFlight → InFlight → Complete`) — but the source itself does not encode the DE-layer push predicates (gold check, route-vs-land decision, multi-stop chaining, in-flight monitor). Those DE predicates are the R-rule contents of this doc.

**Migration choice**. `systems/flight-paths.md` is NOT being promoted into `travel/flight-path.md`. Both stay — they serve orthogonal layers (network topology vs task execution). This doc cross-cites both — `systems/flight-paths.md` for "which node ID", `npc/flight-master.md` for "how to talk to the NPC". Per [`hearthstone.md`](hearthstone.md) R11 cost-cascade, this doc + sibling docs feed the unified DE travel-cost router (SCo.TRAVEL.FLIGHT.5 = sibling of SCo.TRAVEL.HEARTHSTONE.5 = single shared router).

## Overview

`TakeFlightPathTask` lifecycle splits into **three logical phases** atop the seven-state machine:

**Phase 1 — Find + Approach (states `FindFlightMaster + NavigateToFM`).** Bot scans `IObjectManager.Units` for the nearest unit with `UNIT_NPC_FLAG_FLIGHTMASTER (0x08)` set in its `NpcFlags` UpdateField ([`TakeFlightPathTask.cs:54-57`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)). The shipped task picks **nearest** unconditionally — there's no filter on "is this flight master at our planned source node" (a known limitation; if two flight masters are visible the bot picks the closer one which may be the wrong one for the planned route). DE-layer rule R1 below explicitly gates the push on "source node matches the nearest reachable FM". Navigation uses `ObjectManager.MoveToward(Position)` ([`TakeFlightPathTask.cs:78`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)) which is the same primitive `GoToTask` uses — routes through `PathfindingClient` and `NavigationPathFactory`.

**Phase 2 — Dialog + Activate (states `InteractWithFM + WaitForTaxiWindow + ActivateFlight`).** Bot dispatches `_flightMaster.Interact()` ([`TakeFlightPathTask.cs:84`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)) which emits `CMSG_GOSSIP_HELLO`. Per [`flight-master.md`](../npc/flight-master.md) Phase 2-3, the server response routes via `GOSSIP_OPTION_TAXIVENDOR = 4` → `SendTaxiMenu` → `SMSG_SHOWTAXINODES`. The shipped `WaitForTaxiWindow` state currently waits **1500ms before unconditionally advancing to ActivateFlight** ([`TakeFlightPathTask.cs:99-102`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)) — this is a known temporary shim per the TODO comment at line 112 ("TODO: use agent factory"). Production path should use `IFlightMasterNetworkClientComponent.IsTaxiMapOpen` ([`IFlightMasterNetworkClientComponent.cs:19`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IFlightMasterNetworkClientComponent.cs)) as the predicate; FG path goes via `IObjectManager.TaxiFrame.IsOpen`. On `ActivateFlight`, the task currently uses FG-only `TaxiFrame.SelectNodeByNumber((int)_destinationNodeId)` ([`TakeFlightPathTask.cs:117`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)) which routes through `NetworkTaxiFrame` on BG ([`NetworkTaxiFrame.cs:75-103`](../../../Exports/WoWSharpClient/Frames/NetworkTaxiFrame.cs)) and emits `CMSG_ACTIVATETAXI(fmGuid, sourceNodeId, destNodeId)`. The 11-error-gate cascade documented at [`flight-master.md`](../npc/flight-master.md) Phase 5 + R10 fires entirely on the server side from this single CMSG.

**Phase 3 — In-Flight Monitor (states `InFlight + Complete`).** Once `SMSG_ACTIVATETAXIREPLY(ERR_TAXIOK = 0)` lands, server pushes the `FLIGHT_MOTION_TYPE` motion-generator with the spline node list ([`TaxiHandler.cpp:117-119`](D:/MaNGOS/source/src/game/Handlers/TaxiHandler.cpp)) and sets `MOVEFLAG_ONTRANSPORT (0x02000000)` per [`MoveSplineInit.cpp:117`](D:/MaNGOS/source/src/game/Movement/spline/MoveSplineInit.cpp). The bot loses **all movement authority** during the flight — `ObjectManager.MoveToward / StopMovement` calls are silently dropped server-side. Landing detection per [`TakeFlightPathTask.cs:130-156`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs): bot polls `Position.DistanceTo(_lastFlightPos) < 1.0f` and counts 6 consecutive 500ms ticks (3 seconds) of stationary position AFTER ≥5000ms minimum in-flight. The min-5s guard avoids false-landing at the takeoff stationary frame.
