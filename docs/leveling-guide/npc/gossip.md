# Gossip — Overlay Protocol: Hello → Menu Fold → Option Classification → Cascade Dispatch → Sub-Menu / POI / Close

Fourth NPC sub-family after [`quest-giver.md`](quest-giver.md), [`vendor.md`](vendor.md), and [`trainer.md`](trainer.md). Where each of those three docs models a **specific** NPC-type's protocol — `CMSG_QUESTGIVER_HELLO`, `CMSG_LIST_INVENTORY`, `CMSG_TRAINER_LIST` respectively — **gossip is the overlay layer that all three share**. Per the iter-22 trainer.md hand-off: `CMSG_GOSSIP_HELLO` + `CMSG_GOSSIP_SELECT_OPTION` drive [`SelectGossipTask.cs`](../../../Exports/BotRunner/Tasks/SelectGossipTask.cs) and **gate sub-menu selection in 60%+ of all NPC interactions in 1.12.1**. Any NPC carrying `UNIT_NPC_FLAG_GOSSIP = 0x01` ([`UnitDefines.h:660`](D:/MaNGOS/source/src/game/Objects/UnitDefines.h)) presents the gossip menu first — the type-specific CMSG only fires after the player selects the corresponding `GOSSIP_OPTION_*` row.

This is the cross-cutting infrastructure doc. It does NOT replace the three sub-family docs; instead, it names the shared mechanics so future NPC docs (flight-master, innkeeper, banker, stable-master, auctioneer, tabard-designer, battlemaster) can reference this overlay layer in one place. **A future doc-side refactor opportunity (NOT this iteration)** is to back-port a "gossip ref" subsection in each of `quest-giver.md` / `vendor.md` / `trainer.md` pointing here for the multi-option menu fold mechanics — those docs currently inline the gossip overlay description; centralising it here eliminates the drift risk if mangos opcode renumbers in TBC port.

The server-side gossip state machine spans **two CMSG opcodes and two SMSG responses** plus an associated `CMSG_NPC_TEXT_QUERY` text-detail fetch:

## Overview

Gossip overlay interaction in 1.12.1 splits into **five phases**, each with distinct server-side dispatch and bot-side classification responsibilities.

**Phase 1 — Visibility + multi-flag detection.** A gossip-capable NPC is any `Creature` with `UNIT_NPC_FLAG_GOSSIP = 0x00000001` in its `UNIT_NPC_FLAGS` UpdateField. Predicate `unit.NpcFlags & 0x01 != 0` selects them. **Critical multi-flag fact**: `UNIT_NPC_FLAG_GOSSIP` is almost always paired with one or more type-specific bits — a typical major-city trainer carries `0x11 (GOSSIP | TRAINER)`, an innkeeper carries `0x81 (GOSSIP | INNKEEPER)`, a vendor-with-repair-and-gossip carries `0x4005 (GOSSIP | VENDOR | REPAIR)`. The 14 npcflag bits at [`UnitDefines.h:659-674`](D:/MaNGOS/source/src/game/Objects/UnitDefines.h) are independent; multi-function NPCs are the norm, not the exception. `PrepareGossipMenu` at [`Player.cpp:12421-12591`](D:/MaNGOS/source/src/game/Objects/Player.cpp) filters per-option by ANDing each menu row's `npc_option_npcflag` against the creature's actual `npcflags` ([`Player.cpp:12466-12468`](D:/MaNGOS/source/src/game/Objects/Player.cpp) `if (!(itr->second.npc_option_npcflag & npcflags)) continue;`) — so a single NPC's menu surface is dynamically pruned to its actual capability set.

**Phase 2 — Range gate + Hello dispatch.** Once the bot is within `INTERACTION_DISTANCE = 5.0f` yards ([`ObjectDefines.h:24`](D:/MaNGOS/source/src/game/Objects/ObjectDefines.h)) — same gate as quest-giver/vendor/trainer — dispatch `InteractWithUnitTask(npcGuid)` analog which issues `CMSG_GOSSIP_HELLO(npcGuid)`. Server [`NPCHandler.cpp:357-383`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp) runs the standard interaction prologue: `GetNPCIfCanInteractWith(guid, UNIT_NPC_FLAG_NONE)` (note: gossip hello passes `NONE` mask — any reachable NPC qualifies for the hello), interrupts channels (`AURA_INTERRUPT_INTERACTING_CANCELS`), removes feign-death, pauses NPC movement unless `CREATURE_FLAG_EXTRA_NO_MOVEMENT_PAUSE`. For spirit guides ([`NPCHandler.cpp:376-377`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) the server short-circuits with a graveyard-list opcode instead of a normal menu. Script hook `sScriptMgr.OnGossipHello(_player, pCreature)` ([`NPCHandler.cpp:378`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) can take over — scripted NPCs (~5-8% of all gossip-capable creatures in 1.12.1: Onyxia trigger NPCs, BWL attunement, certain bosses) bypass the standard `PrepareGossipMenu / SendPreparedGossip` pair and push a custom UI flow.

**Phase 3 — Menu receive + option classification.** Server replies with `SMSG_GOSSIP_MESSAGE(textId, menuId, itemCount, GossipMenuItem[]+, questCount, QuestMenuItem[]+)`. Per-item payload is `(uint32 gossipListId, uint8 iconId, uint8 isCoded, uint32 boxMoney, string optionText[, string boxText if isCoded])` — `iconId` ∈ `GOSSIP_ICON_*` enum at [`GossipDef.h:58-81`](D:/MaNGOS/source/src/game/GossipDef.h) (CHAT=0, VENDOR=1, TAXI=2, TRAINER=3, INTERACT_1=4, INTERACT_2=5, MONEY_BAG=6, TALK=7, TABARD=8, BATTLE=9, DOT=10, plus 10 more variants added in 1.7/1.10). Quest items append after (`uint32 questId, uint32 questIcon, int32 questLevel, string questTitle`). The bot's `IGossipNetworkClientComponent.HandleGossipMenuReceived(menuData)` ([`IGossipNetworkClientComponent.cs:171`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IGossipNetworkClientComponent.cs)) ingests and emits via the `GossipMenuOpened` observable. The DE then classifies each `GossipOption` row by `Type` ([`IGossipFrame.cs:27`](../../../Exports/GameData.Core/Frames/IGossipFrame.cs)) — one of `GossipTypes` corresponding to the 17 server-side `Gossip_Option` values at [`GossipDef.h:35-56`](D:/MaNGOS/source/src/game/GossipDef.h): `GOSSIP_OPTION_NONE=0`, `GOSSIP=1`, `QUESTGIVER=2`, `VENDOR=3`, `TAXIVENDOR=4`, `TRAINER=5`, `SPIRITHEALER=6`, `SPIRITGUIDE=7`, `INNKEEPER=8`, `BANKER=9`, `PETITIONER=10`, `TABARDDESIGNER=11`, `BATTLEFIELD=12`, `AUCTIONEER=13`, `STABLEPET=14`, `ARMORER=15`, `UNLEARNTALENTS=16`, `UNLEARNPETSKILLS=17`.

**Phase 4 — Option selection + cascade dispatch.** The bot selects a `gossipListId` and issues `CMSG_GOSSIP_SELECT_OPTION(npcGuid, gossipListId, [code if `isCoded`])`. Server [`NPCHandler.cpp:385-433`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp) runs the prologue (channel-interrupt, aura-removal, movement-pause), then dispatches through `Player::OnGossipSelect` at [`Player.cpp:12645-12750`](D:/MaNGOS/source/src/game/Objects/Player.cpp). The cascade is a giant `switch (gossipOptionId)` that fans out to the type-specific subsystem:

After the cascade, the gossip menu either closes (most type-specific cascades end with `CloseGossip()`) OR navigates to a sub-menu (`GOSSIP_OPTION_GOSSIP` with positive `m_gAction_menu` re-runs `PrepareGossipMenu / SendPreparedGossip`). Server emits `SMSG_GOSSIP_POI` ([`Opcodes_1_12_1.h:548-551`](D:/MaNGOS/source/src/game/Protocol/Opcodes_1_12_1.h)) automatically whenever the selected gossip row has `m_gAction_poi != 0` ([`Player.cpp:12671-12672`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) — minimap question-mark markers (Booty Bay treasure-hunter "the chest is here" style) come through this path.

**Phase 5 — POI / NPC text query / close.** Two side-channels run in parallel with the menu fold:
- **NPC text fetch**: each `SMSG_GOSSIP_MESSAGE` carries a top-level `textId` referencing the `npc_text` table. If the client doesn't have it cached, fetch via `CMSG_NPC_TEXT_QUERY(textId, npcGuid)` → `SMSG_NPC_TEXT_UPDATE` with up to 8 (text, broadcastText, emote, lang) tuples. The bot's `IGossipNetworkClientComponent.QueryNpcTextAsync(textId, npcGuid)` ([`IGossipNetworkClientComponent.cs:96`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IGossipNetworkClientComponent.cs)) wraps this. **DE rarely needs the text body** — option `iconId` + `optionText` field are sufficient for classification; NPC text is descriptive flavour for the human player.
- **POI delivery**: `SMSG_GOSSIP_POI` arrives unsolicited after a `GOSSIP_OPTION_GOSSIP` row with `action_poi_id != 0` is selected. Carries `(flags, x, y, icon, data, name)` for minimap overlay. Bot may stash for path-router input but does not need to acknowledge.

The session closes through one of three paths: (a) explicit `CMSG_GOSSIP_HELLO` to a different NPC (server auto-closes the prior one); (b) bot walks out of `INTERACTION_DISTANCE` (server emits `SMSG_GOSSIP_COMPLETE` and the menu auto-closes); (c) cascade through a type-specific terminal option (`GOSSIP_OPTION_INNKEEPER` calls `CloseGossip()` then `SetBindPoint`, etc.). The bot's `IGossipFrame.Close()` ([`NetworkGossipFrame.cs:32-37`](../../../Exports/WoWSharpClient/Frames/NetworkGossipFrame.cs)) explicitly dispatches `CloseGossipAsync()` on the BG agent if the window is open — useful for the DE's "back out and try another NPC" cleanup pattern.

## Live-test acceptance

A live integration test exercising the gossip overlay must satisfy R8 (clean cross-test state) by:

1. **Server-up assertion** — `ServerHealthcheck` returns 200 before the test starts; if not, FAIL FAST. Do not retry-loop on a dead server.
2. **Pre-test snapshot predicate** — `Player.Health > 0 AND Player.IsInCombat == false AND Player.IsChanneling == false AND GossipFrame.IsOpen == false` before triggering Hello.
3. **NPC spawn assertion** — `IObjectManager.Units.Any(u => u.Entry == TargetNpcEntry && u.NpcFlags & 0x01 != 0 && u.Health > 0)` within 5s of test start. Do NOT spawn the NPC; natural world state only per [Test Skip Policy](../../../CLAUDE.md).
4. **Hello dispatch predicate** — push `InteractWithUnitTask(npcGuid)` via fixture; poll `Player.Position.DistanceTo(npc.Position) <= 5.0f` within 30s; then poll `GossipFrame.IsOpen == true` AND `GossipFrame.NPCGuid == npcGuid` within 3s of arrival. Multi-flag NPCs should produce `Options.Count >= 1` (typical major-city innkeeper: 2 options — INNKEEPER + VENDOR).
5. **Option classification predicate** — assert `GossipFrame.Options.Any(o => o.Type == ExpectedType)` for the test's target option type. For multi-option NPCs, verify the classification ladder per R3 picks the correct row matching the Activity intent.
6. **Select dispatch predicate** — issue `SelectGossipTask(optionIndex)` via the Activity-level mechanism (`AssignedActivity` string per fixture today; `IActivity.Start(...)` post-S2.0). Do NOT construct `new ObjectiveMessage { ObjectiveType = ObjectiveType.SELECT_GOSSIP }` in the test body — per [Test Isolation Rules](../../../CLAUDE.md).
7. **Cascade observation** — depending on selected type, poll for the cascade arrival within 3s: `MerchantFrame.IsOpen` for VENDOR, `TrainerFrame.IsOpen` for TRAINER, `BankFrame.IsOpen` for BANKER, `Player.HearthstoneArea` change for INNKEEPER, `Player.QuestLog` for QUESTGIVER, etc. (R5/R10).
8. **Sub-menu test** — for nested gossip (R12), select a `GOSSIP_OPTION_GOSSIP` row leading to a sub-menu; assert a SECOND `GossipMenuOpened` event fires with a different `menuId` AND `GossipFrame.Options` re-populates with the sub-menu items.
