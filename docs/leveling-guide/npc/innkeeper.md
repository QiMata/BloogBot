# Innkeeper — Visibility → Range → Hello → Binder Option → CMSG_BINDER_ACTIVATE → SMSG_BINDPOINTUPDATE → Hearthstone → 10s Channel → Teleport → 60min Cooldown

Sixth and **closing** NPC sub-family after [`quest-giver.md`](quest-giver.md), [`vendor.md`](vendor.md), [`trainer.md`](trainer.md), [`gossip.md`](gossip.md), and [`flight-master.md`](flight-master.md). Where the prior five model **dialog**, **economy**, **spell-acquisition**, the **dialog overlay**, and **server-driven spline teleport**, this one closes the family with the **player-driven recall teleport** — the **bind point + Hearthstone** mechanic that anchors every grind loop, every recovery-after-wipe in a distant zone, and every cross-continent recall. This is the only NPC sub-family in 1.12.1 whose interaction outcome **persists across sessions** in `character_homebind` and whose **teleport invocation is a player-owned 60-minute-cooldown consumable** (the Hearthstone, itemId `6948`), not a server-arbitrated motion-generator like flight.

The shipped state is **structurally simpler than flight-master but more cross-cutting than vendor/trainer**: a two-task family (`SetBindPointTask` + `UseHearthstoneTask`) wires into a static city/hub catalog (`InnkeeperData`) plus a snapshot-side hook (`CharacterInitNetworkClientComponent.HandleBindPointUpdate` parses `SMSG_BINDPOINTUPDATE` and exposes via `BindPointUpdates` observable + `CurrentBindPoint` accessor). `SetBindPointTask` ([`Exports/BotRunner/Tasks/Travel/SetBindPointTask.cs`](../../../Exports/BotRunner/Tasks/Travel/SetBindPointTask.cs)) drives a five-state machine (`FindInnkeeper → NavigateToInnkeeper → Interact → WaitForBind → Complete`) using `unit.NpcFlags & 0x80` for innkeeper discovery and `gossipFrame.SelectGossipOption(5)` to pick the binder row. `UseHearthstoneTask` ([`Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs`](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)) drives a five-state machine (`FindItem → StopAndCast → WaitForCast → DetectTeleport → Complete`) with a `>100y` distance OR `mapId` change as teleport detection. `InnkeeperData` ([`Exports/BotRunner/Travel/InnkeeperData.cs`](../../../Exports/BotRunner/Travel/InnkeeperData.cs)) catalogs 26 known innkeepers across Horde (8) + Alliance (8) + Neutral (6) plus capital cities (6), with `FindNearest(mapId, position, faction)` route helper.

**A pre-existing shipped-code bug surfaced by this doc (see BRIEF CORRECTION D below)**: `SetBindPointTask.cs:104` passes the literal `5` to `gossipFrame.SelectGossipOption(5)`. `SelectGossipOption(int)` on [`NetworkGossipFrame.cs:41-47`](../../../Exports/WoWSharpClient/Frames/NetworkGossipFrame.cs) treats the argument as the **wire `gossipListId` index**, not as a `GossipTypes` enum. The correct pattern is `gossipFrame.SelectFirstGossipOfType(DialogType.Binder)` ([`NetworkGossipFrame.cs:49-60`](../../../Exports/WoWSharpClient/Frames/NetworkGossipFrame.cs)) which scans `Options[]` for the binder-typed row and emits the correct `gossipListId`. This is open gap **SCo.INNKEEPER.1** documented at the tail of this file.

## Overview

Innkeeper interaction in 1.12.1 splits into **six phases**, three for bind-set and three for recall.

**Phase 1 — Visibility detection.** An innkeeper is any `Creature` with `UNIT_NPC_FLAG_INNKEEPER = 0x00000080` in its `UNIT_NPC_FLAGS` UpdateField ([`UnitDefines.h:667`](D:/MaNGOS/source/src/game/Objects/UnitDefines.h) — verified per iter-24 flight-master hand-off). The bot's `IObjectManager.Units` enumeration sees the bit; predicate `((uint)u.NpcFlags & 0x80) != 0` selects innkeepers ([`SetBindPointTask.cs:42-43`](../../../Exports/BotRunner/Tasks/Travel/SetBindPointTask.cs)). Almost all innkeepers also carry `UNIT_NPC_FLAG_GOSSIP = 0x01` (Hello cascade) and most also carry `UNIT_NPC_FLAG_VENDOR = 0x04` (some innkeepers double as food/drink vendors — e.g. Innkeeper Wiley in Ratchet sells refreshments). Multi-flag combinations: `0x81 GOSSIP|INNKEEPER` (most rural innkeepers), `0x85 GOSSIP|VENDOR|INNKEEPER` (most city innkeepers with food bar). The bot's `InnkeeperData` static catalog ([`InnkeeperData.cs:23-60`](../../../Exports/BotRunner/Travel/InnkeeperData.cs)) ships 26 entries with map + position + faction — `FindNearest(mapId, position, faction)` picks the closest reachable innkeeper as a fallback for sparse-spawn zones.

**Phase 2 — Range gate + Hello.** Within `INTERACTION_DISTANCE = 5.0f` yards ([`ObjectDefines.h:24`](D:/MaNGOS/source/src/game/Objects/ObjectDefines.h)), dispatch `_innkeeper.Interact()` ([`SetBindPointTask.cs:82`](../../../Exports/BotRunner/Tasks/Travel/SetBindPointTask.cs)) which sends `CMSG_GOSSIP_HELLO`. **Gossip overlay**: see [`gossip.md` §Overview](gossip.md#overview) Phases 2-4 for canonical Hello/menu/select mechanics. Per-NPC cascade entry: `GOSSIP_OPTION_INNKEEPER = 8` ([`gossip.md` §Overview Phase 4 table](gossip.md#overview)) → server runs `CloseGossip() + SetBindPoint(guid)` which is the bind-set terminal action (R5 below). **Alternative direct path**: dispatch `CMSG_BINDER_ACTIVATE(npcGuid)` (`437 / 0x01B5`) straight to `HandleBinderActivateOpcode` bypassing the gossip overlay (R4 below, plus SCo.INNKEEPER.3 for the not-yet-shipped helper). Innkeeper-specific gate: `HandleBinderActivateOpcode` ([`NPCHandler.cpp:499-517`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) gates on `GetNPCIfCanInteractWith(guid, UNIT_NPC_FLAG_INNKEEPER)`; both Hello AND BinderActivate auto-interrupt channels + remove feign-death (same pattern as the standard gossip channel/feign hook at [`gossip.md` §Pre-conditions](gossip.md#pre-conditions)).

**Phase 3 — Binder option selection.** Innkeepers present a gossip menu with at least one binder row (mangos wire `GOSSIP_OPTION_INNKEEPER = 8` per [`GossipDef.h:45`](D:/MaNGOS/source/src/game/GossipDef.h)) and often additional rows: a `GOSSIP_OPTION_GOSSIP` info row ("What can I do at an inn?"), a `GOSSIP_OPTION_VENDOR` row for food/drink, and (in some city inns) a `GOSSIP_OPTION_QUESTGIVER` row. The bot's `IGossipFrame.Options` populates after the gossip menu opens; for each row, server `npc_option.icon` ([`GossipDef.h:58-81`](D:/MaNGOS/source/src/game/GossipDef.h)) is parsed into WWoW's `GossipTypes` enum at [`GossipNetworkClientComponent.cs:441`](../../../Exports/WoWSharpClient/Networking/ClientComponents/GossipNetworkClientComponent.cs). The binder row carries `icon = 4` (`GOSSIP_ICON_INTERACT_1`) which WWoW maps to `GossipTypes.Healer`. **WWoW's enum is reordered from mangos** — see BRIEF CORRECTION B. Bot should use `gossipFrame.SelectFirstGossipOfType(DialogType.Binder)` to find the correct row by `optionText` ("Make this inn your home"), NOT pass an enum value as a literal `gossipListId` (BRIEF CORRECTION D). Server cascade via `Player::OnGossipSelect → SendBindPoint(unit) → CastSpell(player, 3286, true) → SMSG_BINDPOINTUPDATE`. Selecting `GOSSIP_OPTION_INNKEEPER (8)` is equivalent to dispatching `CMSG_BINDER_ACTIVATE(npcGuid)` directly.

**Phase 4 — Bind-point storage + snapshot.** Server's `SendBindPoint` ([`NPCHandler.cpp:519-529`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) early-returns silently on instance maps (`GetMap()->Instanceable()`) — meaningful only on continent maps. On success, `npc->CastSpell(_player, 3286, true)` triggers spell 3286 ("Bind") which has a `SPELL_EFFECT_BIND` handler that calls `Player::SetHomebindToLocation` (via `Player::PlayerTalkClass->CloseGossip()` at [`NPCHandler.cpp:528`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp) plus the spell's `EFFECT_BIND` script). Server writes `m_homebind = {mapId, areaId, x, y, z}` ([`Player.cpp:16567-16573`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) + persists to `character_homebind` table + emits `SMSG_BINDPOINTUPDATE` to the player. Bot's `CharacterInitNetworkClientComponent.HandleBindPointUpdate` ([`CharacterInitNetworkClientComponent.cs:161-183`](../../../Exports/WoWSharpClient/Networking/ClientComponents/CharacterInitNetworkClientComponent.cs)) parses the 20-byte payload into `BindPointData(x, y, z, mapId, areaId)`, stores in `_bindPoint`, and emits via `_bindPointSubject.OnNext(data)` for DE consumption.

**Phase 5 — Hearthstone item check + cast.** With bind set, recall is invoked by using the Hearthstone item — itemId `6948`, cast spell `8690` ("Hearthstone"). Bot's `UseHearthstoneTask` ([`UseHearthstoneTask.cs`](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)) pre-checks `ObjectManager.Items.Any(i => i.ItemId == 6948)` ([:54-55](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)); if missing, pops immediately. Pre-cast: combat-abort (`Player.IsInCombat == true` pops the task), full-stop movement, then `ObjectManager.CastSpell("Hearthstone")` ([:73](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)) which routes through the standard item-use path → `CMSG_USE_ITEM(itemGuid, spellId=8690)`. Spell 8690 has a 10-second cast time (`CastDurationMs = 10_000` per [:31](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)). The cast is **channel-interruptible** by damage, manual stop, or movement — `WaitForCast` polls `!Player.IsChanneling && !Player.IsCasting` after 2s grace and aborts on interrupt ([:87-93](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)).

**Phase 6 — Teleport detection + cooldown.** On cast completion, server runs `Player::TeleportToHomebind(options, hearthCooldown=true)` ([`Player.cpp:21374-21383`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) which (a) initiates 60-minute cooldown via `AddCooldown(*spellInfo, itemProto)` (applied to BOTH spell 8690 + item 6948 — the cooldown is item-bound, not spell-bound, so any future Hearthstone respects it) and (b) teleports to `m_homebind` coordinates. Server emits `SMSG_NEW_WORLD` (cross-map) or `SMSG_TRANSFER_PENDING` + `SMSG_MOVE_TELEPORT_ACK` (same-map). Bot detects via `(position.DistanceTo(_startPosition) > 100f) OR (player.MapId != _startMapId)` ([`UseHearthstoneTask.cs:106`](../../../Exports/BotRunner/Tasks/Travel/UseHearthstoneTask.cs)) within `TeleportDetectTimeoutMs = 15_000` window. **Cross-map teleport** (e.g. Booty Bay → Stormwind on the Eastern Kingdoms → same-continent path is fine, but Crossroads → Orgrimmar crosses no map boundary) takes longer than the 10s cast — the 15s detection window covers loading.
