# Vendor — Visibility → Sub-Type → Range → List-Inventory → Buy-List → Budget → Buy/Sell/Repair → Buyback → Reagent Restock

Second NPC sub-family after [`quest-giver.md`](quest-giver.md). Where the quest-giver doc opens the **out-of-combat NPC dialog cycle**, **vendor** opens the **out-of-combat economic cycle** that closes two named cross-doc gaps:

1. **Reagent-restock gap** from [`../combat/buff-cycle.md`](../combat/buff-cycle.md) Self-buffer/Druid/Party-buffer rows — `PowerWordFortitude` requires Holy Candle (item `17029`) / Sacred Candle (`17028`); `MarkOfTheWild` requires Wild Berries (`17021`) / Wild Thornroot (`17022`) / Wild Quillvine (`22147`) / Wild Spineleaf (`17026`); `GreaterBlessingOfKings` family requires Symbol of Kings (`21177`). Buff-cycle R8 marks reagent gating as "silent failure if reagent missing" — vendor.md is the restock surface that ConsumableData already enumerates.
2. **Food/drink stack-out cascade gap** from [`../combat/utility-casts.md`](../combat/utility-casts.md) **R15** — `ConsumableData.FindBestFood == null` produces a silent no-op in `RestTask`; utility-casts R15 explicitly flags "a future enhancement would cascade to `VendorVisitTask` for restock". This doc names that cascade.

The shipped state is asymmetric but more complete than quest-giver. ## Overview

Vendor interaction in 1.12.1 splits into **five phases**, each with distinct server-side opcodes and bot-side dispatch patterns.

**Phase 1 — Visibility + sub-type detection.** A vendor is any `Creature` with `UNIT_NPC_FLAG_VENDOR = 0x04` in its `UNIT_NPC_FLAGS` UpdateField. The bot's `IObjectManager.Units` enumeration sees the bit on every nearby creature snapshot; predicate `unit.NpcFlags & 0x04 != 0` selects vendors. Repair-capable vendors additionally carry `UNIT_NPC_FLAG_REPAIR = 0x4000`. `VendorVisitTask.FindVendor` ([`VendorVisitTask.cs:87-120`](../../../Exports/BotRunner/Tasks/VendorVisitTask.cs)) already prefers repair vendors over plain vendors via two `OrderBy(distance).FirstOrDefault()` filters with a `??` short-circuit. **Sub-types are NOT separate npcflag bits in 1.12.1** — ammo/food/poison/reagent specialization is encoded in the `npc_vendor` rows + the creature's gossip menu. The DE infers sub-type from inventory after Phase 3, NOT from the npcflag.

**Phase 2 — Range gate + interact.** Once the bot is within `INTERACTION_DISTANCE = 5.0f` yards ([`ObjectDefines.h:24`](D:/MaNGOS/source/src/game/Objects/ObjectDefines.h)) of a vendor, dispatch `InteractWithUnitTask(npcGuid)` analog (in `VendorVisitTask` this is inline `MoveToVendor` → `InteractVendor`). **Gossip overlay**: see [`gossip.md` §Overview](gossip.md#overview) Phases 2-4 for canonical Hello/menu/select mechanics. Per-NPC cascade entry: `GOSSIP_OPTION_VENDOR = 3` (or `ARMORER = 15`) ([`gossip.md` §Overview Phase 4 table](gossip.md#overview)) → server runs `GetSession()->SendListInventory(guid, ...)` and the bot continues into this doc's R4 (list-inventory query). For direct-route vendors lacking `0x01 GOSSIP`, dispatch `CMSG_LIST_INVENTORY` straight to `SendListInventory` skipping the overlay. Vendor-specific hook: [`ItemHandler.cpp:767-776`](D:/MaNGOS/source/src/game/Handlers/ItemHandler.cpp) auto-cancels feign-death and pauses NPC movement on every vendor opcode entry — same pattern as the standard gossip hook at [`gossip.md` §Pre-conditions](gossip.md#pre-conditions).

**Phase 3 — List-inventory.** `CMSG_LIST_INVENTORY(vendorGuid)` ([`ItemHandler.cpp:745-754`](D:/MaNGOS/source/src/game/Handlers/ItemHandler.cpp)) → server replies `SMSG_LIST_INVENTORY` with up to `MAX_VENDOR_ITEMS = 128` items ([`CreatureDefines.h:656`](D:/MaNGOS/source/src/game/Objects/CreatureDefines.h)). The reply payload per-item is `(uint32 slot, uint32 itemId, uint32 displayInfoID, uint32 maxCount, uint32 price, uint32 maxDurability, uint32 buyCount)` per [`ItemHandler.cpp:847-853`](D:/MaNGOS/source/src/game/Handlers/ItemHandler.cpp). The server applies reputation-discount via `_player->GetReputationPriceDiscount(pCreature)` ([`ItemHandler.cpp:802`](D:/MaNGOS/source/src/game/Handlers/ItemHandler.cpp)) and filters items by `AllowableClass / AllowableRace / RequiredReputation` ([`ItemHandler.cpp:812-839`](D:/MaNGOS/source/src/game/Handlers/ItemHandler.cpp)) — vendor inventory is **per-player class/race/rep filtered**. `IVendorNetworkClientComponent.GetAvailableItems()` ([`IVendorNetworkClientComponent.cs:49`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IVendorNetworkClientComponent.cs)) surfaces the filtered list.

**Phase 4 — Buy-list traversal + sell-junk + repair.** With the inventory in hand, the bot's `DoVendorActions` assembles three concurrent transactions:
- **Sell-junk pass**: `IVendorNetworkClientComponent.SellAllJunkAsync(vendorGuid, options)` — server iterates player's bags, selling all `Quality == ITEM_QUALITY_POOR (0)` non-quest non-soulbound items via `CMSG_SELL_ITEM(vendorGuid, itemGuid, count=0)` (whole stack). Each sale credits `pProto->SellPrice * count` to `Player.Coinage` and pushes the sold item into the **buyback slot ring** ([`Player.cpp` `AddItemToBuyBackSlot`](D:/MaNGOS/source/src/game/Objects/Player.cpp); 12 slots between `BUYBACK_SLOT_START = 69` and `BUYBACK_SLOT_END = 81` per [`Player.h:612-614`](D:/MaNGOS/source/src/game/Objects/Player.h)). Each sale **partially refunds** if the item has lost durability — server computes `repaircost = LostDurability * dmultiplier * dQualitymodEntry->quality_mod` and subtracts ([`ItemHandler.cpp:578-614`](D:/MaNGOS/source/src/game/Handlers/ItemHandler.cpp)).
- **Buy-list pass**: iterate `Dictionary<itemId, quantity>` from `ConsumableData.GetConsumablesToBuy(om, level, usesMana, class)` ([`ConsumableData.cs:340+`](../../../Exports/BotRunner/Combat/ConsumableData.cs)) — emits per-item `CMSG_BUY_ITEM(vendorGuid, itemId, count, 1)`. The server's `Player::BuyItemFromVendor` deducts `pProto->BuyPrice * count` (reputation-discounted) from `Player.Coinage` and stores in the first available bag slot.
- **Repair pass**: `IVendorNetworkClientComponent.RepairAllItemsAsync(vendorGuid)` → `CMSG_REPAIR_ITEM(vendorGuid, itemGuid=0)` triggers repair-all (per the protocol convention `itemGuid=0` means "all equipped + inventory items at this vendor"). Gold cost is the sum of per-item `LostDurability * dmultiplier` deductions. `NetworkMerchantFrame.RepairAll` ([`NetworkMerchantFrame.cs:127-133`](../../../Exports/WoWSharpClient/Frames/NetworkMerchantFrame.cs)) wires this. Per-slot repair (`RepairByEquipSlot`) collapses to RepairAll because the WoW protocol exposes total-only cost ([`NetworkMerchantFrame.cs:111-119`](../../../Exports/WoWSharpClient/Frames/NetworkMerchantFrame.cs)).

**Phase 5 — Buyback safety + close.** The buyback ring buffer holds the last 12 sold items ([`Player.h:612-614`](D:/MaNGOS/source/src/game/Objects/Player.h)) — `CMSG_BUYBACK_ITEM(vendorGuid, slot)` reverses a sale at full original sell-price (no profit/loss for the player). DE policy: if the sell-junk pass accidentally sold a green-or-better item (BoE upgrade missed by the junk-filter), recover via buyback **before closing the vendor window** — the buyback slots are scoped to this vendor only (re-opening another vendor clears the ring). After all dispatches complete, the BG bot calls `IVendorNetworkClientComponent.CloseVendorAsync()` ([`IVendorNetworkClientComponent.cs:45`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IVendorNetworkClientComponent.cs)) which closes the merchant frame; the FG bot's frame closes via the Lua `CloseMerchant()` bridge.
