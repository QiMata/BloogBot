# Flight Master — Visibility → Range → Hello → Taxi-Map → Known-Mask Intersect → Multi-Stop Route → Gold Gate → Activate-Taxi → Spline → Landing

Fifth NPC sub-family after [`quest-giver.md`](quest-giver.md), [`vendor.md`](vendor.md), [`trainer.md`](trainer.md), and [`gossip.md`](gossip.md). Where the prior four cover **dialog**, **economy**, **spell-acquisition**, and the **dialog overlay**, this one covers the only NPC sub-family whose interaction outcome is **bot-position teleport via a server-driven spline movement generator** — the **taxi-network protocol** that bridges Activity-layer travel-Objectives that exceed the bot's run-budget on land.

The shipped state is **the most structurally novel of the five NPC sub-families.** Per [`gossip.md`](gossip.md) iter-23, flight-master is one of four NPC sub-families that the gossip overlay routes through (`TAXIVENDOR` cascade at `Player::PrepareGossipMenu`); but where the other three (quest-giver, vendor, trainer) all end in a **dialog frame** that the bot drives via per-frame UI/component APIs, flight-master ends in a **server-side spline movement** (`MoveTaxiFlight` motion-generator at [`TaxiHandler.cpp:117-119`](D:/MaNGOS/source/src/game/Handlers/TaxiHandler.cpp)) which the bot **cannot influence mid-flight** — landing detection is the only handle. Per the [`wwow_bg_frame_pattern`](../../../../memory/wwow_bg_frame_pattern.md) memory, `NetworkTaxiFrame` SHIPPED in S1.18 ([`Exports/WoWSharpClient/Frames/NetworkTaxiFrame.cs`](../../../Exports/WoWSharpClient/Frames/NetworkTaxiFrame.cs)) — both FG and BG bots expose the same `ITaxiFrame.IsOpen / Nodes[] / SelectNode(int) / Close()` surface, routing through `IFlightMasterNetworkClientComponent.ActivateFlightAsync` on BG. `FlightMasterVisitTask` ([`Exports/BotRunner/Tasks/FlightMasterVisitTask.cs`](../../../Exports/BotRunner/Tasks/FlightMasterVisitTask.cs)) drives a four-state **discovery** machine (`FindFlightMaster → MoveToFlightMaster → DiscoverNodes → Done`) that dispatches `IObjectManager.DiscoverTaxiNodesAsync(fmGuid, ct)` and tracks a session-level visited set; `TakeFlightPathTask` ([`Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)) drives a seven-state **flight** machine (`FindFlightMaster → NavigateToFM → InteractWithFM → WaitForTaxiWindow → ActivateFlight → InFlight → Complete`) ending in stationary-tick landing detection.

The server-side state machine spans **four CMSG opcodes** (plus express + admin variants). **Visibility** is `UNIT_NPC_FLAG_FLIGHTMASTER = 0x00000008` ([`UnitDefines.h:663`](D:/MaNGOS/source/src/game/Objects/UnitDefines.h)) — **NOT** `0x04` (`UNIT_NPC_FLAG_VENDOR` per :662), **NOT** `0x10` (`UNIT_NPC_FLAG_TRAINER` per :664 — the iter-22 trainer value), **NOT** `0x80` (`UNIT_NPC_FLAG_INNKEEPER` per :667). ## Overview

Flight-master interaction in 1.12.1 splits into **six phases**, each with distinct server-side opcodes and bot-side dispatch patterns.

**Phase 1 — Visibility detection.** A flight master is any `Creature` with `UNIT_NPC_FLAG_FLIGHTMASTER = 0x08` in its `UNIT_NPC_FLAGS` UpdateField. The bot's `IObjectManager.Units` enumeration sees the bit on every nearby creature snapshot; predicate `unit.NpcFlags & 0x08 != 0` selects flight masters ([`FlightMasterVisitTask.cs:82`](../../../Exports/BotRunner/Tasks/FlightMasterVisitTask.cs); [`TakeFlightPathTask.cs:55`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)). Most flight masters also carry `UNIT_NPC_FLAG_GOSSIP = 0x01` for the "I want to take a flight" dialog option — but the bot can bypass the menu-select step entirely (see Phase 2 below).

**Phase 2 — Range gate + interact.** Once within `INTERACTION_DISTANCE = 5.0f` yards ([`ObjectDefines.h:24`](D:/MaNGOS/source/src/game/Objects/ObjectDefines.h)), dispatch `_flightMaster.Interact()` ([`TakeFlightPathTask.cs:84`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)) which sends `CMSG_GOSSIP_HELLO`. **Gossip overlay**: see [`gossip.md` §Overview](gossip.md#overview) Phases 2-4 for canonical Hello/menu/select mechanics. Per-NPC cascade entry: `GOSSIP_OPTION_TAXIVENDOR = 4` ([`gossip.md` §Overview Phase 4 table](gossip.md#overview)) → server runs `GetSession()->SendTaxiMenu(pCreature)` which emits `SMSG_SHOWTAXINODES`. **Unique twist**: the bot can also dispatch `CMSG_TAXIQUERYAVAILABLENODES(fmGuid)` directly without ever selecting a gossip option — the taxi flow accepts both paths interchangeably. Flight-master-specific gate: `CMSG_TAXIQUERYAVAILABLENODES` is gated by `GetNPCIfCanInteractWith(guid, UNIT_NPC_FLAG_FLIGHTMASTER)` ([`TaxiHandler.cpp:68`](D:/MaNGOS/source/src/game/Handlers/TaxiHandler.cpp)) — out-of-range drops the opcode silently with `LOG_LVL_DEBUG "not found or you can't interact with him"`.

**Phase 3 — Discovery (one-bit known-mask flip).** Server's `SendLearnNewTaxiNode` ([`TaxiHandler.cpp:122-144`](D:/MaNGOS/source/src/game/Handlers/TaxiHandler.cpp)) compares the flight master's nearest taxi node (`GetNearestTaxiNode(creaturePos)`) against `Player::m_taximask` (the player's bitmap of known nodes — one bit per `TaxiNodes.dbc` entry, ~140 nodes in 1.12.1). If the bit is unset, `SetTaximaskNode(curloc)` flips it, emits `SMSG_NEW_TAXI_PATH` (the "Flight Path Discovered" toast) + `SMSG_TAXINODE_STATUS(guid, 1)`, and returns `true` from `SendLearnNewTaxiNode` — which **short-circuits the subsequent `SendTaxiMenu` call** (taxi map does NOT open on first visit; bot must Hello again to actually pick a destination). On known-node revisit, returns `false` and falls through to `SendTaxiMenu` which emits `SMSG_SHOWTAXINODES`. Bot-side `FlightMasterVisitTask.DiscoverNodes` ([`FlightMasterVisitTask.cs:120-157`](../../../Exports/BotRunner/Tasks/FlightMasterVisitTask.cs)) dispatches `IObjectManager.DiscoverTaxiNodesAsync(fmGuid, ct)` which under the hood emits `HelloFlightMasterAsync` + `QueryAvailableNodesAsync` and listens on the `TaxiMapOpened` observable.

**Phase 4 — Available-nodes query + route planning.** `SMSG_SHOWTAXINODES` carries `(uint32=1, ObjectGuid flightMasterGuid, uint32 currentNodeId, taximask blob)` ([`TaxiHandler.cpp:95-100`](D:/MaNGOS/source/src/game/Handlers/TaxiHandler.cpp)) where the `taximask blob` is the player's `m_taximask` bitset — **all globally-known nodes**, not just reachable from this flight master. The BG agent stores this in `IFlightMasterNetworkClientComponent.AvailableTaxiNodes` ([`IFlightMasterNetworkClientComponent.cs:24`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IFlightMasterNetworkClientComponent.cs)) + `CurrentNodeId` ([:29](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IFlightMasterNetworkClientComponent.cs)). **The reachable subset from this flight master is determined client-side** via the `taxi_path` table — every (sourceNode, destNode) pair has a direct path entry or chains through `taxi_path_transitions`. Bot-side route planning intersects `AvailableTaxiNodes` (known mask) with the per-source-node reachable set (DBC `TaxiPath` table; future SCo.FLIGHTMASTER.1) to produce the candidate dest list. Cost per leg comes from `GetFlightCost(nodeId)` which is null until the bot has flown that route once (cost is populated post-fact from `SMSG_ACTIVATETAXIREPLY`, [`NetworkTaxiFrame.cs:47-56`](../../../Exports/WoWSharpClient/Frames/NetworkTaxiFrame.cs)).

**Phase 5 — Activate-taxi + 11-error-gate cascade.** With source + dest in hand, emit `CMSG_ACTIVATETAXI(flightMasterGuid, sourceNode, destNode)` ([`TaxiHandler.cpp:174-190`](D:/MaNGOS/source/src/game/Handlers/TaxiHandler.cpp)) which dispatches into `Player::ActivateTaxiPathTo` ([`Player.cpp:18240-18494`](D:/MaNGOS/source/src/game/Objects/Player.cpp)). The cascade gates **11 sequential predicates** before liftoff:

**Phase 6 — In-flight monitor + landing detection.** During `FLIGHT_MOTION_TYPE` the bot has **no movement authority** — server pushes spline waypoints from `sTaxiPathNodesByPath[lastPath]` ([`Player.cpp:18424-18435`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) and the player position updates arrive in `SMSG_MOVE_*` packets. Per `TakeFlightPathTask` ([`TakeFlightPathTask.cs:130-156`](../../../Exports/BotRunner/Tasks/Travel/TakeFlightPathTask.cs)), landing detection is **6 consecutive 500ms ticks (3 seconds) of stationary position** AFTER `>= 5000ms` minimum in-flight — guards against false-landing at the takeoff stationary tick. Multi-stop routes (cross-continent zeppelin-bridge chains, e.g. Orgrimmar→Grom'gol→Booty Bay) emit a single `CMSG_ACTIVATETAXI` with source+final-dest; server expands the chain via `taxi_path_transitions` table ([`Player.cpp:18384-18437`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) and pushes the concatenated spline. **Multi-stop is server-arbitrated**, not bot-controlled — the bot's only handle is `CMSG_ACTIVATETAXIEXPRESS` with an explicit node array for client-arbitrated chains.
