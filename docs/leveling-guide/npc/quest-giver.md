# Quest Giver ‚Äî Visibility ‚Üí Status-Query ‚Üí Hello/Gossip ‚Üí Details ‚Üí Accept ‚Üí Objectives ‚Üí Complete ‚Üí Reward-Choice ‚Üí Abandon ‚Üí Log-Cap

Opens the NPC interaction sub-family. Where [combat/melee-rotation.md](../combat/melee-rotation.md) drives the swing/cooldown ladder and [combat/utility-casts.md](../combat/utility-casts.md) drives the residual combat-aware utility surface, **quest-giver** is the foundational **out-of-combat NPC dialog cycle** that drives the entire Questing Activity family ‚Äî every quest row in [`../../Plan/Activities/00_INDEX.md`](../../Plan/Activities/00_INDEX.md) (6 starter + 29 zone rows) walks this pipeline. This is the doc-side counterpart to [`../../Plan/Activities/quests.md`](../../Plan/Activities/quests.md) ‚Äî `quests.md` is the slot/task-family tracker (SQ.1..SQ.7); **this doc is the DE-rules + protocol contract**.

The shipped state is asymmetric (per [`quests.md`](../../Plan/Activities/quests.md) ¬ßRequired task families): the **FG path** is driven through `IObjectManager.QuestFrame` Lua-button clicks ([`AcceptQuestTask.cs:15-25`](../../../Exports/BotRunner/Tasks/AcceptQuestTask.cs) / [`CompleteQuestTask.cs:15-25`](../../../Exports/BotRunner/Tasks/CompleteQuestTask.cs)); the **BG path** is driven through `IQuestNetworkClientComponent` CMSG packets ([`IQuestNetworkClientComponent.cs:77-154`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IQuestNetworkClientComponent.cs)). Both share the same `ObjectiveType` dispatch through `Exports/BotRunner/ActionDispatcher.cs`. Atomic-task wrappers shipped today: `AcceptQuestTask` (no `questId` overload ‚Äî pops QuestFrame only), `CompleteQuestTask(rewardIndex)` (no `(npcGuid, questId)` overload), `SelectGossipTask(optionIndex)` ([`SelectGossipTask.cs:11-25`](../../../Exports/BotRunner/Tasks/SelectGossipTask.cs)), `InteractWithUnitTask(targetGuid)` ([`InteractWithUnitTask.cs:17-72`](../../../Exports/BotRunner/Tasks/InteractWithUnitTask.cs) ‚Äî moves within `Config.NpcInteractRange`, sets target, then pops and lets StateManager dispatch follow-up gossip/accept). The orchestrating `QuestingTask` ([`Exports/BotRunner/Tasks/Questing/QuestingTask.cs`](../../../Exports/BotRunner/Tasks/Questing/QuestingTask.cs)) drives objective hotspot selection inline; the chain-aware `QuestChainRouter` ([`Exports/BotRunner/Tasks/Questing/QuestChainRouter.cs`](../../../Exports/BotRunner/Tasks/Questing/QuestChainRouter.cs)) is a scaffold (SQ.3 not-started).

**Per the [`wwow_bg_frame_pattern`](../../../../memory/wwow_bg_frame_pattern.md) memory: `NetworkQuestFrame` does NOT exist today** under [`Exports/WoWSharpClient/Frames/`](../../../Exports/WoWSharpClient/Frames/) (only `NetworkCraftFrame / NetworkGossipFrame / NetworkLootFrame / NetworkMerchantFrame / NetworkTalentFrame / NetworkTaxiFrame / NetworkTradeFrame / NetworkTrainerFrame` ship; QuestFrame is FG-only via `ForegroundBotRunner.Frames.FgQuestFrame`). **Open SCo.NPC.1: ship `NetworkQuestFrame` adapter** wiring `IQuestNetworkClientComponent` into `IQuestFrame` so BG bots can expose the same `IsOpen / State / NpcGuid / Quests[] / Rewards[]` surface FG bots already do. ## Overview

Quest-giver interaction in 1.12.1 splits into **five phases**, each with distinct server-side opcodes and bot-side dispatch patterns:

**Phase 1 ‚Äî Visibility detection.** A questgiver is any `Creature` with `UNIT_NPC_FLAG_QUESTGIVER = 0x02` in its `UNIT_NPC_FLAGS` UpdateField. The bot's `IObjectManager.Units` enumeration sees the bit on every nearby creature snapshot; predicate `unit.NpcFlags & 0x02 != 0` selects questgivers. A single creature may carry multiple flags (e.g. Innkeepers carry `0x10000` _and_ `0x02` if they offer quests). The minimap question-mark/exclamation icon is driven by `DIALOG_STATUS_*` per-quest-state ([`QuestDef.h:118-130`](D:/MaNGOS/source/src/game/QuestDef.h)) which is computed server-side per the `Player::GetQuestDialogStatus` scan over `qInfo` chains.

**Phase 2 ‚Äî Status query (visit-or-skip routing).** Before path-routing to an NPC, the DE issues `CMSG_QUESTGIVER_STATUS_QUERY` ([`Opcodes_1_12_1.h:389`](D:/MaNGOS/source/src/game/Protocol/Opcodes_1_12_1.h)) for each candidate within scan radius. The server replies with `SMSG_QUESTGIVER_STATUS` carrying a `DIALOG_STATUS_*` byte. Routing rules: `AVAILABLE / REWARD_REP / REWARD2 / REWARD_OLD` ‚Üí route to NPC; `INCOMPLETE` ‚Üí skip (objective elsewhere); `CHAT / NONE / UNAVAILABLE` ‚Üí skip. Implementation note: shipped today, this is bypassed ‚Äî `QuestingTask` reads `Player.QuestLog` directly + `Container.QuestRepository.GetQuestTemplateById` instead of polling per-NPC status. Open SCo.NPC.2 status-query batching for "unknown questgivers" (NPCs not yet in `Player.QuestLog` ‚Äî the discovery path).

**Phase 3 ‚Äî Hello / gossip.** Once the bot is within `INTERACTION_DISTANCE = 5.0f` yards ([`ObjectDefines.h:24`](D:/MaNGOS/source/src/game/Objects/ObjectDefines.h)) of an `AVAILABLE` questgiver, dispatch `InteractWithUnitTask(npcGuid)` which moves within `Config.NpcInteractRange` and issues `CMSG_QUESTGIVER_HELLO` ([`Opcodes_1_12_1.h:391`](D:/MaNGOS/source/src/game/Protocol/Opcodes_1_12_1.h)). **Gossip overlay**: see [`gossip.md` ¬ßOverview](gossip.md#overview) Phases 2-4 for canonical `CMSG_GOSSIP_HELLO` / `_SELECT_OPTION` mechanics. Per-NPC cascade entry: `GOSSIP_OPTION_QUESTGIVER = 2` ([`gossip.md` ¬ßOverview Phase 4 table](gossip.md#overview)) ‚Üí server runs `PrepareQuestMenu(guid) + SendPreparedQuest(guid)` and the bot continues into this doc's R5 (details query + accept). FG opens the WoW QuestFrame UI; BG observes through `IQuestNetworkClientComponent.QuestOperations` ([`IQuestNetworkClientComponent.cs:29`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/IQuestNetworkClientComponent.cs)). Quest-specific hook: [`QuestHandler.cpp:94-100`](D:/MaNGOS/source/src/game/Handlers/QuestHandler.cpp) auto-cancels feign-death (`UNIT_STAT_FEIGN_DEATH`) and pauses NPC movement on `CMSG_QUESTGIVER_HELLO` (parallel to the standard gossip channel/feign hook described at [`gossip.md` ¬ßPre-conditions](gossip.md#pre-conditions)).

**Phase 4 ‚Äî Per-quest details + accept.** The bot's `QuestingTask` selects a quest from the offered list (priority order: chain quest matching `CharacterBuildConfig.QuestChains` ‚Üí ungrouped quest matching level ‚Äî see [`QuestChainRouter.cs`](../../../Exports/BotRunner/Tasks/Questing/QuestChainRouter.cs)). For each, issue `CMSG_QUESTGIVER_QUERY_QUEST` for the details payload (if not already cached in the local `QuestRepository`), then `CMSG_QUESTGIVER_ACCEPT_QUEST` carrying `(guid, questId)`. The server-side `CanTakeQuest` cascade ([`Player.cpp:13015-13039`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) gates acceptance through **8 predicates**: `SatisfyQuestLevel` (player.Level + 7 vs quest.MinLevel; trivial gate uses `+10`); `SatisfyQuestClass` (`ReqClasses` bitmask in `quest_template`); `SatisfyQuestRace` (`ReqRaces`); `SatisfyQuestSkill` (`ReqSkill` rank); `SatisfyQuestReputation` (`ReqMinRepFaction`); `SatisfyQuestPrevChain` (`PrevQuestId` must be in player's rewarded set); `SatisfyQuestNextChain` (`NextQuestId` must NOT be rewarded ‚Äî exclusivity); `SatisfyQuestExclusiveGroup` (`ExclusiveGroup` row ‚Äî one-of-N quests). On reject, server emits `SMSG_QUESTGIVER_QUEST_INVALID` carrying an `INVALIDREASON_*` byte ([`QuestDef.h:45-58`](D:/MaNGOS/source/src/game/QuestDef.h)).

**Phase 5 ‚Äî Objective progress + turn-in + reward-choice.** Objective progress arrives passively as `SMSG_QUESTUPDATE_ADD_KILL` / `_ADD_ITEM` (handled by `WoWSharpClient.Handlers.QuestHandler` ‚Üí `IObjectManager.UpdateQuestKillProgress / UpdateQuestItemProgress`). When all objectives are met server-side, `SMSG_QUESTUPDATE_COMPLETE` flips `QuestSlot.QuestState` from `0` to `1` (`QUEST_STATUS_COMPLETE` per [`QuestDef.h:91-100`](D:/MaNGOS/source/src/game/QuestDef.h)). The bot re-routes to the turn-in NPC (often a different creature than the issuer; resolved via `quest_template.QuestTurnInGiver` or `quest_giver` involvement table), issues `CMSG_QUESTGIVER_HELLO` to reopen the gossip frame in `REWARD2 / REWARD_OLD` mode, then `CMSG_QUESTGIVER_COMPLETE_QUEST(guid, questId)` to fetch the reward UI (`SMSG_QUESTGIVER_OFFER_REWARD` with up to 6 choice rewards + up to 4 always-given rewards). Finally `CMSG_QUESTGIVER_CHOOSE_REWARD(guid, questId, rewardIndex)` commits the selection ‚Äî server-side [`QuestHandler.cpp:399-407`](D:/MaNGOS/source/src/game/Handlers/QuestHandler.cpp) validates `rewardIndex < QUEST_REWARD_CHOICES_COUNT (6)` and logs `LOG_LVL_ERROR` on invalid index ("probably packet hacking"). On success: `Player.QuestLogEntries` loses the slot, `Player.XP` increments, `Player.Coinage` increments, `Player.Inventory` gains the reward item. The "always picks a reward" invariant ([`quests.md` R18](../../Plan/Activities/quests.md)) is enforced through `IRewardSelector` (Phase 2, not-started); today `CompleteQuestTask` defaults `rewardIndex = 0`.

## Live-test acceptance

A live integration test exercising this Activity must satisfy R8 (clean cross-test state) by:

1. **Server-up assertion** ‚Äî `ServerHealthcheck` returns 200 before the test starts; if not, FAIL FAST (per root R4 fail-fast rule). Do not retry-loop on a dead server.
2. **Pre-test snapshot predicate** ‚Äî `Player.Health > 0 AND Player.IsInCombat == false AND Player.QuestLog.Any(q => q.QuestId == TargetQuestId) == false` before triggering the accept (clean start; quest not already on log).
3. **NPC spawn assertion** ‚Äî `IObjectManager.Units.Any(u => u.Entry == TargetQuestGiverEntry && u.NpcFlags & 0x02 != 0 && u.Health > 0)` within 5s of test start. **Do NOT spawn the NPC** per [`CLAUDE.md` Test Skip Policy](../../../CLAUDE.md) ‚Äî natural world state only; failure here is a real bug (NPC despawn / DB gap).
4. **Hello dispatch predicate** ‚Äî push `InteractWithUnitTask(npcGuid)` via fixture; poll `Player.Position.DistanceTo(npc.Position) <= 5.0f` within 30s (varies by start position ‚Äî bot needs path-route to NPC); then poll `IObjectManager.QuestFrame.IsOpen == true (FG)` OR `IQuestNetworkClientComponent.CurrentQuestGiver == npcGuid (BG)` within 3s of arrival.
5. **Accept dispatch predicate** ‚Äî dispatch via Activity-level mechanism (today: `AssignedActivity = "Quest[<id>]"` per fixture; tomorrow: `IActivity.Start(...)` post-S2.0). Poll `Player.QuestLog.Any(q => q.QuestId == TargetQuestId && q.QuestState == 0)` within 3s. **Do NOT construct `new ObjectiveMessage { ObjectiveType = ObjectiveType.AcceptQuest }` in the test body** ‚Äî that violates [`CLAUDE.md` Test Isolation Rules](../../../CLAUDE.md) (remote-controlling the bot, bypassing DecisionEngine).
6. **Objective progress predicate** ‚Äî for kill/collect quest objectives, poll `Player.QuestLog[slot].objectives[i].currentCount >= requiredCount` within a per-quest budget. **Do not stub the kill** ‚Äî bot must complete via real combat (cross-cite [`combat/melee-rotation.md`](../combat/melee-rotation.md) acceptance). On all objectives met, server pushes `SMSG_QUESTUPDATE_COMPLETE` and `QuestState` flips to `1`.
7. **Turn-in dispatch predicate** ‚Äî route bot back to turn-in NPC (often different from issuer; resolved via `quest_template.QuestTurnInGiver`); re-issue Hello; poll `QuestFrame.State == QuestFrameState.Complete` within 3s; dispatch Complete + ChooseReward (defaulting `rewardIndex = 0` today). Poll `Player.QuestLog.Any(q => q.QuestId == TargetQuestId) == false` within 3s (slot freed). Poll `Player.XP` incremented by `quest.RewardXP` and (if money reward) `Player.Coinage` incremented by `quest.RewOrReqMoney`.
8. **Disconnect guard** ‚Äî mid-test client disconnect must fail the test (R8); do not paper over with `ReconnectTask` (separate Activity per [`../recovery/reconnect.md`](../recovery/reconnect.md)).
9. **Screenshot capture** ‚Äî per R11 in root CLAUDE.md, capture FG client screenshots at: (a) pre-Hello (target acquired, NPC in view, distance > 5y), (b) gossip-open (QuestFrame visible with quest title), (c) accepted (quest visible in player quest log), (d) turn-in (reward UI visible, choice rewards listed), (e) post-turn-in (quest removed from log, reward item in inventory). Overwrite `Tests/artifacts/npc/quest-giver/latest-{prehello,gossip,accepted,reward,postturnin}.png`.
10. **Final state dump** ‚Äî write `Snapshot.Player.QuestLog` + last 20 `WoWActivitySnapshot` deltas (including `RecentCommandAcks` for each dispatched CMSG and any `RecentErrors` carrying `INVALIDREASON_*` bytes) + `IQuestNetworkClientComponent.CurrentQuestGiver` final state to `Tests/artifacts/npc/quest-giver/snapshot-final.json` for agent post-mortem.
