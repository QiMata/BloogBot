# Trainer — Visibility → Sub-Type → Range → Trainer-List → Spell-State → Multi-Rank Queue → Gold Gate → Buy-Spell → Spellbook Refresh

Third NPC sub-family after [`quest-giver.md`](quest-giver.md) and [`vendor.md`](vendor.md). Where quest-giver opens the out-of-combat dialog cycle and vendor opens the out-of-combat economic cycle, **trainer** opens the **out-of-combat spell-acquisition cycle** that closes the long-standing **Spellbook-acquisition gap** named by both [`../combat/buff-cycle.md`](../combat/buff-cycle.md) and [`../combat/utility-casts.md`](../combat/utility-casts.md):

1. **Buff-cycle SCo.1 gap** — `BlessingOfWisdom / BlessingOfSalvation / BlessingOfLight` plus all **five Greater Blessing variants** (`GreaterBlessingOfKings/Might/Wisdom/Salvation/Light/Sanctuary`) are absent from [`Exports/GameData.Core/Constants/Spellbook.cs`](../../../Exports/GameData.Core/Constants/Spellbook.cs) — only `BlessingOfKings/Might/Sanctuary` (Spellbook.cs:113-115) are constants today.
2. **Utility-casts SCo.1 gap** — `CurseOfElements / CurseOfTongues / CurseOfRecklessness / CurseOfShadow` (Warlock) and the CC family `Polymorph / Sap / Hibernate / Banish / WyvernSting` are absent from Spellbook.cs (only `CurseOfAgony` :38 ships today). The DE cannot reference these spells by-name predicate, the rotation cannot cast them, and `LearnAllAvailableSpellsAsync` is the only path that gets them onto the player's spellbook — symmetric with how vendor.md just closed the reagent-restock gap by naming the upstream surface.

The shipped state is **the most complete of the three NPC sub-families so far**. Per the [`wwow_bg_frame_pattern`](../../../../memory/wwow_bg_frame_pattern.md) memory, `NetworkTrainerFrame` SHIPPED in S1.19 ([`Exports/WoWSharpClient/Frames/NetworkTrainerFrame.cs`](../../../Exports/WoWSharpClient/Frames/NetworkTrainerFrame.cs)) — both FG and BG bots now expose the same `ITrainerFrame.IsOpen / Spells[] / TrainSpell(int spellIndex) / Close()` surface. `TrainerVisitTask` ([`Exports/BotRunner/Tasks/TrainerVisitTask.cs`](../../../Exports/BotRunner/Tasks/TrainerVisitTask.cs)) drives a four-state machine (`FindTrainer → MoveToTrainer → LearnSpells → Done`) that dispatches through `IObjectManager.LearnAllAvailableSpellsAsync(trainerGuid, ct)`. The atomic wrapper `TrainSpellTask(spellIndex)` ([`Exports/BotRunner/Tasks/TrainSpellTask.cs`](../../../Exports/BotRunner/Tasks/TrainSpellTask.cs)) operates against an already-open trainer frame (`ObjectManager.TrainerFrame.TrainSpell(spellIndex)`); the level-up orchestrator `LevelUpTrainerTask` ([`Exports/BotRunner/Tasks/Progression/LevelUpTrainerTask.cs`](../../../Exports/BotRunner/Tasks/Progression/LevelUpTrainerTask.cs)) ships hard-coded Orgrimmar (7 trainers) and Stormwind (8 trainers) entry-id + position tables for the level-up trigger. Profession-trainer scheduling lives in a parallel scaffold `ProfessionTrainerScheduler` ([`Exports/BotRunner/Tasks/Crafting/ProfessionTrainerScheduler.cs`](../../../Exports/BotRunner/Tasks/Crafting/ProfessionTrainerScheduler.cs)) — not yet wired into the main task-stack.

The server-side state machine spans **only two CMSG opcodes** + the gossip overlay (the trainer flow is simpler than vendor because there is no separate "list" CMSG aside from `CMSG_TRAINER_LIST` and no buyback). **Visibility** is `UNIT_NPC_FLAG_TRAINER = 0x00000010` ([`UnitDefines.h:664`](D:/MaNGOS/source/src/game/Objects/UnitDefines.h)) — **NOT** `0x80` (which is `UNIT_NPC_FLAG_INNKEEPER` per :667) and **NOT** the iter-21 vendor flag `0x04`. The full opcode table:

| CMSG | Decimal | Hex | Handler | Notes |
|---|---|---|---|---|
| `CMSG_TRAINER_LIST` | **432** | **0x01B0** | [`NPCHandler.cpp:97-103`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp) | Args `(ObjectGuid trainerGuid)`; server replies `SMSG_TRAINER_LIST (433 / 0x01B1)` with up to `maxcount` spells |

Authoritative numerics are 432 + 434 per [`Opcodes_1_12_1.h:435-439`](D:/MaNGOS/source/src/game/Protocol/Opcodes_1_12_1.h).

## Overview

Trainer interaction in 1.12.1 splits into **five phases**, each with distinct server-side opcodes and bot-side dispatch patterns.

**Phase 1 — Visibility + sub-type detection.** A trainer is any `Creature` with `UNIT_NPC_FLAG_TRAINER = 0x10` in its `UNIT_NPC_FLAGS` UpdateField. The bot's `IObjectManager.Units` enumeration sees the bit on every nearby creature snapshot; predicate `unit.NpcFlags & 0x10 != 0` selects trainers. Sub-type is then disambiguated by `creature_template.trainer_type` (NOT a separate npcflag bit) which is one of four values per [`CreatureDefines.h:30-36`](D:/MaNGOS/source/src/game/Objects/CreatureDefines.h):

| Constant | Value | Meaning | Server check |
|---|---|---|---|
| `TRAINER_TYPE_CLASS` | 0 | Class trainer (Warrior/Mage/etc.) | `pPlayer->GetClass() == GetCreatureInfo()->trainer_class` ([`Creature.cpp:1298-1336`](D:/MaNGOS/source/src/game/Objects/Creature.cpp)) — wrong-class players get a class-specific rejection gossip (Druid menu 4913, Hunter 10090, Mage 328, Paladin 1635, Priest 4436, Rogue 4797, Shaman 5003, Warlock 5836, Warrior 4985). |
| `TRAINER_TYPE_MOUNTS` | 1 | Mount trainer (race-gated) | `pPlayer->GetRace() == trainer_race` ([`Creature.cpp:1349-1391`](D:/MaNGOS/source/src/game/Objects/Creature.cpp)) — race-mismatch allowed only if `REP_EXALTED` with the trainer's faction (Dwarves can train Tauren mounts at exalted with Thunder Bluff, etc.). |
| `TRAINER_TYPE_TRADESKILLS` | 2 | Profession trainer (mining/herb/etc.) | `pPlayer->HasSpell(trainer_spell)` ([`Creature.cpp:1393-1402`](D:/MaNGOS/source/src/game/Objects/Creature.cpp)) — for specialization trainers (Master Tailor, Goblin Engineer) the player must already know the prerequisite `trainer_spell`. Filters by `reqSkill / reqSkillValue` per-spell at the trainer-list level. |
| `TRAINER_TYPE_PETS` | 3 | Hunter pet trainer | `pPlayer->GetClass() == CLASS_HUNTER` ([`Creature.cpp:1338-1347`](D:/MaNGOS/source/src/game/Objects/Creature.cpp)) — non-Hunters get gossip menu 3620 ("you don't have a pet"). |

**1.12.1 has NO Mage Frost/Fire/Arcane specialization trainer** — that's TBC's Wintertide Mageblade quest line. **No specialization sub-type exists in 1.12.1 vanilla**. The user prompt's "specialization-trainer" row is reserved for a future TBC-coverage doc; explicitly excluded here.

**Phase 2 — Range gate + interact.** Once the bot is within `INTERACTION_DISTANCE = 5.0f` yards ([`ObjectDefines.h:24`](D:/MaNGOS/source/src/game/Objects/ObjectDefines.h)) of a trainer, dispatch `InteractWithUnitTask(npcGuid)` analog (in `TrainerVisitTask` this is inline `MoveToTrainer` → `LearnSpells`, gated by `Config.NpcInteractRange` at [`TrainerVisitTask.cs:183-188`](../../../Exports/BotRunner/Tasks/TrainerVisitTask.cs)). **Gossip overlay**: see [`gossip.md` §Overview](gossip.md#overview) Phases 2-4 for canonical Hello/menu/select mechanics. Per-NPC cascade entry: `GOSSIP_OPTION_TRAINER = 5` ([`gossip.md` §Overview Phase 4 table](gossip.md#overview)) → server runs `GetSession()->SendTrainerList(guid)` and the bot continues into this doc's R4 (trainer-list query). All class trainers carry `UNIT_NPC_FLAG_GOSSIP = 0x01` so the overlay is always engaged; a small number of weapon-master trainers ship without `0x01` and accept direct `CMSG_TRAINER_LIST` instead. Trainer-specific hook: [`NPCHandler.cpp:175-176`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp) auto-cancels channels (`AURA_INTERRUPT_INTERACTING_CANCELS`) on list-receipt — same pattern as the standard gossip channel-interrupt at [`gossip.md` §Pre-conditions](gossip.md#pre-conditions).

**Phase 3 — Trainer-list query.** `CMSG_TRAINER_LIST(trainerGuid)` ([`NPCHandler.cpp:97-103`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) → `SendTrainerList(guid)` ([:149-251](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) → server replies `SMSG_TRAINER_LIST` with header `(ObjectGuid + uint32 trainerType + uint32 count)` + per-spell 38-byte row + greeting title string. The server filters **per-spell** by `_player->IsSpellFitByClassAndRace(triggerSpell)` ([:217](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) and applies reputation discount via `_player->GetReputationPriceDiscount(unit)` ([:204](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)). Each spell carries a `TrainerSpellState` byte ([`Player.h:146-149`](D:/MaNGOS/source/src/game/Objects/Player.h)):

| State | Value | Meaning |
|---|---|---|
| `TRAINER_SPELL_GREEN` | 0 | Available — can be learned right now |
| `TRAINER_SPELL_RED` | 1 | Locked — level / class / race / prev-rank / req-skill / req-spell unmet |
| `TRAINER_SPELL_GRAY` | 2 | Already known (server detects via `HasSpell(triggerSpell)`) |
| `TRAINER_SPELL_GREEN_DISABLED` | 10 | Custom — primary-profession cap reached (server reports as `GREEN`, [`NPCHandler.cpp:121`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) |

The full `GetTrainerSpellState` cascade ([`Player.cpp:4713-4782`](D:/MaNGOS/source/src/game/Objects/Player.cpp)) gates each spell through **7 sequential predicates**: `(1) HasSpell(triggerSpell)` → GRAY; `(2) IsSpellFitByClassAndRace(triggerSpell)` → RED; `(3) GetLevel() < spellLevel` → RED; `(4) prev-rank check (spellChain->prev && !HasSpell(prev))` → RED; `(5) req-spell check (spellChain->req && !HasSpell(req))` → RED; `(6) GetSkillValueBase(reqSkill) < reqSkillValue` → RED; `(7) primary-profession cap (`GetFreePrimaryProfessionPoints() == 0`) → GREEN_DISABLED. Server cap `maxcount = (cSpells ? cSpells->spellList.size() : 0) + (tSpells ? tSpells->spellList.size() : 0)` ([:178](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) — class trainers ship ~60-90 spells per class across all 60 player levels.

**Phase 4 — Multi-rank purchase queue + buy-spell dispatch.** With the trainer list in hand, the bot's `LearnSpells` ([`TrainerVisitTask.cs:194-225`](../../../Exports/BotRunner/Tasks/TrainerVisitTask.cs)) collapses to a single bulk operation `IObjectManager.LearnAllAvailableSpellsAsync(trainerGuid, ct)` which iterates server-side-filtered `GREEN` spells and emits `CMSG_TRAINER_BUY_SPELL(trainerGuid, spellId)` per spell. **Multi-rank progression** is intrinsic: at level 16 a Mage can train Fireball rank 1, 2, 3, and 4 (rank-1 unlocks at level 1; rank-2 at level 6; rank-3 at level 12; rank-4 at level 18 — `GREEN` state automatic per the level + prev-rank cascade). Server-side `HandleTrainerBuySpellOpcode` ([`NPCHandler.cpp:270-355`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) **re-validates the entire `GetTrainerSpellState` cascade** before charging — out-of-state `BUY` returns `TRAIN_FAIL_NOT_ENOUGH_SKILL` ([:312-314](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)). Cost arithmetic: `nSpellCost = uint32(trainer_spell->spellCost * GetReputationPriceDiscount(unit) + 0.5f)` ([:321](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) — gold cost ranges from 1c (rank-1 at level 1) to 1g 80s (Greater Blessing of Wisdom at level 60). Server enforces `_player->GetMoney() < nSpellCost → TRAIN_FAIL_NOT_ENOUGH_MONEY` ([:324-328](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)). Server also enforces `IsWithinLOSInMap(_player)` ([:280](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) **on every buy** — LOS-break mid-purchase aborts with `TRAIN_FAIL_UNAVAILABLE`.

**Phase 5 — Spellbook refresh + close.** After each successful `CMSG_TRAINER_BUY_SPELL`, the server replies `SMSG_TRAINER_BUY_SUCCEEDED(guid, spellId)` ([`NPCHandler.cpp:253-258`](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp)) + applies the spell-cast that triggers `SPELL_EFFECT_LEARN_SPELL` (server-side spell trigger at [:336-345](D:/MaNGOS/source/src/game/Handlers/NPCHandler.cpp) — the `SpellVisual == 222` branch is the trainer-learn animation). Player's `m_spells` map gains the entry; `ObjectManager.RefreshSpells()` ([`TrainerVisitTask.cs:217`](../../../Exports/BotRunner/Tasks/TrainerVisitTask.cs)) re-reads the spellbook for the bot's local cache. On `TrainerBuyFailed`, the failure reason is one of `TRAIN_FAIL_UNAVAILABLE = 0 / TRAIN_FAIL_NOT_ENOUGH_MONEY = 1 / TRAIN_FAIL_NOT_ENOUGH_SKILL = 2` ([`SharedDefines.h:1119-1123`](D:/MaNGOS/source/src/game/SharedDefines.h)). After all `GREEN` spells are bought, the BG bot calls `ITrainerNetworkClientComponent.CloseTrainerAsync()` ([`ITrainerNetworkClientComponent.cs:110`](../../../Exports/WoWSharpClient/Networking/ClientComponents/I/ITrainerNetworkClientComponent.cs)); FG bot's frame closes via the Lua `CloseTrainer()` bridge or by walking out of range.

## Decision-Engine Rules

| # | Predicate | Action | Threshold / Constant | Source citation |
|---|---|---|---|---|
| R1 | **Trainer visibility detection** — `IObjectManager.Units` scan: `unit.NpcFlags & 0x10 != 0 AND unit.UnitReaction != HOSTILE AND unit.Health > 0` selects nearby trainers. **The user prompt's `0x80` is wrong** (that's `INNKEEPER`). Cross-faction trainers blocked by `UnitReaction == HOSTILE`.
