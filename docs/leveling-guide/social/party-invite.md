# Party Invite — Target Resolution → Faction + Ignore Gates → Group-State Check → CMSG_GROUP_INVITE → 60s Decline Window → AddMember Cascade

First doc in the **Social** family (`social/`). Where the Travel family answers "how does the bot move", the NPC family answers "how does the bot talk to NPCs", and the Combat family answers "how does the bot kill things" — the Social family answers "**how does the bot coordinate with other players**". Party Invite is the entry point for every multi-player activity: warlock-summon (caller `warlock-summon.md` R2/R3), mage-portal (caller `mage-teleport.md` R-portal-context), dungeon instances (caller `dungeon/` iter-31+), raids (caller `raids/` iter-31+), and any group-XP-buff or shared-quest objective. Without a successful party invite, none of those activities can begin — making this doc the **upstream prerequisite** for every cross-bot DE cost cascade.

**Scope of this doc**. The 1.12.1 vanilla mangos party-invite flow is a **single CMSG + single SMSG + 60-second decline-window + accept/decline response**, gated by **6 server-side rejection paths** in `HandleGroupInviteOpcode` ([`GroupHandler.cpp:56-154`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp)). The full lifecycle covers (a) invite send by the leader-or-assistant, (b) target receipt of `SMSG_GROUP_INVITE`, (c) target's `[ACCEPT]` or `[DECLINE]` click, (d) leader's `SMSG_PARTY_COMMAND_RESULT` response, (e) group state-machine transitions (`m_invitees` → `m_memberSlots`), (f) leader-handover semantics on leader disconnect, (g) disband cascade. **Raid-conversion** (`CMSG_GROUP_RAID_CONVERT = 654`) and **loot-method changes** are scoped out — they belong to a sibling `social/raid-conversion.md` doc planned for iter-31+.

**Why this is `party-invite` and not `group-invite`**. The mangos source uses "Group" consistently (`m_groupInvite`, `Group::AddMember`, `CMSG_GROUP_INVITE`), but the **user-facing 1.12.1 client UI** uses "Party" terminology (`"%s is not in your party"`, `"Your party is full"` per `PartyResult` enum at [`WorldSession.h:71-81`](D:/MaNGOS/source/src/game/WorldSession.h)). The doc title matches the client UI and the cross-citing travel docs (`warlock-summon.md:120` "send party-invite first"). The mangos internals remain "Group" in citations.

## Overview

Party invite lifecycle splits into **five phases**:

**Phase 1 — Invite dispatch (leader-side).** Leader's bot dispatches `CMSG_GROUP_INVITE` carrying the target's name string. Server's `HandleGroupInviteOpcode` ([`GroupHandler.cpp:56-154`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp)) runs the **6 sequential rejection gates** (name-normalize, target-exists, faction-match, ignore-list, target-already-grouped, leader-permission-OR-not-grouped-yet, group-full). On the first failure, server emits `SMSG_PARTY_COMMAND_RESULT(PARTY_OP_INVITE, targetName, PartyResult)` back to the leader and returns early. On full success, the invite proceeds to Phase 2.

**Phase 2 — `m_invitees` insert + target notification.** Server creates or reuses the `Group` object: if leader is solo, `new Group + AddLeaderInvite(leader) + AddInvite(target)` ([`GroupHandler.cpp:128-139`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp)); if leader is already in a group (re-invite or new-member), just `AddInvite(target)` ([:142-146](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp)). The `Group::AddInvite` at [`Group.cpp:258-275`](D:/MaNGOS/source/src/game/Group/Group.cpp) inserts the target into `m_invitees` set AND sets `target->SetGroupInvite(this)` — the target now sees its own `GetGroupInvite() != nullptr` flag. Server sends `SMSG_GROUP_INVITE(inviterName)` to the target's session and `SMSG_PARTY_COMMAND_RESULT(PARTY_OP_INVITE, targetName, ERR_PARTY_RESULT_OK)` to the leader's session.

**Phase 3 — Target decision window (60-second default UI timeout).** The target's client shows the standard party-invite dialog with `[ACCEPT]` / `[DECLINE]` buttons. **The mangos server does NOT enforce a server-side timeout** on the invite — the `m_invitees` entry persists until (a) target dispatches `CMSG_GROUP_ACCEPT` (114) or `CMSG_GROUP_DECLINE` (115), (b) target disconnects (server calls `UninviteFromGroup` on client-shutdown — see [`Player.cpp:3401`](D:/MaNGOS/source/src/game/Objects/Player.cpp)), or (c) the leader cancels via `CMSG_GROUP_UNINVITE`. The **client-side decline-window** is approximately **60 seconds** before the dialog auto-fades, but the server holds the invite indefinitely. **DE-side assumption**: bot should treat a 60-second elapsed time without `SMSG_GROUP_LIST` arriving as an implicit decline. Open SCo.SOCIAL.PARTY_INVITE.1 = surface server-side `Player.m_groupInvite` in snapshot so the bot can distinguish "still pending" from "declined".

**Phase 4 — Accept path (`HandleGroupAcceptOpcode`).** Target dispatches `CMSG_GROUP_ACCEPT`. Server's handler at [`GroupHandler.cpp:156-199`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp): (a) fetch `group = player->GetGroupInvite()`; return if null; (b) guard against accepting one's own invite (sanity check); (c) `group->RemoveInvite(GetPlayer())` clears the pending state; (d) if `group->IsFull()` (someone else accepted first), reject with `ERR_GROUP_FULL`; (e) if `!group->IsCreated()` (this is the first acceptance — the group only persists to DB once the second player joins), call `group->Create(leaderGuid, leaderName)` + `sObjectMgr.AddGroup(group)`; (f) `group->AddMember(GetPlayer()->GetObjectGuid(), GetPlayer()->GetName())` at [`Group.cpp:330-368`](D:/MaNGOS/source/src/game/Group/Group.cpp) inserts into `m_memberSlots` AND `player->SetGroup(group)` AND fires `player->ResetInstances(INSTANCE_RESET_GROUP_JOIN)` (Phase 5 instance-binding implications); (g) `group->BroadcastGroupUpdate()` and `SendUpdate()` broadcast `SMSG_GROUP_LIST` to all members carrying the new roster.

**Phase 5 — Decline path (`HandleGroupDeclineOpcode`) and decline cleanup.** Target dispatches `CMSG_GROUP_DECLINE`. Handler at [`GroupHandler.cpp:201-220`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp): (a) fetch `group = player->GetGroupInvite()`; return if null; (b) remember leader if online; (c) `GetPlayer()->UninviteFromGroup()` clears the `m_groupInvite` pointer AND removes from `m_invitees` AND — if this was the first invitee and group was never created — **the empty `Group` object is destroyed** ([`Player.cpp:3401-3420`](D:/MaNGOS/source/src/game/Objects/Player.cpp)); (d) if leader is online, server emits `SMSG_GROUP_DECLINE(decliningPlayerName)` to the leader's session. **Leader's bot is notified via SMSG_GROUP_DECLINE** — distinct from `SMSG_PARTY_COMMAND_RESULT`; the bot must listen to both opcodes.

## Failure modes

| Failure | Detection | Recovery |
|---|---|---|
| **Target offline / no such name** (`ERR_BAD_PLAYER_NAME_S = 1`) | Server emits `SMSG_PARTY_COMMAND_RESULT(PARTY_OP_INVITE, name, 1)` to leader. Bot decodes the result byte | Per R1, pre-filter via StateManager `BotRegistry.IsOnline`. For human targets, accept the 1-CMSG round-trip; retry once after 60s in case of brief disconnect; otherwise skip |
| **Faction mismatch** (`ERR_PLAYER_WRONG_FACTION = 7`) | Server emits `PARTY_OP_INVITE` result byte 7 | Per R2, pre-check faction via cross-bot snapshot. Cross-faction party requires server config flip (`CONFIG_BOOL_ALLOW_TWO_SIDE_INTERACTION_GROUP`) which is OFF by default. Cannot recover by retrying — final |
| **Target ignoring inviter** (`ERR_IGNORING_YOU_S = 8`) | Server emits PARTY_OP_INVITE result byte 8 | Per R6, opaque pre-push for human targets. Per-session blocklist after first IGNORING_YOU_S; never retry that target until session reset. Avoids harasser-detection by server admins |
| **Target already in group** (`ERR_ALREADY_IN_GROUP_S = 4`) | Server emits PARTY_OP_INVITE result byte 4 | Per R3, pre-check via cross-bot `target.GroupId == 0 && target.GroupInviteGuid == 0`. Cascade: (a) chat-coordinate per R11 to ask target to disband; (b) wait for target's CMSG_GROUP_DISBAND ack via SMSG_GROUP_LIST(0,0,0); (c) re-push invite |
| **Group full at moment of dispatch** (`ERR_GROUP_FULL = 3`) | Server emits PARTY_OP_INVITE result byte 3 | Per R5, pre-check `Group.MemberCount < 5` for party or `< 40` for raid. If party-cap, dispatch `CMSG_GROUP_RAID_CONVERT` (opcode 654 — see iter-31+ `social/raid-conversion.md`) then re-invite. If raid-cap, hard-fail — no further expansion |
| **Leader-role mismatch** (`ERR_NOT_LEADER = 6`) | Server emits PARTY_OP_INVITE result byte 6 when inviter is in a group but not leader/assistant | Per R4, push `PartyLeaderRequestTask` to chat-coordinate with current leader. | **Target declines (CMSG_GROUP_DECLINE)** | Server emits `SMSG_GROUP_DECLINE(name)` (opcode 116) to leader | Per R10, retry policy: 60s wait + 1 retry; blocklist after 2 declines per session. Use R11 chat-coordination to clarify intent before re-push |
| **Target implicit-declines (60s UI auto-fade, no response)** | Bot's `_inviteSentTickMs + 60_000ms` elapsed without `SMSG_GROUP_LIST` arriving | Per R8, treat as implicit decline. Dispatch `CMSG_GROUP_UNINVITE(targetName)` to free the target's `m_groupInvite` slot, THEN re-push if R10 retry-budget allows. | **Target accepts but group is full (race)** | Server's `HandleGroupAcceptOpcode` rejects with `ERR_GROUP_FULL` at [`GroupHandler.cpp:178`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp) — someone else accepted first | Rare race condition. Target's bot detects via PARTY_OP_INVITE result byte 3 sent back to itself. Recovery: chat-coordinate per R11 ("party full, try the next group"). Future SCo.SOCIAL.PARTY_INVITE.9 = LFG queue substitute |
| **Leader disconnects mid-invite** | Server's `_chooseLeader` cascade at [`Group.cpp:1640-1683`](D:/MaNGOS/source/src/game/Group/Group.cpp) picks next leader; broadcasts `SMSG_GROUP_SET_LEADER` | Per R12 automatic handover. Bot listens for SMSG_GROUP_SET_LEADER; if bot is new leader, resume the prior invite from its own perspective (re-push if target hasn't responded). **`m_invitees` pending entries persist** across the leader-change — new leader can still receive the accept/decline |
| **Target disconnects between invite and accept** | Server's `Player::UninviteFromGroup` fires on client-shutdown ([`Player.cpp:3401-3420`](D:/MaNGOS/source/src/game/Objects/Player.cpp)); clears `m_groupInvite` | Leader's bot detects via SMSG_PARTY_COMMAND_RESULT timeout (no ack). Re-push when target reconnects (cross-bot StateManager observes `BotRegistry.IsOnline` flip). ## Live-test acceptance

A live integration test exercising party invite must satisfy R8 (clean cross-test state) by:

1. **Server-up assertion** — `ServerHealthcheck` returns 200; fail-fast if not (root R4).
2. **Pre-test snapshot predicate** — `leader.GroupId == 0 AND leader.GroupInviteGuid == 0 AND target.GroupId == 0 AND target.GroupInviteGuid == 0 AND leader.Faction == target.Faction AND leader.IsOnline == true AND target.IsOnline == true`. Setup: `.tele` both bots to the same zone for proximity; ensure neither has prior group state via `.dis` (disband) if needed; verify ignore-list is empty via `.unignore` (admin command).
3. **Two-bot fixture** — `MultiBotStateManager` provisions both `leaderBot` and `targetBot`; fixture owns lifecycle of both (per CLAUDE.md fixture-ownership rule).
4. **Push `PartyInviteSendTask(targetName)`** on leader's bot via fixture. **Do NOT construct `new ObjectiveMessage { ObjectiveType = ObjectiveType.PARTY_INVITE }`** in test body — per [Test Isolation Rules](../../../CLAUDE.md).
5. **CMSG_GROUP_INVITE dispatch predicate** — poll `leaderBot.RecentCommandAcks[CMSG_GROUP_INVITE]` within 500ms of push.
6. **SMSG_GROUP_INVITE receipt on target** — poll `targetBot.RecentNetworkEvents.Any(e => e.OpcodeId == 111)` within 1s; assert decoded inviter name matches `leaderBot.Name`.
7. **SMSG_PARTY_COMMAND_RESULT receipt on leader** — poll `leaderBot.RecentNetworkEvents.Any(e => e.OpcodeId == 127 && e.PartyOp == 0 && e.Result == 0)` within 1s of dispatch.
8. **Push `PartyAcceptTask` on target bot** via fixture; poll `targetBot.RecentCommandAcks[CMSG_GROUP_ACCEPT]` within 500ms.
9. **Post-accept GroupMembers verification** — within 1s of accept dispatch, assert `leaderBot.GroupMembers.Any(m => m.Guid == targetBot.Guid) AND targetBot.GroupMembers.Any(m => m.Guid == leaderBot.Guid) AND leaderBot.IsGroupLeader == true AND targetBot.IsGroupLeader == false`.
10. **SMSG_GROUP_LIST broadcast** — both bots receive opcode 125 with full 2-member roster within 1s of accept.
11. **Decline-path test (separate scenario)** — push `PartyDeclineTask` on target; assert `leaderBot.RecentNetworkEvents.Any(e => e.OpcodeId == 116)` (SMSG_GROUP_DECLINE) within 1s; assert `targetBot.GroupInviteGuid == 0` (cleared) and `leaderBot.GroupMembers.Count == 0` (no group formed).
12. **Faction-reject test** — provision Alliance leader + Horde target; push invite; assert SMSG_PARTY_COMMAND_RESULT result byte 7 (`ERR_PLAYER_WRONG_FACTION`).
13. **Ignore-reject test** — set target's ignore-list to include leader (via `.ignore` admin command); push invite; assert result byte 8 (`ERR_IGNORING_YOU_S`).
14. **Group-full test** — fill leader's party to 5 (4 helper bots accept); push invite for 6th; assert result byte 3 (`ERR_GROUP_FULL`).
15. **Leader-handover test** — form 2-member party (leader + target); push `PartySetLeaderTask(targetGuid)` on leader; assert `SMSG_GROUP_SET_LEADER` broadcast within 1s; assert `targetBot.IsGroupLeader == true AND leaderBot.IsGroupLeader == false`.
16. **Disband-on-disconnect test** — kill `leaderBot`'s process mid-session; assert `targetBot` receives `SMSG_GROUP_SET_LEADER` (automatic handover) OR `SMSG_GROUP_DESTROYED` (sole-survivor disband) within 30s.
17. **Instance-lockout-reset test** — leader has prior bind in Deadmines; target has fresh state. Form party; assert `leaderBot.InstanceBinds[Deadmines]` is reset to "use group bind" per R14 (requires bot snapshot of instance-binds — open SCo.SOCIAL.PARTY_INVITE.10 = expose `Player.InstanceBinds[]` in snapshot).
18. **Disconnect guard** — mid-invite disconnect of either bot must fail the test (R8); do not paper over with reconnect retry.
19. **Screenshot capture** — per R11 in root CLAUDE.md: (a) leader pre-invite (target visible in same zone); (b) target receiving invite dialog; (c) post-accept (both bots showing party-frame in UI). Overwrite `Tests/artifacts/social/party-invite/latest-{pre,dialog,post}.png`.
20. **Final state dump** — write both bots' `GroupId / GroupInviteGuid / GroupMembers / IsGroupLeader / IsGroupAssistant` + last 20 deltas + RecentCommandAcks + RecentErrors to `Tests/artifacts/social/party-invite/snapshot-final.json`.
