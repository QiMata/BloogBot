# Party Disband ‚Äî CMSG_GROUP_DISBAND ‚Üí "I Leave" Semantics ‚Üí RemoveMember/Disband Cascade ‚Üí SMSG_GROUP_DESTROYED + Instance-Lockout Reset

Second doc in the **Social** family (`social/`), companion to [`party-invite.md`](party-invite.md) (iter-30). Where party-invite.md covers the **formation** path (target-resolution ‚Üí 6 rejection gates ‚Üí 60s decline window ‚Üí AddMember + ResetInstances(INSTANCE_RESET_GROUP_JOIN)), this doc covers the **dissolution** path: **explicit leave**, **leader-handover-on-disconnect**, **sole-survivor full-disband**, **instance-lockout reset on disband**, **loot-rules destruction**, and the **bot-vs-real-player divergence** in disconnect handling.

**Scope of this doc**. The 1.12.1 vanilla mangos disband flow is built around a single CMSG (`CMSG_GROUP_DISBAND = 123`) with **no payload** ‚Äî the opcode is purely "I leave my own party" per [`GroupHandler.cpp:338-354`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp). The downstream cascade splits into **two branches** depending on `GetMembersCount() > GetMembersMinCount()` (i.e. >2 after counting the leaver): (a) `_removeMember` + `SendUpdate` keeps the group alive for surviving members; (b) `Disband(true /*hideDestroy*/, initiator)` collapses the group entirely. Leader-leave triggers `_chooseLeader` (auto-promote next online member; raid prioritizes assistants); sole-survivor triggers full `Disband` with `SMSG_GROUP_DESTROYED` broadcast + `ResetInstances(INSTANCE_RESET_GROUP_DISBAND)` cascade per [`Group.cpp:643`](D:/MaNGOS/source/src/game/Group/Group.cpp). **Kick semantics** (`CMSG_GROUP_UNINVITE` opcode 117 ‚Äî leader-only ejection of a named member) are covered in [`party-invite.md`](party-invite.md) R13 footnote; this doc focuses on **self-initiated leave** + **disconnect-triggered auto-leave**.

**Why this is `party-disband` and not `group-leave`**. Same client-UI-vs-internal-naming split as [`party-invite.md:26`](party-invite.md) ‚Äî mangos source uses "Group" (`Group::Disband`, `CMSG_GROUP_DISBAND`); client UI uses "Party" (`PARTY_OP_LEAVE = 2` per [`WorldSession.h:66`](D:/MaNGOS/source/src/game/WorldSession.h)). The 1.12.1 client `/disband` slash command actually dispatches `CMSG_GROUP_DISBAND` regardless of whether the bot is leader or non-leader; the **single opcode covers both `/leave` and `/disband` UI commands** ‚Äî semantics differ only by `GetMembersCount()` at dispatch time. See R1 below.

## Overview

Party disband lifecycle splits into **four phases**:

**Phase 1 ‚Äî Disband dispatch (any member).** Any group member (leader OR non-leader) dispatches `CMSG_GROUP_DISBAND` with empty payload. Server's `HandleGroupDisbandOpcode` ([`GroupHandler.cpp:338-354`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp)) runs **two silent gates**: `!GetPlayer()->GetGroup()` (no group ‚Üí silent no-op, no SMSG) and `_player->InBattleGround()` (BG group ‚Üí silent no-op ‚Äî BGs disband via the BG-end flow, not `/disband`). On both gates passing, server emits `SMSG_PARTY_COMMAND_RESULT(PARTY_OP_LEAVE, leaverName, ERR_PARTY_RESULT_OK)` back to the leaver's session, broadcasts `Group::BroadcastGroupUpdate` to all members, then calls `Player::RemoveFromGroup()`.

**Phase 2 ‚Äî RemoveFromGroup delegation.** `Player::RemoveFromGroup()` at [`Player.h:2652`](D:/MaNGOS/source/src/game/Objects/Player.h) is an inline that delegates to `Player::RemoveFromGroup(group, guid)` at [`Player.cpp:3421-3433`](D:/MaNGOS/source/src/game/Objects/Player.cpp). That delegate calls `group->RemoveMember(guid, removeMethod=0 /*GROUP_LEAVE*/)` at [`Player.cpp:3425`](D:/MaNGOS/source/src/game/Objects/Player.cpp); if the return value (post-remove `m_memberSlots.size()`) is `<= 1`, the now-empty Group is removed from `sObjectMgr` and `delete`d. **The size check is post-remove**, so a 2-member group disbanding routes through the `Disband(true, guid)` branch inside `RemoveMember` itself ‚Äî see Phase 3.

**Phase 3a ‚Äî Survive path (3+ members, `_removeMember`).** `Group::RemoveMember` at [`Group.cpp:438-531`](D:/MaNGOS/source/src/game/Group/Group.cpp) routes through `_removeMember` when `GetMembersCount() > GetMembersMinCount()` (i.e. **>2 before remove**, leaving ‚â•2 after). The `_removeMember` ([`:1595-1638`](D:/MaNGOS/source/src/game/Group/Group.cpp)) does: (a) `player->SetGroup(nullptr)` clearing the leaver's group pointer; (b) `_removeRolls(guid)` to clean up any pending loot rolls owned by the leaver; (c) `m_memberSlots.erase(slot)` + `SubGroupCounterDecrease` for raid subgroup accounting; (d) `DELETE FROM group_member WHERE member_guid` DB cleanup; (e) **if `m_leaderGuid == guid`**, call `_chooseLeader()` and return `leaderChanged=true`. Back in `RemoveMember`: emits `SMSG_GROUP_LIST(0,0,0)` to the leaver at [`:499-501`](D:/MaNGOS/source/src/game/Group/Group.cpp); if leader changed, broadcasts `SMSG_GROUP_SET_LEADER(newLeaderName)` to all remaining members at [`:511-513`](D:/MaNGOS/source/src/game/Group/Group.cpp); calls `_homebindIfInstance(leaver)` at [`:504`](D:/MaNGOS/source/src/game/Group/Group.cpp) to start the **homebind timer** if the leaver is currently inside a dungeon (60s default, after which the player is teleported to their hearthstone bind ‚Äî server-side enforced); finally `SendUpdate()` broadcasts the new roster to all surviving members.

**Phase 3b ‚Äî Collapse path (‚â§2 members, `Disband(hideDestroy=true, initiator=leaver)`).** When `GetMembersCount() <= GetMembersMinCount()` at entry to `RemoveMember` (a 2-member party where one leaves leaves a single member behind, which is invalid for a group), execution jumps to the `else` branch at [`Group.cpp:527-528`](D:/MaNGOS/source/src/game/Group/Group.cpp) calling `Disband(true, guid)`. The `hideDestroy=true` flag **suppresses the SMSG_GROUP_DESTROYED broadcast** ([`Group.cpp:581-585`](D:/MaNGOS/source/src/game/Group/Group.cpp)) ‚Äî the surviving member sees only `SMSG_GROUP_LIST(0,0,0)` at [`:592-594`](D:/MaNGOS/source/src/game/Group/Group.cpp). Both members get their `Player::SetGroup(nullptr)` at [`:566-568`](D:/MaNGOS/source/src/game/Group/Group.cpp); the **leaver** gets `_homebindIfInstance` at [`:602`](D:/MaNGOS/source/src/game/Group/Group.cpp) (per the `initiator.IsEmpty() || initiator == player->GetObjectGuid()` guard at [`:601`](D:/MaNGOS/source/src/game/Group/Group.cpp)); the **surviving non-initiator** is tracked as `remainingPlayer` and gets the group's instance-bind transferred to them at [`:628-640`](D:/MaNGOS/source/src/game/Group/Group.cpp) (preserves their dungeon-progress without forcing a kick ‚Äî the player keeps the in-progress instance). `ResetInstances(INSTANCE_RESET_GROUP_DISBAND)` fires at [`:643`](D:/MaNGOS/source/src/game/Group/Group.cpp) on all non-transferred binds. The Group object is then deleted from DB and freed in memory.

**Phase 3c ‚Äî Full-disband path (explicit `Disband(hideDestroy=false)`).** This is the **leader-explicit-disband** path, NOT a member-leave. Invoked from outside `HandleGroupDisbandOpcode` (e.g. `Player::UninviteFromGroup` cleanup at [`Player.cpp:3411`](D:/MaNGOS/source/src/game/Objects/Player.cpp) when the pending-invite path hits a 1-member orphan group; or from admin `.dis` GM commands; or from raid-end / BG-end flows). With `hideDestroy=false`, the **`SMSG_GROUP_DESTROYED` packet IS emitted** to every member at [`Group.cpp:583-584`](D:/MaNGOS/source/src/game/Group/Group.cpp). Same downstream cleanup as 3b: `SMSG_GROUP_LIST(0,0,0)` to each surviving member, `_homebindIfInstance` on the initiator (or all members if `initiator.IsEmpty()`), `ResetInstances(INSTANCE_RESET_GROUP_DISBAND)`, DB delete, Group dtor. **No bot-driven CMSG triggers this path** ‚Äî bots use `CMSG_GROUP_DISBAND` for explicit leave, which always routes through `RemoveMember` (Phase 3a/3b).

**Phase 4 ‚Äî Disconnect-triggered auto-leave (bot vs real-player divergence).** `WorldSession::ForcePlayerLogoutDelay` at [`WorldSession.cpp:379-398`](D:/MaNGOS/source/src/game/WorldSession.cpp) shows the **critical fork**: for bots (`GetBot() != nullptr`), `GetPlayer()->RemoveFromGroup()` is called **immediately** on socket-loss ‚Äî the bot is treated as having dispatched `CMSG_GROUP_DISBAND`. For real players, the disconnected session is preserved for **120 seconds** (`m_disconnectTimer = 120000` at [`:391`](D:/MaNGOS/source/src/game/WorldSession.cpp)) and `OnDisconnected` is called; the player's group entry stays in `m_memberSlots` with `MEMBER_STATUS_OFFLINE` bit cleared via `Group::UpdatePlayerOnlineStatus(player, false)`. **Long-term real-player disconnect**: after `CONFIG_UINT32_GROUP_OFFLINE_LEADER_DELAY` (default 300s per [`World.cpp:803`](D:/MaNGOS/source/src/game/World.cpp)), if the offline player is the **leader**, `Group::UpdateOfflineLeader` at [`Group.cpp:1442-1466`](D:/MaNGOS/source/src/game/Group/Group.cpp) calls `_chooseLeader(offline=true)` to auto-promote the next online member with `SMSG_GROUP_SET_LEADER` broadcast. The offline member is NEVER auto-removed from the group on disconnect alone ‚Äî they persist until explicit `/disband` or full session-end + character-removed-from-world. **This divergence has DE implications**: bot-to-bot parties see immediate disconnect cleanup; bot-to-human parties see 300s leader-auto-promote delay then offline-member-persistence indefinitely.
