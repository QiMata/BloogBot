# Raid Conversion â€” CMSG_GROUP_RAID_CONVERT â†’ Leader+â‰¥2 Gate â†’ m_groupType=RAID â†’ Sub-Group Counter Init â†’ DB UPDATE â†’ SendUpdate Broadcast

Third doc in the **Social** family (`social/`), companion to [`party-invite.md`](party-invite.md) (iter-30, formation) and [`party-disband.md`](party-disband.md) (iter-37, dissolution). Where party-invite + party-disband cover the **lifecycle of a 5-slot party**, this doc covers the **one-way transformation** from a 5-slot party (`GROUPTYPE_NORMAL`) into a 40-slot raid (`GROUPTYPE_RAID`) organised as 8 sub-groups Ã— 5 slots. Raid conversion is the **only** path to a group >5 members in 1.12.1 â€” there is no `CMSG_RAID_INVITE` opcode; the entry sequence is always (a) party-invite to fill 5 slots; (b) `CMSG_GROUP_RAID_CONVERT` to expand the cap to 40; (c) further party-invites to fill the remaining 35.

**Scope of this doc**. The 1.12.1 vanilla mangos raid-convert flow is a **single CMSG with empty payload + immediate SendUpdate broadcast** â€” no decline window, no two-phase handshake, no per-member acknowledgement. Server's `HandleGroupRaidConvertOpcode` ([`GroupHandler.cpp:500-517`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp)) runs **three gates**: `!group` (silent no-op if not in a group), `_player->InBattleGround()` (silent no-op â€” BG groups are server-managed), and the combined `!IsLeader(...) || GetMembersCount() < GetMembersMinCount()` gate (leader-only + â‰¥2 members). On all three gates passing, server emits `SendPartyResult(PARTY_OP_INVITE, "", ERR_PARTY_RESULT_OK)` (note: PARTY_OP_INVITE=0 is **reused** here per [`GroupHandler.cpp:515 comment`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp) `// is it 0 (PARTY_OP_INVITE) correct code` â€” there is no dedicated PARTY_OP_RAID_CONVERT result-byte in 1.12.1), then `group->ConvertToRaid()` ([`Group.cpp:235-256`](D:/MaNGOS/source/src/game/Group/Group.cpp)) flips `m_groupType = GROUPTYPE_RAID`, calls `_initRaidSubGroupsCounter()` to allocate `m_subGroupsCounts[8]`, persists `is_raid=1` to the `groups` DB table, broadcasts `SendUpdate()` (new SMSG_GROUP_LIST with `groupType=1`), and re-evaluates raid-only quest-object visibility for every current member.

**Why this is `raid-conversion` and not `raid-formation`**. Same client-UI-vs-internal-naming split as [`party-invite.md:26`](party-invite.md) and [`party-disband.md:28`](party-disband.md). The 1.12.1 client `/convert` slash command and the right-click party-frame "Convert to Raid" UI both dispatch the same `CMSG_GROUP_RAID_CONVERT` opcode. The mangos source method is `Group::ConvertToRaid()`; the client UI says "Convert to Raid". **Critically**: there is **NO raid-to-party conversion path** in 1.12.1 â€” the transformation is **one-way**. The only way back to a 5-slot party is full disband + re-invite. See R7.

## Overview

Raid conversion is a **single-phase, leader-initiated, ack-and-broadcast** state transition. There are **four phases**:

**Phase 1 â€” CMSG_GROUP_RAID_CONVERT dispatch (leader only).** Leader bot dispatches `CMSG_GROUP_RAID_CONVERT` (opcode 654, empty payload). Server's `HandleGroupRaidConvertOpcode` ([`GroupHandler.cpp:500-517`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp)) runs **three gates**: (a) `!GetPlayer()->GetGroup()` (not in a group â†’ silent no-op, no SMSG); (b) `_player->InBattleGround()` (in a BG â†’ silent no-op â€” BG groups are server-managed, never user-converted); (c) `!group->IsLeader(self) || group->GetMembersCount() < group->GetMembersMinCount()` (combined gate: leader-only AND â‰¥2 members â†’ silent no-op if either fails). **On all three gates passing**: server emits `SendPartyResult(PARTY_OP_INVITE=0, "", ERR_PARTY_RESULT_OK=0)` to the leader's session â€” the `PARTY_OP_INVITE` constant is **reused** here per the source comment at [`GroupHandler.cpp:515`](D:/MaNGOS/source/src/game/Handlers/GroupHandler.cpp) (`// is it 0 (PARTY_OP_INVITE) correct code`), then calls `group->ConvertToRaid()`.

**Phase 2 â€” Group::ConvertToRaid state flip.** `Group::ConvertToRaid` at [`Group.cpp:235-256`](D:/MaNGOS/source/src/game/Group/Group.cpp) executes **four atomic side-effects**: (a) `m_groupType = GROUPTYPE_RAID` flips the in-memory enum; (b) `_initRaidSubGroupsCounter()` ([`Group.h:395-405`](D:/MaNGOS/source/src/game/Group/Group.h)) allocates `m_subGroupsCounts = new uint8[MAX_RAID_SUBGROUPS=8]`, zero-fills it, then walks `m_memberSlots` incrementing `m_subGroupsCounts[itr.group]` for each current member â€” typically `m_subGroupsCounts[0] = N` (all current members default to sub-group 0); (c) **non-BG path only** `CharacterDatabase.PExecute("UPDATE 'groups' SET 'is_raid' = 1 WHERE 'group_id'='%u'", m_Id)` persists the raid flag (BG groups skip DB write â€” they exist only in memory); (d) `SendUpdate()` broadcasts the new `SMSG_GROUP_LIST` with `groupType=1` byte to every current member, triggering the client party-frame â†’ raid-frame UI swap.

**Phase 3 â€” Per-member quest-object refresh.** After SendUpdate, ConvertToRaid loops `m_memberSlots` and schedules `player->UpdateForQuestWorldObjects()` at [`Group.cpp:246-254`](D:/MaNGOS/source/src/game/Group/Group.cpp) via a 1-tick lambda event for each online member. This re-evaluates **raid-only quest GOs** for visibility â€” some quest objectives are flagged `QUEST_FLAGS_RAID` and only show their GOs to raid-grouped players (e.g. AQ40 attunement turn-in GOs, certain world-event raid GOs). The 1-tick offset ensures the client's UI has fully transitioned to raid mode before GO visibility re-evaluates, avoiding a visual flicker.

**Phase 4 â€” Post-conversion capacity expansion via repeated party-invite.** Once converted, the group has **40 slots organized as 8 sub-groups Ã— 5 members**, but **only the original â‰¤5 members are present** in sub-group 0. To fill the remaining 35 slots, the leader (or assistant) repeats the [`party-invite.md`](party-invite.md) flow: dispatch `CMSG_GROUP_INVITE` per target. The `_addMember` auto-subgroup-pick at [`Group.cpp:1509-1530`](D:/MaNGOS/source/src/game/Group/Group.cpp) walks sub-groups 0..7 finding the first with `m_subGroupsCounts[groupid] < MAX_GROUP_SIZE=5` â€” so members are packed densely starting from sub-group 0. After conversion, **reassignment** via `CMSG_GROUP_CHANGE_SUB_GROUP = 638` (leader-or-assistant) lets the raid be reshaped into role-balanced sub-groups (tank+heals together, ranged DPS together, etc).
