## TL;DR

`Services/PathfindingService/Repository/Navigation.cs` is ~5,600 LOC of managed query-time repair (8 time-budgeted phases, 60+ repair methods, ~24 lateral offsets per phase, a 784-LOC `StaticRoutePackCache`, 16 per-spot `LongPathingRouteTests`). It exists because the navmesh tiles in `D:/MaNGOS/data` were generated by an externally-patched vmangos `MoveMapGenerator.exe` that lives outside this repo, doesn't bake transports as off-mesh connections, and isn't editable by agents.

We are pulling the generator in-tree as `tools/MmapGen/`, fixing the mesh to be authoritative for static GameObjects + transports + capsule-correct walkable rules, and then deleting almost all of the managed repair pipeline. The eventual end state is a small native-C++ `PathfindingService` (handling protobuf/TCP itself) that wraps Detour and `dtCrowd`, scales to ~3000 concurrent bots, and returns paths with no managed repair.

## Why this exists

Concrete evidence the current system is structurally broken (collected 2026-05-06):

- `Navigation.cs:5602` â€” managed-side repair pipeline with phases:
  `DynamicOverlay`, `EarlyStatic`, `StaticRoutePack` (15s budget), `PostAffordanceStatic`, `Affordance` (8s budget), `LocalPhysicsReachability`, `LongSegment`, `OverlayStraightStatic`. Total per-request budget â‰ˆ35 s.
- `LongPathingRouteTests.OrgrimmarFlightMasterToZeppelinRoute_AvoidsKnownStaticObjectBlockers` â€” **20-minute test timeout** even after focused tile regeneration. Detour itself solves city-scale paths in single-digit ms; the 20 min is the repair pipeline thrashing.
- `StaticRoutePackCache.cs:784` â€” caches "known-broken corridors" so the system doesn't re-derive them at query time. That's a memoized workaround, not a feature.
- Per-spot tests (`OrgrimmarZeppelinTowerLowerApproach`, `OrgrimmarZeppelinTowerFrictionRecovery`, `OrgrimmarCitySupport_ForwardFrictionSegment_IsBlockedForTaurenCapsule`, ...) â€” pattern-matching corridors instead of trusting Detour.
- BotRunner `TransportWaitingLogic` / `TravelTask` hand-coding `ApproachPosition` vs `BoardingPosition` vs `WalkLegTransportArrivalRadius` â€” there is no mesh path to the deck because zeppelins aren't off-mesh links.
- `Exports/Navigation/PhysicsEngine.cpp:6762` â€” WoW-faithful capsule sweep / slope / step rules, but invoked **per-segment from managed code** to validate Detour output post-hoc. The right knowledge is on the wrong side of the pipeline.
- `Westworld of Warcraft/AGENTS.md` already mandates: *"Static world collision belongs in generated navigation data. Do not hardcode route-specific blocker coordinates, clearance cylinders, detour waypoints, or live-position guards."* The 5,600-line repair pipeline violates this on a more abstract level.

## Architecture decision (ADR)

### Decision

The navmesh is authoritative. The runtime is a thin Detour wrapper. We own the generator in-tree, and we author all static collision + transports at generation time.

### Forces

- **Performance:** repair-pipeline thrashing produces 20-minute test timeouts. Pure Detour solves the same problem in milliseconds.
- **Correctness:** managed validators only know about lips/edges/slopes after Detour returned a path; they cannot prevent Detour from returning a bad one in the first place.
- **Scale:** target is ~3000 concurrent bots. The repair pipeline serializes through `g_navigationMutex` and cannot scale. Mesh-only paths scale linearly with `dtNavMeshQuery` per thread.
- **Maintainability:** every per-spot test, every route-pack seed, every "phase 9" repair adds maintenance surface that future overhaul work has to delete.
- **Authority:** the team already KNOWS WoW's collision model (`PhysicsEngine.cpp` is 6,762 lines of it). That knowledge belongs in `MmapGen`'s walkable-rule config and area types, not in query-time validators.

### Considered alternatives

| Alternative | Why rejected |
|---|---|
| Keep extending the managed repair pipeline | Each new phase makes search slower and locks in the broken model. The 20-min timeout is already the symptom. |
| Bring TrinityCore `mmaps_generator` in-tree | Different tile format (different `MMAP_MAGIC`, different header). Would force either a runtime loader rewrite or a generator output adapter, plus we lose drop-in compatibility with current `D:/MaNGOS/data` content. |
| DotRecast (C# port of Recast/Detour) for generation | Tempting (single-language pipeline) but lags upstream and we already have a C++ runtime. Stay native for parity with `Exports/Navigation`. |
| Stay on external `D:/MaNGOS/source` patched generator | Unowned, invisible to agents, undocumented patches, no regen workflow inside CI. The whole reason we're here. |
| Bake transports as walkable terrain instead of off-mesh links | Geometry is wrong (gangplanks ARE physical, but boarding semantics aren't "walk continuously"). Off-mesh is what Detour/Recast designed for this. |

### Decision in one line

`tools/MmapGen` is the canonical generator; navmesh tiles are authoritative; runtime does no repair.

## Freeze contract

While this overhaul is in flight, the following are **frozen** in `Services/PathfindingService` and `Exports/BotRunner`:

| Frozen surface | What that means |
|---|---|
| `Navigation.cs` repair phases | No new repair phase. No new offset enumeration. No new candidate budget. Existing phases stay until cutover (Phase 5). |
| `PathAffordanceClassifier.cs` | No new affordance categories. No new managed validators. |
| `RouteResultCache.cs` / `StaticRoutePackCache.cs` | No new seed entries. No new warmup paths. Existing seeds stay; cache itself is harmless. |
| `LongPathingRouteTests.cs` | No new per-spot route gates. New regression coverage goes into capsule-walkability property tests at the mesh level. |
| `TransportWaitingLogic.cs` / `TravelTask.cs` | No new `BoardingPosition` / `ApproachPosition` / `WalkLegTransportArrivalRadius` constants. No new "lip-aware acceptance radius." Transport boarding becomes off-mesh-link traversal in Phase 4. |
| `PhysicsEngine.cpp` per-segment validators | Frozen interface. Existing call sites continue to work. New calls into PhysicsEngine from Navigation.cs are not allowed. |

### What IS allowed during the freeze

- **Stability fixes** that keep current behavior working (e.g., a bug in an existing repair phase that crashes).
- **Read-only telemetry** improvements (`NavigationPerformanceMetrics.cs`).
- **Test stabilization** that doesn't add per-spot route gates.
- **Anything in `tools/MmapGen/`** â€” that's where the actual work happens.

### Cutover criteria for unfreezing

Each frozen surface unfreezes only after the overhaul phase that replaces it lands and proves green. See [Phase plan](#phase-plan).

- Delete `BotRunner` `TransportWaitingLogic` boarding-position constants (becomes "follow off-mesh link").
- Reduce `LongPathingRouteTests` to a small property-test set.

**Exit:** Native service handles ~3000 concurrent bot path queries with p99 <50 ms; the .NET PathfindingService project is archived.

## Open questions

- **Per-race tile sets vs largest-capsule tile set.** Current direction is "build to Tauren Male, validate smaller races fit naturally." If we hit a place where a smaller race fits but a Tauren cannot, do we accept the constraint (player would also be blocked) or build per-race meshes? Decide during Phase 4.
- **Tile cache for dynamic obstacles.** Detour ships `DetourTileCache` for runtime-cut obstacles (mob caravans, world-event blockers, fire pillars). Current `DynamicObjectRegistry` overlay does this in managed code. Migrating to tile cache happens in Phase 5 if it pays off; otherwise the overlay stays.
- **Sliced findpath vs single-call findpath at scale.** For 3000 bots on cross-continent routes, sliced findpath spreads cost across frames but adds per-request state. Decide during Phase 6 dtCrowd evaluation.

## Glossary

- **mmap / mmtile** â€” MaNGOS-format navmesh tile files. `.mmap` carries navmesh params, `.mmtile` carries the actual Detour tile payload + 20-byte wrapper header. See [MMAP_FORMAT.md](MMAP_FORMAT.md).
- **off-mesh connection (`dtOffMeshConnection`)** â€” Detour primitive: an explicit nav-graph edge between two polygons that aren't connected by walkable terrain. Designed for elevators, gangplanks, jump-down-cliff edges, teleport pads.
- **GO bake** â€” process of including server-spawned static GameObjects (bonfires, banners, supports) in the heightfield rasterization so Recast treats them as solid.
- **Capsule** â€” agent shape Recast erodes against. Tauren Male is the largest WoW capsule (`r=0.9747`, `h=2.625`).
- **Route pack** â€” managed-side memoized known-bad-corridor cache. Will be deleted in Phase 5.
- **Repair phase** â€” a stage of `Navigation.cs`'s post-hoc path validation/repair. Will be deleted in Phase 5.
