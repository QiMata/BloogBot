## Why this work exists

Two unrelated movement/lifecycle gaps surfaced while inspecting the live
bots in the WoW client:

1. The Shodan GM-liaison bot, which is supposed to be a stable, named
   "Shodan" character on the live server (so human players can address
   it directly), currently runs as an auto-generated random Gnome Female
   character (currently `Lilippidogeo`, guid 349 on the SHODAN account).
   The character name needs to be hard-coded as `Shodan`, and the
   existing character renamed in place.

2. After a BG bot is teleported, a third-party WoW client observing
   the BG bot in the world plays the falling animation **twice** — once
   immediately (client-side prediction lands the bot on the ground
   right after teleport) and a second time later (the authoritative
   position update arrives showing the bot is still falling, then
   lands again). This indicates a desync between the BG bot's
   client-side movement state and the server's authoritative state,
   most likely in the teleport-ack / MovementInfo path.

These have a hidden third dimension: the BG bot's movement controller
and the WoW.exe physics decompilation should agree on how teleports
and falls are acknowledged. Where they disagree, we need tests that
codify the expected behaviour against the decompilation, then fix
the BG side.

### Task A — Rename the Shodan character to "Shodan"

**Current state:**
- DB row: `characters.characters guid=349 name=Lilippidogeo race=7 class=8 account=354` (Gnome Female Mage on the `SHODAN` account).
- The name was generated by `WoWNameGenerator.GenerateName(race, gender, seed)`
  in [Exports/BotRunner/BotRunnerService.cs](../Exports/BotRunner/BotRunnerService.cs) line 954,
  invoked from `BuildCreateCharacterSequence` when no character exists on the account.
- There's currently **no override mechanism** for character name;
  `CharacterSettings` has no `CharacterName` field. (Confirmed via
  `grep -n "CharacterName" Services/WoWStateManager/Settings/CharacterSettings.cs` → no matches.)
- `LiveBotFixture.ShodanCharacterName` reads back whatever the bot
  reports via the snapshot ([Tests/BotRunner.Tests/LiveValidation/LiveBotFixture.cs:245](../Tests/BotRunner.Tests/LiveValidation/LiveBotFixture.cs)).

**Goal:** the Shodan account's character is permanently named `Shodan`,
both for the existing character (rename in place) and for any future
fresh-DB recreations.

**Recommended approach:**
1. Add an optional `CharacterName` field to `CharacterSettings` in
   [Services/WoWStateManager/Settings/CharacterSettings.cs](../Services/WoWStateManager/Settings/CharacterSettings.cs)
   so configs can pin a specific name. Wire it through to the
   `BuildCreateCharacterSequence` path in `BotRunnerService.cs:954`
   so when a config supplies `CharacterName`, the bot uses that
   instead of `WoWNameGenerator.GenerateName(...)`. Fall back to the
   generator when the field is null/empty (preserves existing
   randomized behaviour for TESTBOT1/TESTBOT2/etc.).
2. Set `CharacterName: "Shodan"` on the Shodan entry in every config
   that includes Shodan. Quick way to find them all:
   `grep -rn '"AccountName": "SHODAN"' Services/WoWStateManager/Settings/Configs/`.
3. Rename the existing `Lilippidogeo` row to `Shodan` server-side.
   In MaNGOS via SOAP: `.character rename Lilippidogeo Shodan` (this
   queues an at-login rename; the next time the character logs in
   it'll be renamed). Check the SOAP response for success — see
   `LiveBotFixture.AssertGMCommandSucceededAsync` for the assertion
   helper. Alternative if SOAP `.character rename` is unavailable:
   direct MySQL update — but per `CLAUDE.md` the rule is "no MySQL
   mutations" except for the `EnsureGmCommandsEnabledAsync`
   bootstrap. Prefer SOAP.
4. Add a fixture-level guard that, at session startup, asserts the
   Shodan character's name is `Shodan` (use `_bot.ShodanCharacterName`
   after world-entry). If it isn't, fail fast with a clear error so
   anyone running the suite on a fresh DB knows to either add the
   rename or wait for the bot to recreate the character with the
   pinned name.

**Acceptance:**
- `docker exec maria-db mysql -uroot -proot -e "SELECT name FROM characters.characters WHERE account=(SELECT id FROM realmd.account WHERE username='SHODAN');"`
  returns `Shodan` (after the next Shodan login).
- A live-validation run logs `[ACTION-PLAN] SHODAN <SHODAN>/Shodan: ...`
  (today it logs the random name).
- One commit, message style: `feat(shodan): pin Shodan character name and rename existing Lilippidogeo → Shodan`.

### Task B — Diagnose & fix BG bot post-teleport double-fall animation

**Symptom:** when a BG bot character is teleported (e.g. via
`.go xyz X Y Z mapId` or any `MoveTeleport` flow), an outside
WoW client observing that character in the world plays the falling
animation twice — once immediately (client-side prediction "lands"
the BG bot at the teleport target) and a second time later when the
server's authoritative position update arrives. This means the WoW
client's prediction of when the BG bot will hit the ground runs
ahead of the BG bot's actual reported position, and the client has
to re-snap to the slower authoritative state.

**Likely cause family** (don't take this as proof — investigate):
- The BG bot's `MoveTeleportAck` flow may not include the full
  `MovementInfo` block (per `memory/MEMORY.md`:
  > Teleport ACK must include full MovementInfo. `BuildMoveTeleportAckPayload` must
  > send GUID + counter + full MovementInfo (position, facing, flags, etc.) — not
  > just GUID + counter + timestamp.).
- The BG bot may emit `MOVEFLAG_FALLINGFAR` or `MOVEFLAG_FALLING`
  for longer than necessary post-teleport, or fail to clear those
  flags on landing — same family as FISH-001 (see
  `memory/fishing_fix.md`).
- The BG bot's `MovementController` may apply gravity on the
  client-side too slowly (not matching the WoW.exe decompilation's
  fall acceleration), causing the bot's reported Z to lag what the
  client predicts.

**Where to start:**
- [Exports/WoWSharpClient/Movement/MovementController.cs](../Exports/WoWSharpClient/Movement/MovementController.cs)
- [Exports/WoWSharpClient/WoWSharpObjectManager.Movement.cs](../Exports/WoWSharpClient/WoWSharpObjectManager.Movement.cs)
- [Exports/WoWSharpClient/Handlers/MovementHandler.cs](../Exports/WoWSharpClient/Handlers/MovementHandler.cs)
- [Exports/WoWSharpClient/Parsers/MovementPacketHandler.cs](../Exports/WoWSharpClient/Parsers/MovementPacketHandler.cs)
- Search for `BuildMoveTeleportAckPayload`, `MoveTeleportAck`, `MSG_MOVE_TELEPORT_ACK`,
  `NotifyTeleportIncoming`, `MOVEFLAG_FALLINGFAR`, `MOVEFLAG_FALLING`.
- Reference: [memory/wow_exe_physics_decompilation.md](C:/Users/lrhod/.claude/projects/e--repos-Westworld-of-Warcraft/memory/wow_exe_physics_decompilation.md)
  has the ground-truth physics constants (gravity, terminal velocity,
  fall acceleration) and packet format.
- Reference: [memory/movement_physics.md](C:/Users/lrhod/.claude/projects/e--repos-Westworld-of-Warcraft/memory/movement_physics.md)
  — speed offsets, transport data, jump/fall, recording system.
- Protocol doc: [docs/server-protocol/movement-protocol.md](server-protocol/movement-protocol.md).

**How to capture evidence:**
1. Run a BG-only test that issues a `BotTeleportAsync` (e.g. `.go xyz`)
   and then keeps polling snapshots; concurrently observe with a
   third WoW client logged in nearby. Note the timestamps of the
   teleport ack send, the snapshot's MovementFlags clearing, and the
   client's animation.
2. Capture the BG outbound packets via `WoWSharpClient` packet
   logging. Compare the ack payload bytes to the WoW.exe
   decompilation's expected ack format
   (`memory/wow_exe_physics_decompilation.md`).
3. Capture inbound `SMSG_MOVE_*` to verify when the server sends the
   updated movement broadcast and what flags it carries.

**Hypothesis-driven tests to add** under `Tests/BotRunner.Tests/`:
- A unit-level test for `BuildMoveTeleportAckPayload` (or whatever
  builds the ACK) that asserts the exact byte layout matches the
  decompilation reference. Pair with a fixture that decodes the
  payload back into a `MovementInfo` and asserts every field.
- A unit-level test that simulates a fall: inputs a teleport with
  Z above ground, ticks the `MovementController` long enough to
  land, asserts `MOVEFLAG_FALLING/FALLINGFAR` is set during the
  fall and cleared on land within the same number of ticks the
  decompilation predicts.
- A live test that issues a teleport and asserts the bot's
  snapshot reports stable position + cleared falling flags within
  a tight bound (e.g. 1.5s).

**Acceptance:**
- Bytes-level test passes against the decompilation reference.
- Live teleport test reports cleared `MOVEFLAG_FALLING*` flags
  within the asserted bound.
- Manual observation in a third WoW client: only **one** falling
  animation plays after a BG teleport (or none, if the teleport
  target is on ground).

**Commit style:** `fix(bg-movement): align teleport ack payload with WoW.exe decompilation`
(or whatever the actual fix turns out to be). One commit per
logical fix, separate test additions where they stand alone.

### Task C — Audit BG bot vs WoW.exe decompilation parity for movement / teleports / acks

This is broader than Task B. After Task B exposes one specific gap,
sweep the rest of the movement/ack flow looking for similar
divergences, and codify the expected behaviour as tests.

**Scope:**
- Outbound movement packets: `MSG_MOVE_*`, `CMSG_MOVE_*`, all ACK
  variants (`MSG_MOVE_TELEPORT_ACK`, `CMSG_FORCE_MOVE_ROOT_ACK`,
  `CMSG_MOVE_KNOCK_BACK_ACK`, `MSG_MOVE_FALL_LAND`, etc.).
- Inbound movement packets: ensure the BG bot decodes them with the
  same offsets, flag interpretation, and timestamps the WoW.exe
  client does.
- The `MovementController.cs` integrator: gravity, terminal velocity,
  step-up height, slide angle — all should match the values listed
  in `memory/wow_exe_physics_decompilation.md`.
- Speed change ACKs: `CMSG_FORCE_*_SPEED_CHANGE_ACK` must echo the
  exact speed value the server sent in `SMSG_FORCE_*_SPEED_CHANGE`.

**Method:**
1. List every BG-side movement packet builder/parser. `grep` for
   `Opcode.MSG_MOVE_`, `Opcode.CMSG_MOVE_`, `Opcode.SMSG_MOVE_`,
   `Opcode.CMSG_FORCE_*`, `Opcode.MSG_MOVE_FALL_LAND` across
   `Exports/WoWSharpClient`.
2. For each, write down the exact byte layout the BG bot
   produces/expects.
3. Compare to the decompilation reference (or the relevant
   1.12.1 protocol doc in `docs/server-protocol/`).
4. Where they diverge, add a regression test that pins the
   correct layout, then fix the BG side to match.
5. For physics constants (`MovementController` integrator), add
   tests that drive the controller through known scenarios (jump,
   fall, slide off ledge, walk up step, swim down) and assert the
   per-tick position trace matches a recorded ground-truth trace
   captured from the WoW.exe client via the existing recording
   system (`memory/movement_physics.md` mentions a recording
   system + physics replay accuracy).

**Test harness pattern:**
- Unit-level: `Tests/BotRunner.Tests/Movement/` (create if missing)
  for byte-layout and integrator tests.
- Replay: the existing physics replay machinery
  (`Tests/Navigation.Physics.Tests/`) is the right place for
  recorded-trace comparisons.
- Live: a small set of `Tests/BotRunner.Tests/LiveValidation/`
  scenarios that exercise teleport, knockback, root, speed change,
  and assert the snapshot reports the expected flag/state
  transitions within tight bounds.

**Acceptance:**
- A documented list (in `docs/`) of every BG movement packet, with
  pass/fail status against the decompilation reference.
- Tests passing for each fix.
- Live observation: a third WoW client watching a BG bot through
  a sequence of teleport → root → speed change → knockback shows
  smooth animations with no double-render artifacts.

**Commit style:** one commit per fix, descriptive messages
(`fix(bg-movement): correct MSG_MOVE_FALL_LAND payload to include hSpeed/vSpeed`,
etc.). Separate test additions where they stand alone. Don't bundle.
